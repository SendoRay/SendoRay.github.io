论文 PDF#

1. 问题定义#

1.1 解决了什么问题#

一句话定义：如何在大语言模型推理服务中，将 KV 缓存从 GPU HBM 卸载到 NVMe SSD 时，消除 HBM 与 SSD 之间关键数据和 I/O 控制路径上的 CPU 介入，使 SSD 层级达到接近 DRAM 的性能。

  • 问题类别：系统优化（存储 I/O 子系统 + GPU 存储栈 + 推理引擎集成）
  • 是新问题还是旧问题的新解法：旧问题（KV 缓存分层存储）的新解法——现有方法（LMCache + GDS）仅将 CPU 从数据路径移除，Tutti 进一步将 CPU 从 I/O 控制路径中彻底消除
  • 输入：长上下文 LLM 推理请求（含可复用的前缀 KV 缓存），存储于 NVMe SSD 上的分页式 KV 缓存块
  • 输出：近乎零 GPU 停顿的 KV 缓存恢复，端到端推理延迟接近 DRAM 层级

1.2 为什么要解决这个问题#

动机来源：工程瓶颈——随着上下文窗口扩展到数百万 token 且活跃会话增加，KV 缓存占用迅速超出 GPU HBM 和 CPU DRAM 容量。

现有方法的具体不足：

现有方法不足
纯 HBM 缓存容量有限（80GB HBM 仅能保留约 5 分钟 KV 缓存），命中率低（LEval 8%，LooGLE 4%）
HBM + DRAM（LMCache-DRAM）扩展容量仍不足（2TB DRAM 仅约 5 分钟），长上下文命中率有限（LooGLE 24%）
HBM + DRAM + SSD（LMCache-SSD）分页 KV 布局导致大量微小的随机 I/O，CPU 软件开销严重，GPU 空闲达 70%+
LMCache + GDSGDS 消除了 CPU-GPU 数据拷贝，但 CPU 仍在控制路径——每次 I/O 仍需 CPU 发起，低并行性 CPU 成为瓶颈，GPU 空闲仍高达 70%+

不解决的后果：从 SSD 恢复 KV 缓存导致 70–80% 的 GPU 停顿（§2.2，图 2），大量昂贵的 GPU 周期浪费在等待数据传输上，使得 KV 缓存重用甚至比重新计算还慢 [6, 13, 16, 17, 37]。这使得 SSD 层级形同虚设——系统倾向于将大部分 KV 缓存保留在 DRAM 中，降低了命中率，削弱了前缀缓存的核心优势。随着 GPU 计算能力和模型侧效率持续提升（vLLM v0.12.0 → v0.17.0），I/O 瓶颈相对更加突出（图 2），问题进一步加剧。

为什么是现在解决：

  1. 场景变化：长上下文模型（百万 token 级）和 Agentic 工作负载需要保留超过一小时的 KV 缓存，仅 SSD 容量（100TB+）可满足
  2. 技术成熟度：GPU 中心存储（BaM [40]、GeminiFS [39]）已证明 GPU 可直接发起 NVMe I/O，但尚未适配 KV 缓存场景
  3. NVIDIA Green Contexts [34] 提供了硬件级 SM 隔离能力，使 I/O 内核与计算内核的并行成为可能

1.3 问题难点在哪#

核心技术挑战：

  1. 分页 KV 布局与 SSD 访问模式的不匹配：现代推理引擎（vLLM、SGLang）将 KV 缓存分割为 16–32 token 的非连续块。对于 64 层的 Qwen3-32B 模型（块大小 64），恢复 128K token 的 KV 需要获取约 256K 个分散的 80KB 对象（§2.2），产生大量小规模随机 I/O
  2. CPU 中心化的 I/O 控制路径：即使使用 GDS 绕过了 CPU-GPU 数据拷贝，每次 I/O 仍需 CPU 发起，CPU 介入次数为 \(O(N \times L)\) （\(N\) 为请求数，\(L\) 为层数），低并行性的 CPU 成为严重瓶颈
  3. GPU 上哈希表性能极差：将 KV 缓存管理下放到 GPU 需要在设备代码中实现哈希分配和查找，但 GPU 哈希表插入/查找开销比 CPU 高 9.0×–24.2× 和 25.6×–50.0×（§2.4，图 3），因为哈希计算和探测形成串行依赖链，难以映射到 SIMT 执行模型

根本矛盾 / Tradeoff：

  • 容量 vs 延迟：SSD 提供近乎无限的容量但 I/O 延迟远高于 DRAM/HBM
  • 粒度 vs 带宽利用率：细粒度分页利于内存管理和前缀共享，但导致 SSD 随机 I/O 效率低下；粗粒度聚合提升带宽但牺牲缓存管理粒度
  • I/O 并行 vs 计算资源：增加 I/O 并行性可提升带宽利用率，但 I/O 内核会与推理计算竞争 GPU SM 资源
  • 读写并发 vs 带宽崩溃：逐层流水线中前一层的写入 I/O 与下一层的读取 I/O 并发时，总带宽下降 60%（§3.3，图 6），因大块读写竞争 NVMe 内部缓存

为什么简单方法不行：

  • 简单使用 GDS → CPU 仍在控制路径，每次 I/O 仍需 CPU 发起，并行性受限
  • 简单增大 I/O 粒度（如 LMCache 默认 256 token 数据块）→ 128K token 的 KV 仍需 1,000+ 次访问，且牺牲前缀共享粒度
  • 简单将通用 GPU 中心存储（BaM/GeminiFS）直接用于 KV 缓存 → 抽象不匹配（块/文件接口 vs KV 管理需求）、粒度差距（4KB I/O vs ~100KB KV 传输）、资源竞争（同步忙等待 I/O 占用 SM）
  • 简单流水线化（逐层传输）→ 在 SSD 上导致传输碎片化，进一步降低有效带宽

2. 方法#

2.1 核心思想#

一句话概括：构建一个以 GPU 为中心的两级（HBM-SSD）KV 缓存对象存储系统，CPU 仅在每层异步加载一次 I/O 内核至 GPU，由 GPU 直接向 SSD 发起大量并行 I/O 请求，通过 GPU 原生对象抽象、GPU io_uring 和感知松弛的 I/O 调度，充分利用 NVMe SSD 带宽并将 GPU 停顿降至接近零。

Key Insight（“啊哈"时刻）：

KV 缓存管理的所有逻辑（索引、跨请求全局共享、引擎可见的映射）均可由 CPU 在 I/O 关键路径之外（off-path）处理，而 GPU 仅持有直接 I/O 提交所需的元数据。因此可以将 CPU 开销从 \(O(N \times L)\) 降低至 \(O(N)\) ——每层仅异步加载一次 I/O 内核，而非每请求每层都需 CPU 介入。

这个洞察之所以关键，是因为它打破了"GDS 已足够"的幻觉。GDS 仅移除了 CPU 在数据路径中的角色（P2P DMA 绕过 CPU 缓冲区），但 CPU 仍在控制路径上（发起每次 I/O）。Tutti 认识到 KV 缓存管理的元数据操作（分配、索引、映射）可以与 I/O 提交解耦——CPU 在路径之外准备好 I/O 控制块（IOCB），GPU 在路径之上自主执行 I/O，从而彻底消除 CPU 瓶颈。

2.2 技术细节#

设计 1：GPU 中心的对象存储（§3.1）#

解决的问题：抽象不匹配——推理引擎需要动态分配和索引 GPU 内存块，而 GPU 中心存储仅提供底层块/文件接口；GPU 上哈希表性能极差（9×–50× 开销）。

核心设计：将每个内存块表示为一个对象，使存储分配与推理引擎的 KV 块管理器对齐。

┌─────────────────────────────────────────────────┐
│              GPU KV Cache Pool (HBM)             │
│  ┌─────────┐  ┌─────────┐           ┌─────────┐│
│  │ Block A │  │ Block B │   ...     │ Block Z ││  ← 推理引擎可见
│  └────┬────┘  └────┬────┘           └────┬────┘│
│       │            │                     │      │
│  ═════╧════════════╧══════════════════════╧════│
│           P2P Memory Mapping Table              │  ← 虚拟地址→PCI物理地址
│  ═══════════════════════════════════════════════│
│              GPU File Pool                      │
│  ┌─────────────┐  ┌─────────────┐              │
│  │ GPU File 0  │  │ GPU File 1  │   ...        │  ← GPU 端对象抽象
│  │ Layer0: K,V │  │ Layer0: K,V │              │
│  │ Layer1: K,V │  │ Layer1: K,V │              │
│  │   ...       │  │   ...       │              │
│  └──────┬──────┘  └──────┬──────┘              │
└─────────┼────────────────┼─────────────────────┘
          │                │
    ┌─────┴─────┐    ┌─────┴─────┐
    │ NVMe File │    │ NVMe File │   ...          ← GeminiFS 管理
    │  (SSD 0)  │    │  (SSD 1)  │
    └───────────┘    └───────────┘

GPU 文件组织：一个 GPU 文件被组织为 \(2 \times L\) 个对象（\(L\) 为层数），每层对应一个 Key 对象和一个 Value 对象。该映射保留了推理引擎原生的块粒度，使动态分配、索引和共享在 HBM 和 SSD 两级存储中保持一致。

Tensor-Stripe 布局：将每个 GPU 文件映射到多个 NVMe 文件，遵循原始张量粒度而非细粒度的存储分条。GPU 文件形状与 KV 缓存内存对象 \((2 \times L \times \text{Block})\) 匹配，使存储 I/O 与 KV 传输粒度对齐。对象以行顺序均匀分布在多个 NVMe SSD 上，平衡 I/O 流量并减少索引开销。

P2P 内存映射表 — SGL 替代 PRP：

NVMe 通过物理区域页（PRP）描述 GPU HBM 地址。对于 80GB HBM 上的 60GB KV 缓存：

方案每页描述开销总内存开销问题
PRP（4KB 粒度）每页一个指针\(15{,}728{,}640 \text{ pages} \times \text{ptr}\) → 64KB 粒度分配 PRP 列表页 → 983,040 页 → 3.75 GB严重浪费 HBM
SGL（Scatter-Gather List）16 字节描述一大块连续内存\(983{,}040 \times 16\text{B} \approx\) 15 MB内存开销降低 250×

SGL 每条目包含：物理地址（8 字节）、长度（4 字节）、标识符（4 字节），仅使用 16 字节描述一大块连续内存。

CPU-GPU 职责划分：

操作执行方时机
NVMe 文件预分配CPU启动时
KV 缓存→GPU 文件 ID 哈希映射CPU运行时（off-path）
块查找 + P2P 表查找CPU（推理引擎）运行时
生成 I/O 上下文（IOCTX）CPU运行时（off-path）
I/O 提交与执行GPU（自主）运行时（on-path）

CPU 将分配、索引和高并发 GPU 访问整合为单一操作，提供逐层批量的 Store 和 Retrieve 接口。CPU 开销从 \(O(N \times L)\) 降低至 \(O(N)\) 。

设计 2：GPU io_uring（gio_uring）（§3.2）#

解决的问题：如何让 GPU 高效地异步提交和完成 I/O，同时避免与计算内核的资源竞争。

架构：遵循"CPU 准备，GPU 执行"模型。CPU 提前准备 I/O 控制块（IOCB）并将 GPU I/O 内核与计算内核一并入队，GPU 端依赖跟踪决定何时发起 SSD 访问。

零拷贝环形缓冲区：

    CPU 端 (mmap, 非缓存)          GPU 端 (HBM)
    ┌───────────────────┐         ┌───────────────────┐
    │  SQ (提交队列)     │ ←mmap→ │  SQ (GPU HBM)     │
    │  ┌──────────────┐ │         │  ┌──────────────┐ │
    │  │ IOCB[0]      │ │         │  │ IOCB[0]      │ │
    │  │  IOCTX×2048  │ │         │  │  IOCTX×2048  │ │
    │  │ IOCB[1]      │ │         │  │ IOCB[1]      │ │
    │  │  IOCTX×2048  │ │         │  │  IOCTX×2048  │ │
    │  │ ...          │ │         │  │ ...          │ │
    │  └──────────────┘ │         │  └──────────────┘ │
    │                   │         │                   │
    │  CQ (完成队列)     │ ←mmap→ │  CQ (GPU HBM)     │
    └───────────────────┘         └───────────────────┘

与 CPU io_uring 的关键区别：每个 SQ 条目（IOCB）包含 2,048 个 I/O 上下文（IOCTX），而非单一命令。IOCTX 记录 SGL 地址、GPU 文件偏移量和长度，数量与 GPU 极小调度单元对齐。在 H100 上（单元为 2 个 SM），每个 SM 支持 64 个线程束 × 32 线程 = 4,096 并发线程，考虑寄存器压力除以 2，可支持大量并发 I/O 请求。

SM 分区（SM Partitioning）：

通过 NVIDIA Green Contexts [34] 在硬件层面将 GPU 资源隔离为"计算域"和"I/O 控制域”：

  • I/O 控制内核在专用 SM 上运行，不受计算负载波动影响
  • 避免传统协作多任务中的长尾延迟和资源饥饿
  • 提供确定性的服务质量（QoS）

这解决了简单 CUDA 流并发无法实现细粒度重叠的问题——GPU 硬件调度器高度非抢占性，长时间运行的 I/O 内核会独占资源，即使有空闲 SM 也会阻塞关键计算内核。

异步 I/O 处理流程：

1. init_queue(depth)     → 创建 depth 个 IOCB 的 SQ 和 CQ
2. get_iocb(nums, event) → 获取 IOCB，填入 CPU 侧虚拟地址
                           插入 CUDA 事件保证乱序流正确性
3. issue_io(IOCB_ids, SMs) → 使用指定 IOCB ID 和 SM 分配入队
                             GPU 内核在设备内并行生成 SSD 命令
                             完成后原子写入 CQ
4. wait_cqe()            → 检查 CQ 中特定 IOCB 索引
                           细粒度等待，无需 CPU 参与每次 I/O

设计 3：感知松弛的 I/O 调度器（§3.3）#

解决的问题：仅用异步 I/O 和 SM 分区不足以实现稳定的计算-I/O 重叠，仍存在两种干扰源：(1) 读写 I/O 竞争 SSD 带宽（并发时总带宽下降 60%）；(2) I/O 内核与模型执行竞争 SM 资源（嵌入/规范化/GEMM 可占用 90% GPU 资源）。

离线分析 SM 松弛窗口：

预填充复杂度随前缀长度变化（主要来自注意力复杂度增长），因此对每一层进行离线性能分析，生成以输入长度（\(C\) ）和前缀长度（\(P\) ）为索引的查找表：

\[\text{Slack}(C, P, \text{layer}_i) = T_{\text{compute}}(C, P, \text{layer}_i) - T_{\text{transfer}}(\text{layer}_i)\]

每条表项记录可调度松弛窗口的持续时间及可用 SM 预算。步长与单个线程束处理的 token 长度对齐，减少分析时间。分析结果只需创建一次，在相同部署下复用。

读写解耦调度：

预填充阶段:
  ┌─────────────────────────────────────────────────────┐
  │ Prefill 1    Prefill 2    Prefill 3    ...  Prefill n│
  │ ┌─────────┐ ┌─────────┐ ┌─────────┐         ┌─────┐│
  │ │Compute  │ │Compute  │ │Compute  │         │Comp ││
  │ │  +R     │ │  +R     │ │  +R     │   ...   │ +R  ││  ← 读优先
  │ │  +W(slack)│ │+W(slack)│ │+W(slack)│       │     ││  ← 写在松弛窗口
  │ └─────────┘ └─────────┘ └─────────┘         └─────┘│
  │                                                  R   │
  │                                              W Kernel│  ← 异步 I/O
  │  <─────── 可调度松弛窗口 ──────>                     │
  └─────────────────────────────────────────────────────┘
                                                         
解码阶段:
  ┌─────────────────────────────────────────────────────┐
  │ Decode 1  Decode 2  Decode 3  ...                    │
  │ ┌──────┐  ┌──────┐  ┌──────┐                        │
  │ │Comp+ │  │Comp+ │  │Comp+ │   ...   ← 尽力而为刷新写 │
  │ │W(best)│  │W(best)│  │W(best)│                      │
  │ └──────┘  └──────┘  └──────┘                        │
  └─────────────────────────────────────────────────────┘
  • 读优先：在预填充阶段，读内核优先级更高（KV 检索位于重用关键路径）。请求到达时立即入队读 IOCB，在每层开始前查询松弛表，在下一个松弛窗口内启动尽可能多的 IOCB
  • 写延迟：写请求仅在关键路径读取被调度后才处理。如果当前层存在可调度松弛窗口，发出查表允许数量的写操作；否则推迟以缩短预填充时间
  • 解码阶段尽力刷新：剩余写操作在解码阶段（GPU 利用率较低）机会性刷新，但松弛窗口短且不可预测，依赖查找表

实现与集成#

  • 代码规模：~8,000 LoC C++ + ~1,500 LoC Python
  • vLLM 集成：通过 KVConnector 接口，保留 vLLM 的块粒度 KV 管理
  • GPU 文件池接口：暴露 retrieve_layer 和 store_layer，支持逐层 KV 传输
  • 多 GPU 支持：每 GPU 一个 Tutti 实例 + 本地守护进程分配 GPU 内存和初始化 NVMe 队列对（每 GPU 32 个队列，Solidigm D7-PS1010 支持 256 个）
  • 分布式可扩展性：Tutti 保持本地快速路径，Mooncake [38] 作为跨簇控制平面（空间分配、副本元数据、位置查找），本地优先路由策略

  1. LMCache-GDS 每次仍需 CPU 发起 I/O → Tutti 通过 gio_uring 让 GPU 自主提交 I/O
  2. LMCache-GDS 使用 cuFile 暂存缓冲区 → Tutti 深度集成推理引擎，直接管理 GPU 内存（无暂存缓冲区，避免长序列 OOM）
  3. LMCache-GDS 无资源隔离 → Tutti 使用 Green Contexts 硬件级 SM 分区
  4. LMCache-GDS 不解耦读写 → Tutti 基于松弛查找表分离读写调度
  5. LMCache-GDS 粗粒度聚合（256 token）→ Tutti 对象语义保留原生块粒度（16–64 token）

2.4 Tradeoff 分析#

优势（量化）：

指标提升场景
TTFT降低 78.3%vs LMCache-GDS, vLLM v0.17.0, LEval
可达请求速率2×1s TTFT SLO 约束下 vs LMCache-GDS

SGL vs PRP 写带宽91.3× (2.922 vs 0.032 GB/s)单线程微基准

640K 前缀 TTFT1.2s（GDS OOM 失败）GLM-4-9B-1M, 2-GPU

代价：

  1. 硬件依赖：需要 NVIDIA Green Contexts（H100 及以上）实现 SM 硬件分区；需要 GeminiFS 配套文件系统支持
  2. HBM 开销：SGL 映射表仍占用 ~15MB HBM（虽远优于 PRP 的 3.75GB），环形缓冲区和 IOCB 也消耗 HBM
  3. 离线分析开销：松弛窗口查找表需对每层进行离线 profiling，虽然只需一次但增加了部署复杂度
  4. 实现复杂度：~8,000 LoC C++ + ~1,500 LoC Python，需修改 vLLM KVConnector，深度耦合 GPU 存储栈
  5. 远程路径未优化：分布式场景下远程获取路径仍使用 CPU 介导的 RDMA，未实现 GPU 直接 RDMA

适用边界：

  • 最优场景：长上下文 LLM 推理服务（64K–1M token），前缀缓存命中率高，NVMe SSD 作为第三级存储，使用 H100+ GPU
  • 不适用场景：短上下文（KV 缓存可完全放入 HBM 或 DRAM）；无 NVMe SSD 的环境；非 NVIDIA GPU（依赖 Green Contexts）

设计选择理由：

  • 选 SGL 而非 PRP → PRP 的 4KB 粒度导致 3.75GB HBM 浪费，SGL 用 16 字节描述大块连续内存，开销降至 15MB
  • 选 Green Contexts 而非 CUDA 流 → GPU 硬件调度器非抢占性，CUDA 流无法实现细粒度 SM 级隔离
  • 选离线查找表而非在线建模 → 在线建模本身消耗 GPU 资源，离线分析只需一次且可复用
  • 选读写解耦而非逐层流水线 → 逐层流水线中并发读写导致带宽下降 60%，解耦后可达饱和带宽

4. 评估#

4.1 局限性#

方法局限：

  1. 硬件耦合：SM 分区依赖 NVIDIA Green Contexts，仅限 H100 及以上 GPU；SGL 和 P2P DMA 依赖特定 NVMe 驱动支持。在 AMD GPU 或非 NVIDIA 生态中无法直接使用
  2. 远程路径未优化：分布式场景下远程 KV 获取仍使用 CPU 介导的 RDMA 路径（GPU 文件→主机内存→RDMA 传输），增加了额外 CPU 开销。论文承认这是后续工作
  3. 松弛表静态性：离线 profiling 生成的松弛查找表基于特定配置（模型、硬件），部署环境变化（如 SSD 型号更换、GPU 频率调整）需重新分析
  4. 写入带宽受限：存储带宽受 SSD 设备本身限制（单盘顺序写入 ≤10 GB/s），无法通过软件优化突破硬件极限

实验局限：

  1. 模型规模有限：最大测试模型为 9B（GLM-4-9B-Chat-1M），未验证 70B+ 大模型的性能。大模型层数更多（如 Llama3-70B 有 80 层），KV 传输粒度和 I/O 压力显著不同
  2. 单节点评估：分布式可扩展性仅在 2-GPU 上测试，未验证多节点集群规模。Mooncake 集成的远程路径为未优化原型
  3. 工作负载单一：仅使用 LEval 和 LooGLE 两个基准，未覆盖 Agentic 工作负载（多轮对话、工具调用），尽管论文在引言中强调 Agentic 是动机之一
  4. 未报告收敛性：仅报告延迟和吞吐指标，未展示输出质量（如 KV 缓存恢复后的生成准确性）

假设局限：

  1. 固定 KV 内存池假设：P2P 映射表基于"推理引擎在初始化时预分配固定 KV 缓存内存池且生命周期内稳定"的假设。若动态调整 KV 内存池大小（如 vLLM 的动态扩容），映射表需重新计算
  2. 松弛可预测假设：松弛调度假设预填充的注意力复杂度可由输入长度和前缀长度充分刻画。实际上，注意力计算的 sparsity pattern（如 FlashAttention 的分块策略）可能引入难以预测的松弛变化
  3. SSD 带宽稳定假设：离线 profiling 假设 SSD 带宽稳定。实际上，SSD 内部 GC（垃圾回收）和磨损均衡会导致带宽波动，特别是在写入密集场景下

4.2 启示#

对领域的贡献：

  1. 实践贡献：首个开源的 GPU 中心 SSD KV 缓存解决方案，证明了消除 CPU 控制路径介入可使 SSD 层级达到接近 DRAM 性能——这改变了"SSD 太慢不适合延迟敏感推理"的普遍认知
  2. 方法论贡献：提出了"CPU 准备、GPU 执行"的 I/O 模型，将 KV 缓存管理的元数据操作与 I/O 提交解耦。这一抽象不仅适用于 KV 缓存，可推广至任何 GPU 中心的大规模数据传输场景
  3. 洞察贡献：量化揭示了三个关键现象——(a) GDS 不够（CPU 仍在控制路径）；(b) 读写并发导致 60% 带宽下降；(c) 有效 I/O-计算重叠可超越 DRAM 原始延迟（中等复用下 +13.4%）

对工程实践的指导：

  • GDS 不是终点：在 GPU 中心存储设计中，仅消除数据路径的 CPU 介入不够，控制路径的 CPU 介入同样是瓶颈
  • SGL 优于 PRP：对于中等大小（~100KB）的可变传输，SGL 比 PRP 带宽高 31–91×，且 HBM 开销降低 250×
  • I/O 与计算必须协同设计：论文明确指出"高性能 I/O 必须与计算深度集成并协同优化，而不应被简单视为一个第三方插件"——LMCache-GDS 的 OOM 失败正是插件式设计的局限
  • SM 硬件分区：Green Contexts 提供了确定性的 I/O SM 隔离，优于 CUDA 流的协作式调度

可扩展方向：

  1. GPU 直接 RDMA：扩展 gio_uring 支持 GPU 发起的 RDMA 操作，消除远程路径的 CPU 介导，实现真正的分布式 GPU 中心存储
  2. 更大模型：在 70B+ 模型上验证，层数增多可能进一步放大 I/O 碎片化问题，但也提供更多松弛窗口
  3. Agentic 工作负载：多轮对话和工具调用产生大量可复用的 KV 缓存，是 Tutti 的理想场景，但调度策略需适配非预填充的 I/O 模式
  4. CXL 内存：CXL 内存池可作为 HBM 和 SSD 之间的新层级，Tutti 的对象抽象和 gio_uring 可扩展至 CXL 设备

未来工作方向：

  • 扩展设计支持 GPU 驱动的远程路径（数据暂存于 GPU 内存，由 GPU 发起 RDMA）
  • 在线自适应松弛分析（应对 SSD 带宽波动）
  • 与其他推理引擎（SGLang、TensorRT-LLM）的集成

4.3 一句话总结#

这篇论文用GPU 中心的对象存储 + GPU io_uring + 感知松弛的 I/O 调度解决了长上下文 LLM 服务中 SSD KV 缓存恢复的 CPU 瓶颈问题，核心创新是将 KV 缓存管理的元数据操作与 I/O 提交解耦（CPU 准备、GPU 执行），并用 Green Contexts 硬件级 SM 分区实现 I/O 与计算的确定性并行，代价是依赖特定 GPU 硬件（Green Contexts）和配套文件系统（GeminiFS），且远程分布式路径尚未优化。
