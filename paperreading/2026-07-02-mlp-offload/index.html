论文 PDF#

1. 问题定义#

1.1 解决了什么问题#

一句话定义：如何在大模型预训练中，将优化器状态卸载到多级存储（GPU HBM → 主机 DRAM → NVMe/PFS）时，缓解三级存储层级的 I/O 瓶颈，从而加速训练迭代。

  • 问题类别：系统优化（存储 I/O 子系统 + 训练运行时）
  • 是新问题还是旧问题的新解法：旧问题（内存墙）的新解法——现有方法仅利用节点本地 NVMe 作为三级存储，本文引入远程并行文件系统（PFS）作为额外的 I/O 路径
  • 输入：大模型（40B–280B 参数）的训练配置（模型分片、子组、混合精度）
  • 输出：加速后的端到端训练迭代（2.5× 更快）

1.2 为什么要解决这个问题#

动机来源：工程瓶颈——模型参数量以 450×/2 年的速度增长，而 GPU 内存仅以 2×/2 年增长（论文图 1），形成不可持续的"内存墙"。

现有方法的具体不足：

现有方法不足
ZeRO-3 冗余消除仍需大量 GPU 内存；无法与流水线并行结合
DeepSpeed DeepNVMe仅使用节点本地 NVMe，未利用远程 PFS 带宽；多 GPU 共享 NVMe 导致带宽竞争
FSDP仅支持主机内存卸载，不支持 NVMe
Colossal-AI TensorNVMe类似 DeepNVMe，单路径 I/O

不解决的后果：以 40B 模型为例，仅 GPU+DRAM 卸载时迭代耗时 2.3 秒，加入 NVMe 卸载后飙升至 242 秒（慢 ~105×），其中 89% 是更新阶段的 I/O 等待（§3.1）。这使得在资源受限环境下训练大模型几乎不可行。

为什么是现在解决：HPC 数据中心配备高速 PFS（如 VAST、Lustre），其聚合 I/O 带宽在中等规模下可达 3.6–13.7 GB/s，且在训练期间利用率极低（仅检查点时使用），存在大量可利用的空闲带宽。

1.3 问题难点在哪#

核心技术挑战：

  1. I/O 带宽层级差异巨大：GPU HBM（TB/s 级）→ 主机 DRAM（55 GB/s D↔H）→ NVMe（5–13 GB/s）→ PFS（3.6–13.7 GB/s），每级下降一个数量级
  2. 更新阶段的 I/O 主导：40B 模型中更新阶段占 89% 迭代时间，其中 99% 是磁盘 I/O（图 3）
  3. 多 GPU 共享存储竞争：同一节点上 4 个 GPU 进程并发访问共享 NVMe，导致带宽饱和、每进程延迟恶化（图 4）

根本矛盾 / Tradeoff：

  • 容量 vs 带宽：更慢的存储层级容量更大但 I/O 带宽更低
  • 通用性 vs 专用性：PFS 带宽受其他用户作业影响，不稳定但容量巨大
  • 并发 vs 竞争：更多并发 I/O 路径可提升聚合带宽，但共享总线/存储子系统会产生竞争

为什么简单方法不行：

  • 简单增加 NVMe 数量 → 受限于 PCIe 通道数和节点内空间
  • 简单使用 PFS 替代 NVMe → 远程访问延迟更高，且受其他用户影响
  • 简单增大主机内存 → 内存容量增长也远跟不上模型增长

2. 方法#

2.1 核心思想#

一句话概括：将本地 NVMe 和远程 PFS 统一为虚拟第三级存储层级，按带宽比例分配子组到各路径，配合缓存友好的子组调度和就地梯度转换，最大化多路径并行 I/O 吞吐。

Key Insight（“啊哈"时刻）：

HPC 系统的外部存储（PFS/对象存储）在 LLM 训练期间通常处于低利用率状态，可以与节点本地 NVMe 结合，显著提升 I/O 带宽。

这个洞察之所以关键，是因为现有方法（DeepSpeed、Colossal-AI）都将 PFS 视为检查点专用设施，从未将其纳入训练热路径的 I/O 路径。论文量化证明了这一机会的可行性——在 Testbed-2 上，远程 PFS 的读写吞吐量甚至高于本地 NVMe（13.7 GB/s vs 4.8 GB/s 写吞吐，表 1）。

2.2 技术细节#

设计原则 1：虚拟层级的统一多级、多路径异步卸载#

将所有替代存储（本地 SSD、PFS、对象存储）统一为一个虚拟第三级层级。子组按 I/O 带宽比例分配到各存储路径：

\[T_i = \left\lceil M \cdot \frac{B_i}{\sum_{i=0}^{N} B_i} \right\rceil, \quad \text{adjusted such that} \sum_{i=0}^{N} T_i = M \tag{1}\]

其中 \(M\) 为子组总数，\(B_i\) 为存储层级 \(P_i\) 的 I/O 带宽。目标：让不同路径的 I/O 操作大致同时完成，避免慢路径拖慢整体。

\(B_i\) 初始由微基准测试测量，首次迭代后根据实际观测带宽数据动态调整——这对于应对 PFS 带宽波动（其他用户作业竞争）至关重要。

设计原则 2：多路径 I/O 优化的虚拟层级并发控制#

  • 每个节点上同时只有一个工作进程可以访问特定的替代存储（独占访问）
  • 其他工作进程可自由使用 CPU 核心计算预取子组的更新，或访问其他存储路径
  • 无需全局同步，仅节点级轻量级控制
  • 工作进程可使用多个 I/O 线程饱和其分配的存储带宽

设计原则 3：缓存友好的子组处理顺序#

利用 Adam 优化器更新的尴尬并行性——子组处理顺序不影响正确性：

Iteration 1: SG_0 → SG_1 → SG_2 → SG_3  (升序)
             更新后高 ID 子组驻留快速层级

Iteration 2: SG_3 → SG_2 → SG_1 → SG_0  (降序，反转)
             高 ID 子组已在主机内存 → 免 I/O
             更新后低 ID 子组驻留快速层级

Iteration 3: SG_0 → SG_1 → SG_2 → SG_3  (再次升序)
             ...持续交替

现有方法（DeepSpeed）每次迭代都按固定顺序处理，导致所有子组每次都需从磁盘重新加载。

设计原则 4：延迟就地混合精度梯度转换#

现有方法流程（反向传播阶段）：

GPU FP16梯度 → 主机内存 → FP16→FP32转换 → 写入三级存储(含FP32梯度)

MLP-Offload 流程：

反向传播: GPU FP16梯度 → 直接存主机内存(不转换、不写入磁盘)
更新阶段: 从磁盘取子组(不含梯度) → 就地 FP16→FP32 转换 → CPU 更新

关键数据：CPU 上 FP16→FP32 转换吞吐为 65 GB/s（Testbed-1），比从三级存储获取 FP32 梯度的 I/O 吞吐高一个数量级。同时消除了反向传播中可能延迟计算的异步 FP32 梯度刷新操作。

算法伪代码#

def run_update(iter, subgroups, bandwidths):
    # 交替升序/降序以最大化缓存命中
    if iter % 2 == 0:
        order = range(len(subgroups))
    else:
        order = reversed(range(len(subgroups)))

    for i in order:
        tier = storage_tier(i)
        s = f2h_prefetch_wait(i)           # 从存储取子组到主机
        s.grad = host_grad_accum_FP16[i].to(FP32)  # 就地梯度转换
        cpu_update_kernel(s)                # CPU 执行 Adam 更新
        async_h2d_transfer(s.params.to(FP16))  # 异步推送 FP16 参数到 GPU
        new_tier = assign_storage_tier(i, bandwidths)  # 性能模型分配新层级
        async_h2f_flush(s, new_tier)        # 异步刷新到新层级
        async_f2h_prefetch(next(i), tier)   # 异步预取下一子组

系统集成#

  • 作为独立开源库实现，与 DeepSpeed ZeRO-3 集成
  • 扩展 DeepNVMe 无主卸载引擎（基于 libaio 内核加速异步 I/O）
  • 基于 C++ 实现，避免 Python GIL 和 PyTorch 池化内存管理低效
  • 通过两个 JSON 键值对启用配置
  • 可扩展至 Colossal-AI TensorNVMe 等

与最接近 Baseline（DeepSpeed ZeRO-3 + DeepNVMe）的核心差异：

  1. DeepNVMe 仅用本地 NVMe → MLP-Offload 增加 PFS 作为第二 I/O 路径
  2. DeepNVMe 多进程共享 NVMe 无控制 → MLP-Offload 节点级独占访问
  3. DeepSpeed 固定子组顺序 → MLP-Offload 交替升降序
  4. DeepSpeed 反向传播时转 FP32 写盘 → MLP-Offload 延迟到更新阶段就地转换

2.4 Tradeoff 分析#

优势（量化）：

指标加速倍数场景
端到端迭代速度2.5×40B–120B, Testbed-1
反向传播13.5×消除了 FP32 梯度刷新
更新阶段2.3×多路径 I/O + 缓存
I/O 吞吐量2.6×3.2 → 8.2 GB/s
弱可扩展性2×280B, 32× A100
成本效益2× vs 纯 GPU10× 更少 GPU

代价：

  1. 额外存储依赖：需要 HPC 级别的 PFS（VAST/Lustre），普通云环境可能不具备
  2. PFS 带宽不确定性：受其他用户作业影响，虽然性能模型可动态调整但仍有波动
  3. 实现复杂度：需修改 DeepNVMe 引擎，增加多引擎管理、并发控制逻辑
  4. 主机内存开销：需为每个存储层级分配独立的 I/O 缓冲区，加剧本已紧张的主机内存

适用边界：

  • 最优场景：HPC 数据中心，配备高速 PFS，训练 40B+ 参数模型，GPU 内存不足以容纳优化器状态
  • 不适用场景：模型小到优化器状态可完全放入主机内存（<40B）；无 PFS 可用的普通云环境；推理场景

设计选择理由：

  • 选独占并发控制而非全局调度 → 避免跨节点通信开销，保持可扩展性
  • 选就地梯度转换而非预转换 → CPU 转换吞吐（65 GB/s）远高于磁盘 I/O，计算不是瓶颈
  • 选交替序而非 LRU 缓存 → 子组大小一致且处理顺序可控，交替序实现简单且可预测

4. 评估#

4.1 局限性#

方法局限：

  1. PFS 依赖：方法效果依赖于 HPC 级 PFS 的可用性和带宽。在普通云环境（如 AWS 单实例 NVMe）中，无法获得额外的 I/O 路径，加速效果将退化至单路径的 1.6×
  2. PFS 带宽波动：PFS 由多用户共享，带宽受其他作业影响。虽然性能模型可动态调整 \(B_i\) ，但调整本身有滞后性——首次迭代仍使用初始测量值，且带宽在迭代内可能波动
  3. DeepSpeed 绑定：当前实现与 DeepSpeed ZeRO-3 的子组分片机制紧耦合，无法直接用于 FSDP（无子组概念）

实验局限：

  1. 最大规模 8 节点 / 32 GPU：未验证数百节点规模的可扩展性，届时 PFS 聚合带宽可能成为新瓶颈
  2. 未测试收敛性：论文仅报告迭代速度，未展示训练 loss 曲线或最终模型质量。交替子组处理顺序虽不影响数学正确性，但可能影响数值精度（FP16→FP32 就地转换的累积误差）
  3. 单路径消融未测 PFS-only：消融实验中 PFS 仅作为 NVMe 的补充，未测试仅使用 PFS（无 NVMe）的场景

假设局限：

  1. PFS 空闲假设：核心假设是"PFS 在训练期间低利用”。在多用户 HPC 环境中，若多个训练作业同时运行，这一假设可能不成立
  2. 子组等大假设：性能模型公式 (1) 假设所有子组大小相等。对于不等长子组（如 layer-wise 参数分布不均），需要更复杂的分配策略
  3. 带宽可测量假设：假设 \(B_i\) 可通过微基准测试准确获得。实际上，带宽随负载动态变化，微基准测试可能不代表训练时的真实带宽

4.2 启示#

对领域的贡献：

  1. 实践贡献：首次将 PFS 纳入训练热路径的 I/O 路径，打破"PFS 仅用于检查点"的惯例
  2. 方法论贡献：提出了可推广的虚拟层级抽象和子组分配性能模型，不仅适用于第三级存储，也可泛化至第二级（GPU-CPU HBM/DDR/CXL 混合）
  3. 洞察贡献：量化揭示了更新阶段 I/O 主导（89% 迭代时间）和 PFS 带宽未被利用的现象

对工程实践的指导：

  • 立即可用：MLP-Offload 已开源，可通过两个 JSON 配置项启用
  • 子组大小调优：使用 1 亿参数子组（而非 DeepSpeed 默认 10 亿）可获得更优 I/O 负载均衡
  • 存储层级规划：在 HPC 环境中规划大模型训练时，应将 PFS 带宽纳入总 I/O 预算

可扩展方向：

  1. CXL 内存池：论文明确提到下一步探索 CXL 内存池作为新的并行 I/O 路径，可能成为第二级层级的虚拟化方案
  2. 其他训练框架：核心原理可扩展至 Colossal-AI TensorNVMe，只需创建多个 DiskOffloader 对象
  3. 推理场景：推理时的 KV 缓存卸载面临类似的内存墙问题，多路径 I/O 思路可迁移

未来工作方向：

  • I/O 竞争环境下全局共享存储层级行为的深入研究
  • 可预测 I/O 带宽波动的缓解策略
  • 与流水线并行的结合（当前 DeepSpeed 不支持 ZeRO-3 + PP）

4.3 一句话总结#

这篇论文用虚拟多级存储层级 + 多路径并发 I/O + 缓存友好的子组调度解决了大模型预训练中优化器状态卸载的 I/O 瓶颈问题，核心创新是将训练期间空闲的 PFS 带宽纳入热路径 I/O 路径，代价是依赖 HPC 级存储设施且引入了带宽波动的不确定性。
