从一台机器的硬件结构出发，逐步讲清 DMA、Pinned Memory、Zero-Copy 和 GPUDirect RDMA 的本质。

第一步：先搞清楚一台机器长什么样#

┌─────────────────────────────────────────────────────────────────────┐
│                        一台物理机（一个节点）                         │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │                    主板 (Motherboard)                    │       │
│   │                                                          │       │
│   │   ┌──────────┐        ┌──────────────────────────────┐  │       │
│   │   │   CPU    │        │       内存条 (DRAM)           │  │       │
│   │   │  (大脑)  │◄──────►│    CPU 能直接读写的地方       │  │       │
│   │   │          │  内存总线│       DDR5, 64~512GB         │  │       │
│   │   └────┬─────┘        └──────────────────────────────┘  │       │
│   │        │                                                  │       │
│   │        │  PCIe 总线（高速公路，连接各种外设）             │       │
│   │   ─────┼──────────────────────────────────────           │       │
│   │        │         │              │            │            │       │
│   │   ┌────┴───┐ ┌───┴────┐ ┌──────┴───┐ ┌─────┴────┐      │       │
│   │   │  GPU   │ │  NIC   │ │  NVMe    │ │  其他    │      │       │
│   │   │(显卡)  │ │ (网卡) │ │  (硬盘)  │ │  外设    │      │       │
│   │   │ HBM内存│ │        │ │          │ │          │      │       │
│   │   └────────┘ └────────┘ └──────────┘ └──────────┘      │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                      │
│   这整个大框框 = 一个"节点" (Node)                                   │
│   多台这样的机器用网线连起来 = 集群                                   │
└─────────────────────────────────────────────────────────────────────┘

关键认知：

  • CPU 能直接读写的只有 DRAM（主内存）
  • GPU/NIC/NVMe 都是"外设"，挂在 PCIe 总线上
  • CPU 不能直接用指针访问 GPU HBM / NVMe 里的数据

第二步：搞懂 DMA（最重要的基础）#

DMA 出现之前的世界（PIO 模式）#

问题场景：CPU 想把 DRAM 里的数据搬到网卡发出去。

PIO (Programmed I/O) 时代：

  DRAM                CPU                 NIC
  ┌──────┐           ┌─────┐            ┌──────┐
  │data  │  读一个字 │     │  写一个字  │      │
  │[0]   │──────────►│ 寄 │───────────►│ buf  │
  │[1]   │  读一个字 │ 存 │  写一个字  │      │
  │[2]   │──────────►│ 器 │───────────►│      │
  │[3]   │    ...    │     │    ...     │      │
  └──────┘           └─────┘            └──────┘

  CPU 全程参与每一个字节的搬运！
  CPU 在搬数据期间完全被占用，不能干别的
  1GB 数据 → CPU 忙几秒钟 → 浪费！

DMA 出现后#

DMA = Direct Memory Access
      直接  内存  访问

核心思想：
  在 CPU 和 外设 之间，加一个"搬运工" —— DMA Controller
  CPU 只需要"下命令"，具体搬运由 DMA Controller 完成

  CPU 说：                DMA Controller 执行：
  "把 DRAM 地址 0x1000   ┌──────────────────────┐
   开始的 1GB 数据        │ 自己去 DRAM 读数据   │
   搬到 NIC 的缓冲区"     │ 自己写到 NIC         │
        │                 │ 搬完了通知 CPU(中断) │
        │ 下完命令就去     └──────────────────────┘
        │ 干别的了
        ▼
      CPU 解放！

  DRAM              DMA Controller           NIC
  ┌──────┐         ┌────────────────┐      ┌──────┐
  │data  │◄───────►│ 有自己的       │─────►│ buf  │
  │      │  总线访问│ 地址寄存器    │      │      │
  │      │         │ 长度寄存器     │      │      │
  └──────┘         │ 方向寄存器     │      └──────┘
                   └────────────────┘
                         ▲
                    CPU 写这几个寄存器
                    相当于"下命令"
                    然后 CPU 去干别的

DMA 的本质#

┌─────────────────────────────────────────────────────────────┐
│  DMA 的本质是：                                              │
│                                                             │
│  让外设（NIC/GPU/NVMe）能够                                 │
│  直接访问主内存（DRAM），                                    │
│  而不需要 CPU 一个字节一个字节地中转                         │
│                                                             │
│  DMA Controller 本质上就是：                                │
│  一个能发起内存总线事务的小处理器                            │
│  它会在总线上说：                                           │
│  "我要读地址 0x1000 的数据"                                 │
│  "我要写数据到地址 0x2000"                                  │
│  跟 CPU 访问内存的方式完全一样，只是发起者不是 CPU           │
└─────────────────────────────────────────────────────────────┘

现代系统里，DMA Controller 通常内置在外设里：

  • NIC 内置 DMA → NIC 自己去读写 DRAM
  • NVMe 内置 DMA → NVMe 自己去读写 DRAM
  • GPU 内置 DMA（叫 Copy Engine）→ GPU 自己去读写 DRAM

第三步：为什么需要 Pinned Memory（理解 Zero-Copy 的前提）#

问题：DMA 需要知道物理地址，但 CPU 用的是虚拟地址

现代操作系统用虚拟内存：
  程序看到的地址：0x1000（虚拟地址）
  实际物理内存：   可能在 0xABCD1000（物理地址）
  OS 随时可以把这页内存"换出"到磁盘，物理位置会变！

  程序的虚拟地址空间：        实际物理内存：
  0x0000 ┌──────────┐        ┌──────────┐ 物理页 A
         │  page 1  │───────►│  在这里  │
  0x1000 ├──────────┤        ├──────────┤ 物理页 B
         │  page 2  │  ┌───► │  在这里  │
  0x2000 ├──────────┤  │     ├──────────┤
         │  page 3  │  │     │  (被换出 │ ← 可能在磁盘上！
         └──────────┘  │     │   到磁盘)│
                       │     └──────────┘
  OS 维护页表，把虚拟地址翻译成物理地址
  OS 可以随时把某页换出到磁盘，物理地址变了

DMA 的困境：

  • DMA Controller 只认物理地址
  • 如果 DMA 正在搬运，OS 突然把这页内存换到磁盘
  • DMA 就会读到错误的数据，或者访问到别人的内存 → 崩溃！

Pinned Memory 解决方案#

普通内存（Pageable）：
  malloc() 分配
  OS 可以随时换页
  DMA 不安全
  ┌──────────┐
  │  page    │ ← OS说：我可以把你换到磁盘
  └──────────┘

Pinned Memory（页锁定内存）：
  cudaMallocHost() 或 mlock() 分配
  告诉 OS：这块内存不许换出，永远在物理内存里
  物理地址固定 → DMA 可以安全访问
  ┌──────────┐
  │  page 📌 │ ← 钉在这里，OS不能动
  └──────────┘

代价：
  占用物理内存，OS 不能灵活调度
  过多 pinned memory 会导致系统内存压力大

没有 Pinned Memory 时发生什么（有 CPU bounce）#

cudaMemcpy(gpu_ptr, cpu_ptr, size, H2D)
如果 cpu_ptr 是普通 malloc 的内存：

  DRAM (pageable)    DRAM (pinned临时)      GPU HBM
  ┌─────────────┐    ┌─────────────┐       ┌─────────┐
  │你的数据     │───►│  临时缓冲区  │──────►│         │
  │(普通malloc) │CPU │(CUDA偷偷创建)│  DMA  │         │
  └─────────────┘拷贝└─────────────┘       └─────────┘
       第一次拷贝          第二次拷贝
     (CPU执行，慢)       (DMA执行，快)

总共两次拷贝！第一次是"CPU bounce"（CPU被迫参与）

如果 cpu_ptr 是 cudaMallocHost() 的 pinned memory：

  DRAM (pinned)                              GPU HBM
  ┌─────────────┐                           ┌─────────┐
  │你的数据 📌  │──────────────────────────►│         │
  │             │        DMA直接搬           │         │
  └─────────────┘                           └─────────┘
       只有一次拷贝，CPU不参与 ✅

DMA 登记一览#

设备DMA 访问的内存如何登记
GPU (Copy Engine)CPU DRAM（必须 pinned）cudaMallocHost() CUDA 驱动负责登记
NIC (DMA Engine)CPU DRAM（必须 pinned）普通 TCP: 内核自动处理；RDMA: ibv_reg_mr() 手动注册
NVMe SSD (DMA Engine)CPU DRAM（必须 pinned）内核 NVMe 驱动自动处理；io_uring 可用用户态 buffer
NIC (DMA Engine)GPU HBM（跨设备！）GPUDirect RDMA，nvidia-peermem 内核模块，NIC 的 DMA 直接访问 GPU 内存

所有 DMA 都遵循同一个原则：

  1. 目标内存必须物理地址固定（pinned）
  2. 必须向 IOMMU 登记，获得访问权限
  3. 然后 DMA Engine 就能绕过 CPU 直接搬运数据

第四步：Zero-Copy 到底是什么#

Zero-Copy 的定义#

┌─────────────────────────────────────────────────────────────┐
│  Zero-Copy 的含义：                                          │
│                                                             │
│  数据从 A 到 B，中间没有任何额外的内存拷贝                   │
│  （拷贝次数 = 0 次额外拷贝，不是说完全没有移动）             │
│                                                             │
│  关键词：额外的（unnecessary copy）                          │
└─────────────────────────────────────────────────────────────┘

对比：发送文件的场景#

传统方式（4次拷贝！）：
  NVMe       内核缓冲区      用户空间        内核Socket缓冲区     NIC
  ┌────┐    ┌──────────┐   ┌──────────┐    ┌──────────────┐   ┌────┐
  │文件│───►│page cache│──►│  你的    │───►│   socket     │──►│发送│
  │    │    │          │   │  buffer  │    │   buffer     │   │    │
  └────┘    └──────────┘   └──────────┘    └──────────────┘   └────┘
            ①DMA拷贝       ②CPU拷贝        ③CPU拷贝           ④DMA拷贝
            磁盘→内核       内核→用户        用户→内核           内核→NIC

  涉及2次CPU参与的拷贝 + 2次DMA拷贝 = 4次总拷贝
  CPU必须把数据"摸一遍"

sendfile() Zero-Copy（2次拷贝）：
  NVMe       内核缓冲区                                          NIC
  ┌────┐    ┌──────────┐                                       ┌────┐
  │文件│───►│page cache│──────────────────────────────────────►│发送│
  │    │    │          │           内核直接传输                 │    │
  └────┘    └──────────┘                                       └────┘
            ①DMA拷贝                                           ②DMA拷贝
            数据从没进入用户空间！CPU完全不碰数据！

  只有2次DMA拷贝，0次CPU拷贝 → Zero-Copy

GPU 场景的 Zero-Copy#

场景：GPU 计算结果 → 发送到网络

有 CPU bounce（老方式）：3次拷贝

  GPU HBM        CPU DRAM        CPU DRAM         NIC          网线
  ┌──────────┐  ┌──────────┐   ┌──────────┐   ┌────────┐
  │ 计算结果  │  │ staging  │   │ socket   │   │        │   ════════
  │ tensor   │─►│ buffer   │──►│ buffer   │──►│        │──►
  └──────────┘  └──────────┘   └──────────┘   └────────┘
       ①              ②               ③
   PCIe DMA       CPU 拷贝          DMA
   GPU→CPU内存    内核内拷贝        内核→NIC
                      ↑
                 CPU 被迫参与！
                 而且数据在 CPU DRAM
                 里被摸了两次

GPUDirect RDMA（Zero-Copy）：1次拷贝

  GPU HBM                          NIC              网线
  ┌──────────────────────────┐   ┌────────┐
  │ 计算结果 tensor 📌        │   │        │   ════════
  │ (已向NIC驱动登记)         │──►│        │──►
  └──────────────────────────┘   └────────┘
                ①
           PCIe DMA
        NIC 直接读 GPU HBM
        CPU 完全不在数据路径上！
