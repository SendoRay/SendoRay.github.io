一次新功能开发的工程方法论：以接入 Min-P 采样算子为例

最近在给推理框架接入一个新的采样算子（Min-P Sampling）时，复盘了一下自己的开发节奏，发现一个比较稳定的"四步法"非常好用。这篇博客不聊采样算法本身的数学细节，而是把它当成一个载体，记录一下我现在做新功能时的工程方法论。

TL;DR：先理解背景 → 再看历史方案 → 写 minimal test 跑通新功能 → 接入大框架。每一步都要有明确的退出条件，不要跳步。

为什么需要方法论？#

做推理框架这种"大型系统中的小改动"，最容易踩的坑不是算法本身，而是：

  • 上来就在大框架里改代码，跑一次端到端要好几分钟，debug 链路超长；
  • 不了解既有方案，自己造了一个和已有 API 风格冲突的轮子；
  • 改完之后才发现假设错了，要回退一大片；
  • 接入后某个边界条件挂了，但因为没有最小可复现用例，定位非常困难。

我现在的做法是把整个开发过程拆成四个阶段，每个阶段产出一个最小的、可验证的交付物，然后再往下走。

Step 1：详细分析背景#

这一步的目标是回答清楚三个问题：

  1. 我到底要做什么？——把需求翻译成一句可以贴在 PR 描述里的话。
  2. 它在系统里的位置在哪？——是新增一条独立路径，还是替换/扩展已有的某个模块？
  3. 成功的标准是什么？——什么样的测试通过、什么样的指标达标，才算"做完了"。

以 Min-P 采样为例，这一步我会写下来类似这样的笔记：

需求：在现有采样器中新增 Min-P 采样策略，作为 Top-K / Top-P 之外的可选项。

位置：推理框架的 Sampler 模块，紧跟在 logits 后处理之后、multinomial 采样之前。和现有的 Top-K / Top-P 是兄弟关系，不替换它们。

成功标准：

  • 给定一组固定的 logits 和 min_p 阈值，输出的候选 token 集合可被手工验证；
  • 与已有的 Top-K / Top-P 可以叠加使用，行为符合预期；
  • 接入后端到端跑一个小模型，开启 Min-P 不会触发任何已有测试回归。

这一步的关键不是写得多漂亮，而是把模糊的需求收敛成可以验证的条件。如果连成功标准都说不清楚，后面的所有工作都是没有方向的。

Step 2：弄清楚之前是怎么做的#

我以前的习惯是看完需求就开始动手，结果经常出现两种尴尬：

  • 写到一半发现框架里已经有几乎一样的工具函数；
  • 自己设计的 API 和已有的"邻居"风格完全不一致，code review 时被打回重写。

现在我会强制自己花一段时间，把"邻居"的实现完整读一遍。对 Min-P 来说，“邻居"就是 Top-K 和 Top-P：

  • **它们的函数签名长什么样？**输入是不是 (logits, config) 的形式？返回的是 mask 还是处理过的 logits？
  • **它们在哪里被调用？**是 sampler 的主循环里直接调，还是通过某种注册机制？
  • **它们是怎么测试的？**测试的输入构造方式、断言的粒度，都是后面写自己测试的参照模板。
  • **有没有已经被验证过的工具函数？**比如对 logits 排序、构造 mask、scatter 回原位置这些。

这一步的产出是一份"对照表”：我要新增的代码，每一块都能在已有代码中找到对应的范式。新功能尽量长得像它的兄弟，这样审查者读起来心智负担最低，未来的维护者也最容易上手。

如果发现已有的某个抽象不够，再决定要不要扩展抽象——但一定要意识到自己在改抽象，而不是无意识地引入一种新风格。

Step 3：写 minimal test 验证新功能能跑#

这是我觉得最重要、也最容易被跳过的一步。

很多人写新功能的路径是：直接在主代码里写实现 → 端到端跑一遍 → 发现挂了 → 加 print → 再跑一遍。这个循环每轮的成本巨大，而且越往后调试越困难。

我现在的做法是：先把新功能当成一个孤立的纯函数来实现和测试，完全脱离大框架。

对 Min-P 来说，第一版的测试大概长这样：

import torch
from my_sampler import min_p_filter  # 还不存在，这是即将要写的函数

def test_min_p_basic():
    # 构造一个非常小、可手算的输入
    logits = torch.tensor([[2.0, 1.0, 0.5, -1.0, -3.0]])
    min_p = 0.1

    # 手算预期：softmax 后概率约为 [0.59, 0.22, 0.13, 0.03, 0.004]
    # max_prob = 0.59, 阈值 = 0.59 * 0.1 = 0.059
    # 保留索引 0, 1, 2
    expected_keep = torch.tensor([[True, True, True, False, False]])

    mask = min_p_filter(logits, min_p)
    assert torch.equal(mask, expected_keep)


def test_min_p_all_kept_when_threshold_zero():
    logits = torch.randn(4, 100)
    mask = min_p_filter(logits, min_p=0.0)
    assert mask.all()


def test_min_p_batch_independence():
    # batch 内不同样本应独立计算阈值
    logits = torch.tensor([
        [5.0, 1.0, 0.0],   # 分布尖锐，应过滤更狠
        [1.0, 0.9, 0.8],   # 分布平坦，应保留更多
    ])
    mask = min_p_filter(logits, min_p=0.2)
    assert mask[0].sum() < mask[1].sum()

这三个测试很短，但已经覆盖了：基本逻辑、边界条件、批处理独立性。它们的特点是：

  • 跑起来快——毫秒级反馈，可以无脑反复跑；
  • 失败信息明确——失败了一定是我新写的这个函数错了，不会被框架里其他东西干扰；
  • 可手算验证——输入数字小到可以拿草稿纸算，避免"测试通过但其实算的是另一个东西"。

只有当这一层的测试全部稳定通过，我才允许自己进入下一步。这一步的退出条件非常硬：新功能在脱离框架的前提下，行为完全符合预期。

Step 4：接入大框架#

到这一步，新功能本身已经被证明是对的，剩下的工作就只剩"把它放到正确的位置"。这一步的心态和前面三步完全不同：

  • 主要风险不在算法，而在集成：参数怎么从配置传进来？怎么和已有的 Top-K / Top-P 叠加？开关默认值是什么？
  • 改动应当尽可能小、尽可能局部化。如果发现要大改框架，说明 Step 2 没做透，应该回头补课，而不是硬改。

我的执行顺序通常是：

  1. 在配置/参数对象里加一个新字段，默认值设为"关闭"——保证零回归；
  2. 在 sampler 的主流程里加一个最小的调用点，复用 Step 3 里已经写好的纯函数；
  3. 加一个集成测试：构造一个小模型，开启 Min-P 跑一次完整 forward + sample，断言输出 token 在预期集合内；
  4. 跑一遍已有的回归测试套件，确认默认关闭路径完全没受影响。

注意第 3 步的集成测试和 Step 3 的 minimal test 是互补关系，而不是替代：

  • minimal test 保证算子本身正确；
  • 集成测试保证接线正确。

两层都要有。只有 minimal test，可能接错位置；只有集成测试，挂了不知道是算子的问题还是框架的问题。

一些反复踩过的坑#

最后记几个我自己反复踩过、现在已经形成肌肉记忆的点：

  • 不要跳过 Step 2。“我大概知道之前怎么做的"和"我读过之前的代码"是两件事，区别会在 code review 时血淋淋地暴露出来。
  • minimal test 的输入要小到可以手算。一旦输入大到只能靠"看起来差不多"来判断对错，这个测试就失去了锚定作用。
  • 新功能默认关闭。任何接入大框架的新路径，第一版都应该是"默认不走"的，让回归风险趋近于零，再逐步切换默认值。
  • 每一步都要有明确的退出条件。“差不多了"是最危险的信号，它通常意味着下一步会用大量时间去补这一步本该做完的事。

小结#

回过头看，这四步本质上是在做一件事：把一个大的、模糊的、风险高的任务，拆解成一连串小的、清晰的、低风险的任务。每一步都有自己明确的产出和验证方式，前一步不通过就不进入下一步。

这套方法不是什么银弹，但至少能让我在做"系统里加一个小功能"这种最常见的工程任务时，少走很多弯路。下次再遇到"接入一个新的算子/策略/模块"的需求，不妨试试这个节奏。
