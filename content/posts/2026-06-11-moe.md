> MoE（Mixture of Experts）如何用"稀疏激活"换取海量参数容量？Expert Parallelism 如何让数百个 Expert 分布在上百张 GPU 上协同训练？

## 一、开篇引入

2024 年以来，几乎所有头部大模型都不约而同地走向了同一条路线——**MoE 架构**。从 Mixtral-8x7B、DeepSeek-V2/V3、Qwen2-MoE/Qwen3-235B，到 GPT-4 系列被广泛"猜测"采用的稀疏专家结构，业界正在用一种朴素的工程哲学回答一个根本问题：**当稠密模型的训练成本随参数量平方级膨胀时，我们还能怎样把模型做得更大？**

回顾过去三年的稠密模型规模曲线，会看到一个非常残酷的现实：**每多一倍参数，训练成本至少多出两倍**。GPT-3 175B 用 3640 PFlop/s-day 跑了几个月；Llama-2 70B 在 2T token 上要烧掉数千张 A100；如果再往上推到 500B 稠密模型，按当前算力预算几乎不可承受。换句话说，稠密路线在 2023 年底已经触到了"经济墙"——不是不能训练更大模型，而是再没人愿意为线性增长的能力付指数级增长的电费。

正是在这个背景下，**2024 年被业内普遍称为"MoE 元年"**。这并不是说 MoE 这个想法在 2024 年才出现——它早在 1991 年就由 Jacobs 等人提出，2017 年 Shazeer 用 Sparsely-Gated MoE 把它带进 LSTM 时代，2020 年 GShard 把它工程化到 600B 翻译模型。但真正让 MoE 走出研究院、变成所有头部 lab 默认架构的，是 2024 年 Mixtral-8x7B 的开源、DeepSeek-V2 的横空出世以及 Qwen3 的全面 MoE 化这三件事的连续发生。从那一刻起，"做大模型"和"做 MoE"几乎就是同义词。

MoE 给出的答案非常直接：**让模型的"知识容量"和"每次推理的计算量"解耦**。一个 671B 参数的 DeepSeek-V3，每个 token 实际只激活 37B 参数；一个 235B 的 Qwen3-MoE，每个 token 只走 22B 计算路径。**总参数量决定了模型能存多少知识，激活参数量决定了它跑得多快、多省电**。这种"宽而稀疏"的设计，让我们得以在固定算力预算下，把模型的参数容量推到稠密模型望尘莫及的尺度。

打一个不太严谨但易懂的比方：稠密模型像是一支**全员上场的足球队**，每一次进攻都要求 11 个球员同时跑动；而 MoE 更像一支**有 100 名球员的俱乐部**，每场只派出最适合本场战术的 2 名球员上场比赛。俱乐部能积累的经验和战术多样性远超 11 人小队，但单场比赛的体力消耗反而更小。MoE 让模型在"知识储备"上向数据库靠拢，在"激活路径"上向稠密模型对齐，正是对"宽而稀疏"这四个字最朴素的工程诠释。

但 MoE 的工程实现远没有数学公式那么优雅。当几百个 Expert 被切分到上百张 GPU 上，**每一个 token 都要先被路由到某几个 Expert，再跨设备传输、计算、回传、聚合**——这背后是 All-to-All 通信、Token Dropping、负载均衡、Capacity Factor、计算通信重叠……一个个深坑等着工程师。一个错误的负载均衡策略可能让 80% 的 Expert 永远不被激活；一个糟糕的 All-to-All 调度可能让 GPU 利用率从 60% 跌到 20%。

在进入正文之前，还要先建立一个心理预期：**MoE 不是免费的午餐**。它在算力维度上省下来的成本，会在显存、通信、调度、负载均衡这四个维度上重新还回去——而且是连本带利地还。一个 32B 激活参数的稠密模型，可以塞进 8 张 A100；但一个等价激活参数的 MoE 模型，可能需要 32 张甚至 64 张 GPU，仅仅是为了把所有 expert 的权重和优化器状态存下来。这意味着如果你只是想做小规模微调，MoE 反而是更贵的；MoE 的真正甜区，是在"超大规模预训练"这个特定语境下才会显现的。理解这一点对于读懂后面 EP/DP/TP 各种并行组合、各种通信调优非常关键——很多看似复杂的工程决策，背后都是在和"MoE 把账记到了显存和通信上"这条经济学规律博弈。

本文将系统性地拆解 MoE 训练的全栈知识，按"原理 → 并行 → 框架 → 前沿 → 工程"的顺序展开：

- **第二章** 讲清楚 MoE 的基础架构、Gating 机制、负载均衡问题与演化历程；
- **第三章** 深入 Expert Parallelism 的核心原理，包括 All-to-All 通信、Capacity Factor、计算通信重叠等关键技术；
- 后续章节将进一步覆盖 DeepSpeed-MoE / Megatron-Core / DeepSeek-V3 的工程实现，以及大规模训练中的实战调优经验。

读者只需具备 Transformer 基础与基本的并行训练概念（DP / TP / PP），就能跟随本文的伪代码与公式，把 MoE 从"听过的名词"变成"能动手实现的系统"。

**一个心理准备**：本文会在多个章节重复出现 "All-to-All、Capacity Factor、负载均衡、Token Drop、GroupedGEMM、Parallel Folding" 这几个关键词。它们在不同章节会被从“是什么”、“为什么重要”、“如何实现”、“怎么调优”四个角度反复拆解。如果你在某个章节看到某个名词还不太明白，**不妨先跳过、往后翻一翻**，许多概念经过三四次不同视角的阐述后会意外地明朗。本文的设计哲学也是如此——**不追求一词一意，而是在不同语境里让同一个概念被眼睛看明白、被手摸明白、被调试明白**。

## 二、MoE 基础原理

### 2.1 MoE 核心架构

MoE 层本质是一个 **"条件计算"** 的 FFN：每个 token 不再被送进同一个 FFN，而是被一个**Router**（也叫 Gate）选择性地分发到若干个 Expert FFN 中，再把这些 Expert 的输出按权重加和。

可以把它想象成一个"分诊台"：医院里 Expert 是各专科医生，Router 是分诊台护士，token 是病人。**护士不会让所有医生都看每个病人**，而是根据病人特征（token embedding）挑选 1~2 个最合适的专科医生，看完后再把多个医生的诊断意见加权综合。

如果还嫌这个比喻抽象，可以再想象一个**真实的会诊场景**：一个心脏病患者来到医院，分诊台不会让骨科医生、皮肤科医生都对他做一遍体检——这既浪费时间，也浪费医疗资源。分诊台快速评估之后，把患者送到心血管内科和心外科两位专家那里，最后由这两位专家给出加权综合的治疗方案。MoE 之所以能省算力，本质就是这种"按需调用专家"的思想——**对一个具体输入而言，绝大多数参数都是不相关的，把它们激活只会浪费 FLOPs**。

这里还隐藏着一个常被忽略的问题：**为什么不直接做一个超大的 FFN 层？**毕竟一个 8 倍宽的 FFN 也能拥有 8 倍的参数量。答案是——超大的 FFN 在每个 token 上都要全量计算，FLOPs 也跟着 8 倍线性增长，并没有省下任何东西；而 MoE 通过"路由 + Top-K"机制把"参数量"和"激活量"解耦，这才是它和"加宽 FFN"的本质差别。换句话说，MoE 不是在"做更大的 FFN"，而是在"做更稀疏的 FFN"。

整个 MoE 层的前向传播可以用如下伪代码精确描述：

```text
# 输入：x.shape = [B, S, d]   (Batch × Seq × hidden)
# 假设：N_exp 个 expert，每个 token 选 k 个

# Step 1: Router 打分（轻量 MLP）
scores = Softmax(x @ W_router)        # [B, S, N_exp]

# Step 2: TopK 选择
topk_val, topk_idx = TopK(scores, k=k)   # [B, S, k]
gate_weights = Normalize(topk_val)        # [B, S, k]

# Step 3: Token Dispatch  (All-to-All)
#   按 expert_id 重排 token，跨 GPU 传输到对应 expert 所在设备
dispatched_x = Dispatch(x, topk_idx)     # 每张 GPU 收到分给本地 expert 的 token

# Step 4: Expert Forward（本地 GPU 内独立计算）
for e in local_experts:
    expert_out[e] = FFN_e(dispatched_x[e])

# Step 5: Token Combine (All-to-All 反向)
#   把 expert 输出送回 token 原属 GPU
combined = Combine(expert_out, topk_idx)  # [B, S, k, d]

# Step 6: Gate 加权聚合
y = sum( gate_weights[:, :, j] * combined[:, :, j, :]   # [B, S, d]
         for j in range(k) )
```

可以把上面这段伪代码再解读一遍：**Step 1~2 是路由决策阶段**，在每张 GPU 上独立完成，几乎不耗通信；**Step 3 和 Step 5 是两个对称的 All-to-All 通信阶段**，是整个 MoE 层最大的通信开销来源；**Step 4 才是真正的 "算力消耗" 阶段**，但相对通信往往不是瓶颈。换句话说，**MoE 的工程优化重点从来不在 expert 自身的 GEMM 上，而在如何让那两次 All-to-All 跑得更快、更早地与计算重叠**。这一观察会在第三章的 overlap 策略里被反复强化。

整个流程的精髓在于三个核心组件：

- **Expert**：通常是标准 FFN（Linear → SwiGLU/GELU → Linear），少数变体使用 Conv1D 或 GLU。Expert 是 MoE 的"参数容器"，绝大部分参数都集中在这里。
- **Router / Gate**：一个**极其轻量的 MLP**（通常一个线性层 `[d, N_exp]`），负责为每个 token 输出 N_exp 维分数，参数量相对 Expert 可以忽略不计。
- **Token Dispatcher**：负责把 token 按 expert_id 重排、打包、跨 GPU 通信、解包、回传——它**不是一个神经网络层，而是一套通信原语**，是工程实现的核心难点。

值得强调一点：**MoE 通常只替换 Transformer 中的 FFN 层，Self-Attention 不变**。这是因为 Attention 的 KV 计算具有"全局依赖"，而 FFN 是逐 token 独立的，天然适合稀疏路由。

### 2.2 关键参数定义

理解 MoE 必须区分两个参数量概念，这也是初学者最容易混淆的地方：

$$
\Theta_{t} = \Theta_{shared} + N_{exp} \times \Theta_{expert}
$$

$$
\Theta_{a} = \Theta_{shared} + k \times \Theta_{expert}
$$

其中：

- $\Theta_{t}$（**Total**）：模型的**总参数量**，决定了你需要多少显存来存模型；
- $\Theta_{a}$（**Activated**）：每个 token 实际**激活的参数量**，决定了一次前向计算的 FLOPs；
- $\Theta_{shared}$：共享部分，包括 Embedding、Self-Attention、LayerNorm、未被 MoE 替换的 FFN 等；
- $\Theta_{expert}$：单个 Expert 的参数量；
- $N_{exp}$：Expert 总数，$k$：TopK 中的 k。

来看几个真实模型的数字：

| 模型 | $\Theta_{t}$（总） | $\Theta_{a}$（激活） | 比值 | N_exp | k |
|------|------------|------------|------|-------|---|
| Mixtral-8x7B | 47B | 13B | ~3.6× | 8 | 2 |
| DeepSeek-V2 | 236B | 21B | ~11× | 160 | 6 |
| DeepSeek-V3 | 671B | 37B | ~18× | 256 | 8 |
| Qwen3-235B-A22B | 235B | 22B | ~10× | 128 | 8 |

可以看到，**$\Theta_{t}/\Theta_{a}$ 的比值通常在 10~30 倍之间**，这正是 MoE "稀疏激活"的核心收益：模型容量可以做到很大，但每次推理的算力消耗只取决于激活部分。

为了让这个收益更加具象，我们不妨做一个 **Dense vs MoE 的成本对比**。假设我们要训练一个"能力对齐 70B 稠密模型"的模型，有两种选择：**A 方案是直接训练 70B Dense，B 方案是训练一个 200B/15B 的 MoE**（总参 200B，激活 15B）。两者的账如下：

| 项 | 70B Dense | 200B/15B MoE |
|------|-----------|--------------|
| 训练 FLOPs / token | $\propto$ 70B | $\propto$ 15B，**约 Dense 的 1/5** |
| 模型权重显存 | 140 GB (BF16) | 400 GB (BF16)，**是 Dense 的 ×2.8** |
| Adam 优化器状态 | 280 GB | 800 GB，**同样 ×2.8** |
| 训练同等 token 所需 GPU 时 | 约 X | 约 0.3X |
| 推理延迟 | $\propto$ 70B | $\propto$ 15B，**几乎减半** |
| 推理部署 GPU 数 | 约 4 张 A100 | 约 16 张（权重装下来就要 8张）|

换句话说，MoE 赢在"训练和推理的算力成本"，输在"部署的显存成本"。如果你的业务场景是"大规模预训练 + 大规模推理服务"，MoE 按 token 摄成本下降 60%~70% 是常见的；但如果你的场景是"小规模微调、边缘部署"，70B Dense 反而是更之选择。**这就是 MoE 最难处理的 trade-off：你赚的是“大规模场景下的边际成本”，输的是“小规模场景下的固定成本”**。

这个账还有一个更深的拓展：**为什么业界主流 MoE 都选择 $\Theta_{t}/\Theta_{a}$ 在 8~20 倍这个区间？**比值太低（如 4 倍），MoE 带来的算力节省不够明显，不足以补偿额外的负载均衡、通信、调度成本；比值太高（如 50 倍），expert 需要高度专业化，对数据质量、路由质量要求极高，实际收益会遭遇边际递减。从工程经验看，$\Theta_{t}/\Theta_{a} \approx 10$ 是一个“甜点”，既能拿到足够的算力收益，又不至于让负载均衡变得难以控制。

但要小心：**不同的资源维度对应不同的 Θ**，混淆会导致严重的容量规划错误。下表给出一个清晰的对照：

| 资源维度 | 对应参数量 | 解释 |
|---------|-----------|------|
| 推理 FLOPs / token | $\Theta_{a}$ | 只激活 k 个 expert，计算量按激活算 |
| 训练 FLOPs / token | $\Theta_{a}$（前向）+ $\Theta_{a}$（反向）| 同上 |
| 模型权重显存 | $\Theta_{t}$ | 所有 expert 都得存到显存 |
| 优化器状态显存 | $\Theta_{t}$（Adam: 2×$\Theta_{t}$） | momentum、variance 都要全量存 |
| 激活值显存 | $\propto \Theta_{a}$ | 只有被激活的 expert 才有激活值 |
| All-to-All 通信量 | $\propto k$ | 只与 TopK 的 k 有关，与 N_exp 无关 |

也就是说：**算的是 $\Theta_{a}$，存的是 $\Theta_{t}$**。这正是 MoE 训练为什么"显存压力远大于算力压力"的根本原因——你不能像稠密模型那样用"模型大小 ≈ 算力消耗"这种朴素直觉来做容量规划。

### 2.3 Gating 机制

Gating 是 MoE 的"灵魂"。它决定了哪些 token 被送到哪些 Expert，**直接决定了模型质量、负载均衡和通信开销**。下面我们详细讲解三种主流路由方式。

#### 2.3.1 Top-K Routing

最经典也是目前主流的方案。Router 是一个简单的线性层，对每个 token 计算它与每个 Expert 的"亲和度"分数：

$$
s_{i,t} = \text{Sigmoid}(u_{t}^{T} \cdot e_{i}), \quad i \in \{1, 2, \dots, N_{exp}\}
$$

其中 $u_{t}$ 是第 $t$ 个 token 的隐状态，$e_{i}$ 是第 $i$ 个 Expert 对应的 router 权重向量（一列 $W_{router}$）。

得到 N_exp 维分数后，选出最大的 k 个，再做 softmax 归一化作为 gate 权重：

```text
scores      = sigmoid(x @ W_router)              # [B, S, N_exp]
topk_val,
topk_idx    = topk(scores, k=k, dim=-1)          # [B, S, k]
gate_weight = softmax(topk_val, dim=-1)          # [B, S, k]
```

不同模型的 k 选择：

- **Switch Transformer (Google, 2021)**：$k=1$，最简单粗暴，每个 token 只走一个 expert。优点是 dispatch 极简、通信量最小；缺点是 gate 噪声大、训练不稳定，需要大量 trick。
- **GShard (Google, 2021) / Mixtral / Qwen-MoE**：$k=2$。两个 expert 的输出加权平均，比 k=1 更稳定，是早期主流。
- **DeepSeek-V2/V3**：$k=6$ 或 $k=8$。配合大量小 expert（细粒度切分），用更高的 k 换取更精细的"专家组合"表达力。

一个关键工程细节：**TopK 必须在前向时确定，反向不能动**。否则梯度路径会发生改变，导致训练不收敛。这也是为什么 TopK 是个**不可微算子**——梯度只通过 gate_weight 的 softmax 流过，不通过 topk_idx 流过。

#### 2.3.2 Expert Choice Routing

由 Google 在 2022 年提出，是一种"反向选择"的路由方式：

- **Top-K Routing**：token 选 expert，每个 token 选 k 个 expert；
- **Expert Choice**：expert 选 token，每个 expert 从所有 token 中选 capacity 个最匹配的。

伪代码对比：

```text
# Top-K (token chooses expert)
for each token t:
    pick top-k experts with highest score[t, :]

# Expert Choice (expert chooses token)
for each expert e:
    pick top-capacity tokens with highest score[:, e]
```

它的天然优势：**完美负载均衡**。因为每个 expert 选定的 token 数都恰好等于 capacity，**不存在 expert collapse 问题，也不需要辅助损失函数**。

但它也有明显劣势：

- **某些 token 可能被 0 个 expert 选中**（特征不显著、与所有 expert 都不太匹配），等同于跳过该 token 的 FFN 计算；
- **某些 token 可能被多个 expert 选中**（k 不固定），破坏了"每个 token 激活参数量恒定"的假设；
- **不适合自回归生成**：推理时每个 token 必须立即决定路由，但 Expert Choice 需要看到一个 batch 才能比较 token 间的分数。

因此 Expert Choice 主要用于训练或非自回归场景，主流的对话模型仍然采用 Top-K Routing。

**工程选择建议（Top-K vs Expert Choice）**：如果你要训练一个 **面向推理场景**的对话模型（LLM 对话、code completion、实时生成），**几乎只能选 Top-K**——因为 Expert Choice 需要看到一个完整 batch 才能做决策，而推理时 batch=1 带不起“为了选择一个 token就要等一个全 batch”这种延迟。如果你是做 **纯供训练的实验性预训练**，考虑负载均衡是个重要考点，Expert Choice 可以考虑作为辅助方案。总之业界主流还是 Top-K：**Mixtral、Qwen-MoE、DeepSeek-V2/V3、GPT-4 猜测架构都是 Top-K 路线**。

#### 2.3.3 Hash Routing

最极端的方案：**完全确定性、无可学习参数**。

$$
\text{expert\_id}(t) = \text{Hash}(\text{token\_id}_{t}) \mod N_{exp}
$$

伪代码：

```text
# 每个 token 的 expert_id 仅由其 token_id 决定
for each token t:
    expert_id = hash(t.token_id) % N_exp
    dispatch t to expert[expert_id]
```

优势：**绝对负载均衡**——只要词表和数据分布合理，每个 expert 收到的 token 数严格相等，不需要辅助损失，不需要 Capacity Factor，dispatch 逻辑极简。

劣势同样明显：**完全忽略语义**。两个语义相似的 token（"猫"和"狗"）可能被路由到完全不同的 expert，让 expert 失去"专业化"的机会。实测效果通常差于 Top-K Routing 几个点的 PPL，因此除了在某些研究性 baseline 外，工业界很少采用。

**三种 routing 方案的最终对比**：为了让读者有一个一页总结，我们把三者放在同一张表里横向对比：

| 维度 | Top-K Routing | Expert Choice | Hash Routing |
|------|---------------|---------------|---------------|
| 可学习 | 是 | 是 | 否 |
| 语义性 | 高 | 高 | 无 |
| 负载均衡 | 需辅助损失 | 天然均衡 | 绝对均衡 |
| 适合推理 | 是 | 否（需 batch）| 是 |
| 实现复杂度 | 中 | 高 | 低 |
| 工业主流 | 是 | 仅训练场景 | 几乎不用 |

从这张表可以看出一个有意思的规律：**“效果”与“均衡性”在路由设计上是一对折中**。Hash Routing 换到了绝对均衡，却完全丢了语义；Expert Choice 保留了学习能力并获得均衡，却在推理上不可用；Top-K 保留了语义和推理可用性，却需要额外的负载均衡机制去“打补丁”。这也从另一个角度解释了为什么业界主流选择 Top-K——**不是因为它完美，而是因为它的“不完美”是可以被工程手段（辅助损失、bias 校正、capacity）补足的**。

### 2.4 负载均衡问题

这是 MoE 训练**最大的工程难题**。如果不加任何约束，几乎所有 MoE 模型都会迅速塌缩成"少数 expert 永远忙、大多数 expert 永远闲"的状态。

#### 2.4.1 Expert Collapse 现象

**Expert Collapse** 指的是部分 expert 在训练早期就因为得不到 token 而停止学习，从而陷入"永远不被选中"的死循环。其底层是一个典型的**正反馈级联**：

```text
[初始随机]  Expert_3 略微"好用"
     ↓
[Router 学习] Router 给 Expert_3 更高分数
     ↓
[路由集中] 更多 token 被分配给 Expert_3
     ↓
[Expert_3 学得更好] 更高质量的输出
     ↓
[Router 更偏向 Expert_3]
     ↓                          ┌──────────────────────────┐
     │                          │ Expert_5,7,11... 永远拿不到 token │
     ↓                          │ 参数停止更新              │
[最终] 80% expert 处于"僵尸"   │ 等同于浪费的参数容量     │
状态，模型实际容量只剩 20%     └──────────────────────────┘
```

实测数据：**不加任何均衡约束时，Switch Transformer 训练 1k step 后通常只有 30~40% 的 expert 还在被激活**，其他全部塌缩。这意味着你训练的不是 1.6T 模型，而是一个 ~600B 的"半成品"。

**实际中如何发现与诊断 Expert Collapse？**经验丰富的 MoE 训练工程师会在训练的前 500~1000 step 内就盯上以下几个监控指标。第一个是 **expert utilization rate**，在每个 step 统计各个 expert 接收的 token 数，画出其最大/最小比值（max/min ratio）——**如果这个比值 >5，基本可以判定已经出现 collapse 趋势**。第二个是 **routing entropy**，计算 router 在整个 batch 上的概率分布的香农熵，一个健康的 MoE 应该接近 $\log(N_{exp})$，如果掉到 70% 以下就是危险信号。第三个是 **dead expert count**，连续 N 个 step 接收 token 数为 0 的 expert 数量，超过 N_exp 的 5% 就需要警报。

踩坑警示：在实际训练中，如果你发现 loss 出现"以阶梯状下降"的奇怪现象（下降一点点然后抹平，重复几轮），很可能不是学习率的问题，而是 **expert collapse 正在进行性发生**。每几百 step 丢掉几个 expert，loss 就会在该 expert 负责的 token 上跳一下。这种问题不能用调学习率解决，只能提高 $\alpha$、退回检查点重训。

#### 2.4.2 辅助损失函数（Auxiliary Loss）

GShard 与 Switch Transformer 提出的经典方案：在主任务损失之外，**额外加一个鼓励 token 均匀分配的损失项**。

$$
L_{aux} = \alpha \times N_{exp} \times \sum_{i=1}^{N_{exp}} f_{i} \cdot P_{i}
$$

其中：

- $f_{i}$ = 第 i 个 expert 实际接收的 **token 比例**（统计量，由 TopK 后实际计数得出）；
- $P_{i}$ = 第 i 个 expert 的**平均 router 概率**（由 softmax 输出对 batch 求平均）；
- $\alpha$ = 损失系数，通常 0.01 ~ 0.1。

直觉理解：

- 如果路由完全均匀，$f_{i} \approx 1/N_{exp}$，$P_{i} \approx 1/N_{exp}$，$\sum f_{i} P_{i} \approx 1/N_{exp}$（最小值）；
- 如果某个 expert 独大，其 $f_{i}$ 和 $P_{i}$ 都接近 1，$\sum f_{i} P_{i}$ 急剧增大，损失被惩罚。

这个公式有个聪明的设计：**$f_{i}$ 是不可微的（来自 TopK 计数），但 $P_{i}$ 是可微的（来自 softmax 概率）**。所以梯度只通过 $P_{i}$ 流回去，但优化目标本身确实在惩罚 $f_{i}$ 的不均衡——一种"借不可微指标引导可微参数"的工程技巧。

伪代码实现：

```text
# 一次前向后的统计
# scores: softmax 后的概率 [B, S, N_exp]
# topk_idx: TopK 选中的 expert id [B, S, k]

P = scores.mean(dim=[0, 1])                       # [N_exp] 平均概率
mask = one_hot(topk_idx, N_exp).sum(dim=2)        # [B, S, N_exp]
f = mask.float().mean(dim=[0, 1])                 # [N_exp] 实际频率

L_aux = alpha * N_exp * (f * P).sum()
total_loss = L_main + L_aux
```

但 $\alpha$ 的选择是个**两难**：

- $\alpha$ 太小：均衡约束不足，仍然 collapse；
- $\alpha$ 太大：辅助损失主导优化方向，router 强行均匀化，**牺牲了路由的语义性**（变得越来越像 Hash Routing），主任务效果下降。

实践中，$\alpha$ 通常在 0.001 ~ 0.01 之间精细调参，且在训练过程中经常需要 warmup 与 decay。

**辅助损失的“梦魇期”**：在训练中期（约 30%–60% steps），aux loss 常常会出现一个有趣的现象——主 loss 在下降，但 aux loss 却在缓慢上升，表现为 router 在“反抗均衡”。**这是一个危险信号**：表现为主任务梯度与均衡信号出现了对抗，几乎达到互相抵消的状态。经验上这种状态多动一动就会导致 collapse。常见的应对手法是：在这个阶段 **逆向调低 $\alpha$ 一半**（从 0.01 变 0.005），让主任务优先取得主导权；到了训练末期再阶梯恢复。这是 DeepSeek-V2 之前几乎所有主流 MoE 都采用过的“人工介入”。DeepSeek-V3 取消辅助损失的动机，正是为了从根本上解决这种“主任务与均衡互相抵消”的问题。

#### 2.4.3 无辅助损失方案（DeepSeek-V3）

DeepSeek-V3 提出了一个更优雅的方案：**完全去掉辅助损失，改用动态 bias 校正**。核心思想是——既然辅助损失会污染主任务梯度，那就**不通过梯度调整路由**，而是用一个"人工干预"的 bias 直接校正分数：

$$
s_{i,t}^{\,'} = s_{i,t} + b_{i}, \quad \text{TopK on } s'^{}
$$

其中 $b_{i}$ 是每个 expert 独立的 bias，**只参与 TopK 选择，不参与 gate 权重计算**。

伪代码：

```text
scores       = sigmoid(x @ W_router)              # [B, S, N_exp]
biased_scores = scores + bias                     # [B, S, N_exp]，bias 仅用于 topk 选择
topk_val, topk_idx = topk(biased_scores, k=k)     # 选 k 个

# 关键：gate_weight 用原始 scores，不用 biased_scores
gate_weight  = softmax(scores.gather(topk_idx))   # [B, S, k]
```

每个 step 末尾，根据本 step 的 expert 负载情况动态调整 bias：

```text
# step 末尾的 bias 更新
expected = total_tokens / N_exp                   # 期望每个 expert 接收的 token 数
for i in range(N_exp):
    actual = count_tokens_dispatched_to(expert_i)
    if actual < expected:
        bias[i] += gamma   # 这个 expert 用得太少，下一 step 提升它的分数
    else:
        bias[i] -= gamma   # 这个 expert 用得太多，压一压
```

$\gamma$ 是一个很小的步长（如 0.001）。这样做的好处：

- **不污染主任务梯度**：bias 是非梯度更新，主损失曲线干净；
- **不影响 gate 权重的语义**：路由的"专家选择"仍然忠于 router 学到的真实分数；
- **均衡效果立竿见影**：通常 1k step 内就能让 expert 利用率均匀。

DeepSeek-V3 用这套方案训练了 256 个 expert 的 671B MoE，**实测 expert 激活率均匀且无任何辅助损失**——这是 2024 年 MoE 工程领域最重要的一个突破之一。

可以看到一个清晰的趋势：**从"少而大"走向"多而小"**。早期 GShard / Mixtral 用 8 个大 expert，每个几 B 参数；DeepSeek-V3 / Qwen3 用 128~256 个小 expert，每个只有几百 M 参数。**更细的粒度 + 更高的 k**，让"专家组合"的表达空间从 $\binom{8}{2}=28$ 跃升到 $\binom{256}{8} \approx 4 \times 10^{14}$，模型表达能力大幅提升。

趋势分析：为什么会从"少而大"转向"多而小"？背后有三个驱动力。**第一是表达空间**：expert 粒度越细，模型能“组合出的专家意见”越多，在同样总参数量下能表达的函数空间越大。**第二是负载均衡**：N 个 expert 的均衡要求需要所有 expert 在一个 batch 里收到差不多多的 token，这在大 expert 场景下是一个“广义大均值”（每个拿到几千 token），但在小 expert 场景下只需要“狭义大均值”（每个拿到几百 token），反而更容易达到。**第三是推理友好**：小 expert 可以使用更细粒度的 GroupedGEMM，在 H100/H800 上能获得更高的 SM 占用率。

这条演化路径，正是接下来第三章 Expert Parallelism 工程挑战的来源——更细的 expert 意味着更复杂的 All-to-All、更敏感的负载均衡、更精密的通信调度。

**用一组数字感受规模升级**：从 Mixtral 的 8 expert × 7B 走到 DeepSeek-V3 的 256 expert × 2B，单层 expert 总参数量从 56B 涨到 512B（×9 倍），单 token 激活的 expert 从 2 个涨到 8 个（×4 倍），单层 All-to-All 通信量按 $k \times B \times S \times d$ 估算也线性放大 4 倍以上。再叠加层数从 32 涨到 61，整个网络的 expert 参数量从 1.8T 量级走到 31T 量级。**“多而小”不是把模型变小，而是把通信压力和调度复杂度推上一个新的台阶**——这就解释了为什么 2024 年之后 MoE 训练框架几乎全部转向了 fused permutation、device-limited routing、计算通信深度重叠这些之前被认为“锦上添花”的技术：在 256 expert 的世界里，它们已经从可选项变成了必选项。

**本章小结**：第二章我们把 MoE 的“是什么”从架构、Gating、负载均衡、演化历程四个维度讲齐。一个清晰的脉络是——**算法上 MoE 解决了“扩参数”的问题，但把代价转移到了系统层面**：路由、通信、显存、负载均衡，每一个都是工程师真正要去和硬件博弈的战场。从下一章开始，我们把视角从“算法”切换到“系统”，看看 Expert Parallelism 如何把这些战场组织成一套可执行、可调优、可扩展的训练流水线。

## 三、Expert Parallelism 核心原理

前面两章我们解决了"是什么"和"为什么"的问题：MoE 是一种条件计算结构，靠稀疏激活来打破"参数 = 计算"的魔咒。但当我们真正把它放到几百上千张 GPU 上训练时，会发现一个非常棘手的事实——**MoE 的算法逻辑只占总开销的不到一半，剩下的全是工程问题**。Router、Dispatch、Combine、Capacity、Token Drop、All-to-All、Overlap……这些词每一个背后都对应着"少做几个百分点优化，整个训练就要多烧几十万块钱"的代价。

本章会从一个完整的 EP 训练循环入手，一步步拆开每一步的实现、瓶颈、优化点，最后落到 DeepSeek 级别系统才会用的 MegaKernel 终极方案上。

### 3.1 EP 完整训练循环

我们先用通俗的语言把整个流程讲一遍：一个 batch 的 token 进入 MoE 层，**第一件事是 Router 给每个 token 投票，决定它该去哪几个 expert**；第二件事是把 token 按目的地打包，通过 All-to-All 通信发到对应的 GPU；第三件事是各 GPU 让本地的 expert 算一遍 FFN；第四件事是把结果原路送回；第五件事是把 k 个 expert 的输出按 router 给的权重加权合并。这五步缺一不可，少了任何一步语义都不完整。

下面是这五步的伪代码：

```text
# ============================================================
# EP Forward Pass —— 完整训练循环（单层 MoE）
# ============================================================
# 输入：tokens_in  shape = [B*S, d]   # B=batch, S=seq_len, d=hidden
# 设定：N_exp 个 expert 均分到 EP_world_size 张 GPU
#       每张 GPU 上有 N_exp / EP_world_size 个 local expert
# ------------------------------------------------------------

# Step 1. Router 计算（每张 GPU 独立做）
#   - 每个 token 算出一组打分，TopK 选出 k 个 expert
scores       = tokens_in @ W_router            # [B*S, N_exp]
topk_score   = topk(scores, k, dim=-1).values  # [B*S, k]
topk_idx     = topk(scores, k, dim=-1).indices # [B*S, k]
gate_weight  = softmax(topk_score, dim=-1)     # [B*S, k]，求和=1

# Step 2. Token Dispatch（跨 GPU 的 All-to-All）
#   - 按 expert_id 把 token 排序：去 expert_0 的连续放在一起，去 expert_1 的接着……
#   - 然后调用 NCCL all_to_all，把 token 发到 expert 所在的 GPU
perm_idx     = argsort(topk_idx.flatten())     # [B*S*k]
tokens_perm  = tokens_in.repeat_k()[perm_idx]  # [B*S*k, d]
send_counts  = histogram(topk_idx, bins=N_exp) # 每个 expert 该收多少 token
recv_counts  = all_to_all(send_counts)         # 提前同步：本 GPU 会收到多少
tokens_local = nccl.all_to_all(tokens_perm,
                               send_counts,
                               recv_counts)    # [N_local_tok, d]

# Step 3. Expert Compute（GPU 本地，无通信）
#   - 每张 GPU 只对自己 owns 的 local expert 做 FFN
out_local = empty_like(tokens_local)
for e in local_experts:
    mask = (recv_expert_id == e)
    out_local[mask] = experts[e](tokens_local[mask])  # GEMM(d, 4d, d)

# Step 4. Token Combine（All-to-All 反向）
#   - 把 expert 输出送回原 GPU，并恢复原始 token 顺序
tokens_back  = nccl.all_to_all(out_local,
                               recv_counts,
                               send_counts)    # 方向反过来
inv_perm     = inverse_permutation(perm_idx)
tokens_unperm = tokens_back[inv_perm]          # [B*S*k, d]

# Step 5. Gate 加权合并
#   - 把同一 token 的 k 个 expert 输出按 gate 权重加和
tokens_unperm = tokens_unperm.reshape(B*S, k, d)
output        = (tokens_unperm * gate_weight.unsqueeze(-1)).sum(dim=1)
# 最终 output shape = [B*S, d]，与 tokens_in 同形状
```

**反向传播**走的是同一条路径，只是把 All-to-All 的方向倒过来：先反向 combine（把上游梯度按 gate 权重拆开 → All-to-All 把梯度送到 expert 所在 GPU），然后是 expert 的反向 GEMM，再 All-to-All 把 dx 送回，最后求 router 的梯度。整条路径的特点是 **All-to-All 出现 4 次**（前向 2 次 + 反向 2 次），这就是为什么"通信占比"是 EP 性能调优中最重要的指标。

**伪代码逐步拆解**：上面这五个步骤到底发生了什么？我们重新按双视角（计算路径 + 通信路径）讲一遍。**Step 1 Router**：是一个轻量级 GEMM，输入 [B*S, d]，输出 [B*S, N_exp] 的打分。在实际训练中这个 GEMM 几乎是免费的（不到 1ms），但选出的 topk_idx 决定了后面三个阶段的负载分布。**Step 2 Token Dispatch**：这是第一个跨 GPU 的调度点。需要先用 histogram 算出本 GPU 要发多少 token 给每个其他 GPU，再通过 NCCL all_to_all 同步这些计数，最后才能发起真正的 token 传输。**Step 3 Expert Compute**：本地 GPU 上一个或多个 expert 的 FFN，是唯一不跨卡的阶段。**Step 4 Token Combine**：是 Step 2 的反方向版本，完全镜像对称。**Step 5 Gate 加权**：是一个极简单的元素乘加求和，在 GPU 上几乎不占时间。

进一步看 **哪里是瓶颈**：从时间占比看，不同阶段在一个典型 8 卡 EP 训练中的展开可能是：Router ~1ms，Dispatch 通信 ~50ms，Expert 计算 ~80ms，Combine 通信 ~50ms，Gate 加权 ~1ms。这意味着 **两次 All-to-All 加起来占了近 55% 的时间**，这也是为什么应用里奇招各异的 overlap 策略都是必需的。如果你调试过 NSight Systems 或者 PyTorch profiler，在 EP 训练的 timeline 上会看到两个干净的"通信柱"，其间夹着一段"计算柱"，整个 MoE 层在 GPU 上凹凸有致地呈现"双峰骆"的形状。

### 3.2 Capacity Factor 与 Token Dropping

上面的伪代码里有一个被掩盖的难题：**前向开始时，没人知道每个 GPU 会收到多少 token**。Router 的输出是数据相关的，今天这个 batch 大家都喜欢 expert_0，明天又都涌向 expert_42。GPU 上的 buffer 必须**事先开好**，所以工程上必须做出妥协——给每个 expert 准备一个**固定大小**的容器，溢出的 token 直接丢弃。

#### 3.2.1 Capacity 的定义

首先算一下"理想容量"：如果 routing 完全均匀，每个 expert 平均会收到

$$\text{avg\_tokens\_per\_expert} = \frac{B \times S \times k}{N_{exp}}$$

个 token。但实际不可能均匀，所以工程上引入一个 **Capacity Factor (CF)** 系数，预留 buffer 大小为：

$$\text{capacity} = \text{CF} \times \frac{B \times S \times k}{N_{exp}}$$

举个具体的数：B×S=4096，k=2，N_exp=8（Mixtral 规模），avg = 4096×2/8 = 1024。如果 CF=1.25，那每个 expert 的 buffer 大小就是 1280。意思是"我相信你不会比平均水平多出 25% 的负载"。

#### 3.2.2 Token Dropping 的语义

当某个 expert 实际收到的 token 数 > capacity 时，**多余的 token 会被直接丢弃**——不会进入这个 expert，combine 阶段也不会有它的输出贡献。在数学上等价于把这些 token 在该 expert 上的 gate 权重置为 0：

```text
# combine 阶段的实际实现
if token_dropped(t, e):
    contribution[t, e] = 0          # 该 expert 对该 token 无贡献
else:
    contribution[t, e] = gate[t, e] * expert_e(t)
```

如果 token 的 k 个 expert 全被 drop（极端情况），这个 token 走的就是"残差路径"——只有 attention 出来的 residual，没有 MoE 贡献。这对模型质量是有伤害的，但偶尔发生影响有限。

#### 3.2.3 CF 的取舍

CF 太小或太大都是病：

- **CF 太小（< 1.0）**：必然丢 token，模型欠拟合，loss 曲线会比 dense baseline 高出可观的一截。
- **CF 太大（> 1.5）**：buffer 显存占用线性放大，相当于变相减小了 batch size，每张 GPU 能塞的样本变少，整体训练吞吐降低。

业界经验值是 **CF = 1.25**：基础 1.0 + 25% 冗余，刚好能覆盖大多数 batch 的 routing 抖动。当 N_exp 很大、k 很大（如 DeepSeek-V3 的 256/8）时，单 expert 平均负载本身就小，相对偏差更剧烈，普通 CF 已经压不住，所以需要更复杂的 device-limited routing（在 router 阶段就硬性限制每个 device 接收的 token 数）。

下表是几个代表性模型的实测情况：

| 模型 | N_exp | k | CF | 预期 Drop% |
|------|-------|---|------|-----------|
| Switch Transformer | 128 | 1 | 1.25 | <1% |
| GShard | 64 | 2 | 1.25 | <2% |
| Mixtral 8×7B | 8 | 2 | 1.3 | <1% |
| DeepSeek-V3 | 256 | 8 | device-limited | 0% |

DeepSeek-V3 的做法尤其值得一提：它放弃了"先 routing 再 drop"的妥协式方案，改成"在 routing 阶段就保证每个 device 不会超载"，因此 drop 率严格为 0。代价是 router 算法更复杂，需要做**device-balanced TopK**。

**实际训练中 CF 如何选择？**这是 MoE 调优中最高频被问到的问题之一。一个可以参考的决策流程是这样的：**如果你用的是辅助损失方案，且 N_exp 小于 32，可以从 CF=1.25 开始**；如果 N_exp 在 32~128 之间，推荐 CF=1.5，因为路由粒度变细后单 expert 均衡变难，需要多预留 buffer；如果 N_exp 超过 128（如 DeepSeek-V3 的 256），传统 CF 已经压不住 routing 抖动，这时推荐切换到 device-limited routing，从源头上避免超载。

**常见错误**：一个初学者常犯的错是把 CF 设得太小，比如为了"省显存"设为 0.8。表面上看起来一切正常，loss 也在下降，但实际上有 30%~50% 的 token 被静默丢弃了，模型准确率会隐性下降 1~2 个点，除非你主动打印 drop 率日志，否则根本发现不了。**在调试 MoE 训练时，务必把 drop 率加进 tensorboard 监控面板**，这比加什么 histogram 都重要。

另一个不太明显的踩坑是 **CF 应该在 forward 和 backward 中保持严格一致**。有些早期实现为了加速，在 forward 用 1.25、在 backward 用 1.0，这会导致梯度与前向的 topology 不一致，表现为训练 loss 偶发性 spike，且难以复现。调试这类问题需要仔细对齐两个方向的 capacity。

### 3.3 Token Permutation 优化

Token permutation 这个步骤听起来不起眼——不就是把 token 按 expert_id 重排一下吗？但它在朴素实现下会成为 EP 训练的隐形杀手。

#### 3.3.1 朴素实现为什么慢

```text
# 朴素 permutation：每个 token 单独搬运
for t in range(B*S):
    for j in range(k):
        e = topk_idx[t, j]
        offset = expert_offset[e]
        expert_buffer[e][offset] = tokens_in[t]   # 随机内存写
        expert_offset[e] += 1
```

这段代码有三个致命问题：

1. **完全随机的内存访问模式**：tokens_in 是连续的，但 expert_buffer[e] 在不同 expert 之间是分散的，每写一个 token 就要换一个 buffer 地址 → cache miss 极高。
2. **数据流不连续**：GPU 喜欢 "一段一段地搬"，朴素实现是 "一个一个地搬"，DRAM 带宽利用率不到 30%。
3. **kernel launch 开销**：在 PyTorch 实现中，每一次 indexing 都可能触发独立的 CUDA kernel launch，单层 MoE permutation 在小 batch 下能花掉 2-3ms。

#### 3.3.2 优化实现的关键技巧

Megatron / DeepEP / DeepSeek 的优化版本，思路完全不同：

```text
# 优化版 permutation（融合 kernel）
# Step A. 提前估算每个 expert 会收多少 token（直接 histogram，<10us）
send_counts  = histogram(topk_idx, bins=N_exp)

# Step B. 一次性 all_to_all 同步双方 counts
recv_counts  = nccl.all_to_all(send_counts)   # 通信只传 N_exp 个 int

# Step C. 预分配连续的大 buffer（DRAM 友好）
total_send   = send_counts.sum()
perm_buffer  = empty([total_send, d])

# Step D. 一个融合 kernel 完成 permutation + scatter
fused_permute_kernel(
    src       = tokens_in,        # 连续读
    dst       = perm_buffer,      # 连续写
    topk_idx  = topk_idx,         # 路由表
    send_off  = exclusive_scan(send_counts),
    block_dim = 256,              # 一个 thread block 处理 256 tokens
    use_smem  = True,             # 共享内存做 staging，避免 bank conflict
)
```

关键收益来自三点：

- **预分配连续 buffer**：DRAM 一次性按 page 对齐分配，写入是 coalesced。
- **融合 kernel**：把"找 expert + 算 offset + 拷贝数据"全部塞进一个 kernel，没有中间张量的物化。
- **bank conflict 规避**：在 shared memory 里 stage 数据时，按 32 个 bank 错位访问，避免同一 warp 内多个 thread 撞同一个 bank。

性能收益是数量级的：**单层 MoE permutation 从 2-3ms 降到 0.1-0.3ms**，对 61 层的模型就是几百 ms 的差距，乘以训练几百万 step，总开销能差出几个 GPU-月。

**这里有个隐性经验值值得强调**：在中等规模 MoE（如 N_exp=8~32）的训练中，如果你用 PyTorch 原生算子拼出 permutation，应该能在 NSight 里看到一条明显的"蜗牛路径"：几乎所有 MoE 层都会出现一个 2~3ms 的空隔，这就是 permutation 在"拖后腿"。如果看到了这个现象，不要费力去调 NCCL 策略了，直接换为 Megatron 的 fused permute kernel 或 DeepEP 的 bare-metal 实现，能一口气拿回 30%~50% 的 GPU 利用率。

**不同场景下的优化优先级**：如果 batch size 越小、seq len 越短（如 微调场景），permutation 的相对开销越高，优化收益越大；反之，如果你的 batch size 已经走到几百万 token（如大规模预训练），permutation 的相对占比本身就会下降，这时优先级反而应该让位给 All-to-All 优化。工程实践中**务必先 profile 再优化**，不要默认 permutation 一定是瓶颈。

### 3.4 All-to-All 通信详解

如果说 permutation 是 EP 的隐形杀手，那 All-to-All 就是显性杀手——它通常吃掉 EP 总时间的 30%~50%。理解它的成本模型和算法选择，是做好 EP 性能调优的前提。

#### 3.4.1 通信成本模型

单次 All-to-All 的总数据量公式是：

$$V_{a2a} = 2 \times k \times B \times S \times d \times \text{bytes}$$

这五个系数各有含义：

- **2**：dispatch + combine，一次 forward 要走两个方向的 All-to-All。
- **$k$**：每个 token 复制 k 份，分别发往 k 个 expert（这里假设 k 个 expert 不在同一张 GPU 上）。
- **$B \times S$**：一个 batch 的总 token 数。
- **$d$**：hidden dimension，决定每个 token 的字节数。
- **bytes**：数据类型大小，BF16 = 2 字节，FP8 = 1 字节。

#### 3.4.2 实战数值：DeepSeek-V3 规模

带入一组真实数字感受一下规模：d = 7168，k = 8，B×S = 4M（4 百万 token），bytes = 2（BF16）：

```text
单层单次 All-to-All：
    V = 2 × 8 × 4M × 7168 × 2
      ≈ 920 GB

61 层 × 一次 forward + 一次 backward ≈ 920 GB × 61 × 2
      ≈ 112 TB / iteration

如果 step 时间目标是 4 秒：
    所需有效带宽 = 112 TB / 4s
                ≈ 28 TB/s
```

这是一个非常恐怖的数字。NVLink 4.0 单链路 900 GB/s，需要把所有 GPU 的 NVLink + IB 带宽全部打满，才能勉强追平。这就是为什么 DeepSeek 必须自己写 DeepEP、写 MegaKernel——商用 NCCL 在这种规模下打不到理论上限。

**对比小模型场景**：换个负担轻得多的场景，比如 Mixtral-8x7B 的训练，d=4096，k=2，B×S=128k token/batch，BF16：

```text
单层单次 All-to-All：
    V = 2 × 2 × 128k × 4096 × 2
      ≈ 4 GB

32 层 × 一次 forward + 一次 backward ≈ 4 GB × 32 × 2
      ≈ 256 GB / iteration

step 时间目标 0.5 秒：
    所需有效带宽 = 256 GB / 0.5s
                = 512 GB/s
```

可以看到，**同样都是两次 All-to-All，每 TB 的成本在 Mixtral 上仅是几百 GB/s**，节点内 NVLink 完全够用，根本不需要什么 "特殊优化"。**但到了 DeepSeek-V3 这个规模，同样的两次 All-to-All 要追 28 TB/s，这不是调参能解决的，是架构问题**。这个对比说明了为什么 MoE 的通信优化会随模型规模出现“质变”：在十几 B 激活参数以下是调优问题，在几十 B 激活参数以上是架构问题。

**从带宽预算反推集群选型**：这里隐含一个重要的集群选型准则——**在决定 MoE 模型规模之前，应该先算一算集群能提供多少有效带宽**。如果你补充的带宽达不到模型需求，那么不管调什么 NCCL 参数、写什么融合 kernel都只是杯水车薪。一个实用的准则是：**集群总带宽 ≥ 10× 模型单 step All-to-All 总量**。还达不到这个下限，最佳选择是要么减少 expert 数量，要么加大 hidden dim 以提高计算/通信比。

#### 3.4.3 算法选择

NCCL 内部对 All-to-All 有三种实现策略：

| 算法 | 适用规模 | 通信跳数 | 带宽利用率 | 备注 |
|------|----------|----------|------------|------|
| Direct All-to-All | 节点内 (≤8 GPU) | 1 跳 | 中等 | n² 条点对点连接，延迟最低 |
| Ring All-to-All | 中等集群 | n-1 跳 | 接近 100% | 环形传递，单次延迟较高 |
| Tree All-to-All | 超大集群 | log(n) 跳 | 较高 | 分层聚合，适合 thousands of GPU |

NCCL 会根据拓扑自动选择：

- 节点内 8 GPU：用 NVLink Direct，因为 NVLink 形成 fully-connected 拓扑。
- 跨节点中等集群（几十到几百 GPU）：Ring 拿满 IB 带宽。
- 超大集群（1000+ GPU）：Tree 把延迟从 O(n) 降到 O(log n)。

实际工程中，DeepEP 在节点内会**绕过 NCCL 直接用 IBGDA + NVLink P2P**，把延迟从 NCCL 的 ~50us 压到 ~5us。

**怎么在自己的集群上验证这三种算法？**有三个常用手段。**第一**，打开 NCCL 的 `NCCL_DEBUG=INFO` 环境变量，在训练启动时 NCCL 会打印它选中的拓扑类型与算法。**第二**，用 `nccl-tests` 中的 `alltoall_perf` 脚本跨不同规模跑一道，对比实测带宽与硬件上限，如果跨节点只能跑出 60% 以下，跳不出去就是拓扑错位，跳出去了就是 NCCL 选错了算法。**第三**，针对犹豫不定的中等集群，可以用 `NCCL_ALGO=Ring` 或 `NCCL_ALGO=Tree` 强制选择一种，再运行一个 50 step 的 dryrun 对比 step 时间。实践上这三个手段能让你成为一个集群上“定义最佳 NCCL 配置”的人，而不是“默认配置的使用者”。

### 3.5 计算-通信重叠策略

看完前面的成本分析就能发现一个朴素的事实：**EP 训练时如果通信和计算严格串行，GPU 一半时间是闲着的**。把通信藏到计算后面是必修课。下面我们看四种由浅到深的方案。

#### 3.5.1 方案 1：朴素串行（Baseline）

```text
# 一层 MoE 的时间线
Dispatch  ████████████                     100ms
Compute              ████████████████████   200ms
Combine                                  ████████████   100ms

总时间 = 100 + 200 + 100 = 400ms
GPU 占用率 ≈ 50%（200/400），剩下 50% 时间 GPU 在等通信
```

这是教学版本，几乎所有 MoE 论文里画的都是这个图。但它在生产环境下是不可接受的。

**适用场景**：只适用于演示代码、单元测试、调试里用一用。**实际生产训练中从不采用串行方案**，因为 50% 的 GPU 闲置意味着 50% 的电费被浪费。**实现难度**：最低，只要能调通 NCCL 就能跑。

#### 3.5.2 方案 2：微批次流水线 (Micro-batch Pipeline)

把一个大 batch 拆成 m 个 micro-batch，让 MB1 的 compute 和 MB2 的 dispatch 并行：

```text
# 4 个 micro-batch 流水线
MB1: D1 ████   C1 ████████   X1 ████
MB2:           D2 ████        C2 ████████   X2 ████
MB3:                          D3 ████       C3 ████████   X3 ████
MB4:                                        D4 ████       C4 ████████   X4 ████

稳态阶段每段都被压满，但单次 step 时间不变（因为最后一段尾巴还是要走完）
```

这种方案的好处是**实现简单**——只要把 batch 切片，让训练框架按 micro-batch 调度即可。坏处是**单 batch 时间没有变短，整体吞吐才提升**。在 Megatron-LM 的早期 EP 实现中，这是默认方案。

**适用场景**：适用于能够容忍“训练吞吐够快、但单个 step 还是偏慢”的场景，以及在 PP 与 EP 同时开启、本身就需要 micro-batch 拆分的大集群。**实现难度**：中，需要主动调整训练环路逻辑、处理梯度累加，但不需要代码 stream 、CUDA event。

#### 3.5.3 方案 3：分块融合 Kernel (Tiled Overlap)

这是当前主流的方案。核心思想是把一个大 batch 在 token 维度上切成 chunk（典型 256~1024 tokens），让"上一个 chunk 在算"和"下一个 chunk 在传"同时发生：

```text
# n=2 chunks 的时间线
chunk_0:  D0 ████   C0 ████████████   X0 ████
chunk_1:           D1 ████             C1 ████████████   X1 ████
                   └──────── 两段 dispatch / compute / combine 重叠 ────┘

稳态下：dispatch、compute、combine 三者并行
新总时间 ≈ max(总通信时间, 总计算时间) + tail
         ≈ max(200ms, 200ms) + 50ms ≈ 250ms
相对 baseline 加速 1.6×
```

实现要点：

- chunk 大小要选好，太小 kernel launch 开销占比高，太大重叠粒度不够。
- 需要**多 stream**支撑：通信走一个 stream，计算走另一个 stream，CUDA event 同步。
- buffer 要双倍开（double buffering），允许 chunk_0 的 buffer 还在被 NIC 读时，chunk_1 的 buffer 已经开始被 compute 写。

Megatron-Core、vLLM、SGLang 等主流系统的 EP 优化都用这个方案。

**适用场景**：适用于中大规模训练（几十到几百张 GPU）、对单 step 时间有明确要求、且集群不是 H800/H100 的场景。**实现难度**：高，需要熟悉 CUDA stream、event、double buffering、多个 NCCL communicator 的调度，一个均有经验的工程师初次实现需要 2-4 周。但一旦实现正确，这是"性价比最高"的优化方案。

#### 3.5.4 方案 4：MegaKernel（DeepSeek 终极方案）

DeepSeek 把上面的方案再推到极致——**整个 MoE 层只 launch 一个 kernel**，dispatch / compute / combine 在 kernel 内部以 tile 为粒度异步流水：

```text
# MegaKernel 内部的 tile-level 流水（伪代码）
__global__ void moe_megakernel(...) {
    for tile in tiles:                    # 一个 thread block 负责一组 tile
        # 分三组 warp 并行干活
        if warp_group == 0:
            load_from_nic(tile)           # IBGDA 拉远端 token
        if warp_group == 1:
            compute_tile(tile)            # tensor core GEMM
        if warp_group == 2:
            store_to_nic(tile)            # IBGDA 推回结果
        async_barrier();                  # warp 间细粒度同步
}
```

它带来三个极致优势：

- **零 kernel launch overhead**：商用 NCCL 一次 All-to-All 至少要 launch 几十个内部 kernel，MegaKernel 只 launch 一次。
- **完美的 compute-comm overlap**：tile 级流水让 SM 上的 tensor core 和 NIC 同时满载。
- **绕开 NCCL，直接打 IBGDA**：通信延迟从 ~50us 压到 ~5us 量级。

代价同样极致：

- **极难 debug**：kernel 内部异步语义复杂，一个 race condition 可能要花几天定位。
- **硬件特定**：当前主要针对 H800/H100 + IBGDA + NVLink 4 优化，A100 没有 IBGDA 跑不动。
- **代码复杂度极高**：DeepEP 的 MegaKernel 单文件 3000+ 行 PTX/CUDA，普通工程师改不动。

**适用场景**：只适用于拥有 H800/H100 + IBGDA 集群且训练规模走到 1000+ 卡、且能提供专职 CUDA 专家团队维护的顶尖机构。**实现难度**：极高，除了资深 CUDA 专家之外，还需要熟悉 IBGDA、RDMA、PTX 资源调度，初次开发周期在月级以上。广大中小团队不需要也不能走这条路。

四种方案的总结对比：

| 方案 | 单 step 总时间 | 加速比 | 实现复杂度 | 典型代表 |
|------|---------------|--------|------------|---------|
| 朴素串行 | 400ms | 1.0× | 低 | 教学示意 |
| 微批次流水线 | 400ms（吞吐 ↑） | ~1.0× | 中 | 早期 Megatron |
| 分块融合 (Tiled Overlap) | ~250ms | 1.6× | 高 | Megatron-Core / SGLang |
| MegaKernel | ~200ms | 2.0× | 极高 | DeepSeek DeepEP |

至此，Expert Parallelism 的核心原理就拼齐了：**Router → Permutation → All-to-All Dispatch → Expert Compute → All-to-All Combine → Gate Combine**，每一步都有自己的瓶颈和对应的优化。Capacity Factor 解决“不知道每张卡会收多少”的问题；融合 kernel 解决 permutation 的内存访问问题；通信成本模型告诉我们硬件需求；四种 overlap 策略告诉我们怎样把通信藏到计算后面。下一章我们将进一步讨论“负载均衡”——当 routing 本身分布不均时，前面这套机器还能不能高效运转。

**本章小结与过渡**：第三章我们从一个完整的 EP 训练循环入手，拆开了每一步的实现、瓶颈、优化点。现在你应该能回答以下几个问题：（1）为什么 token permutation 需要一个单独的 fused kernel？（2）Capacity Factor 是怎么在 “显存”与“drop 损失” 之间取得平衡的？（3）Mixtral 规模与 DeepSeek-V3 规模的通信预算为什么会差上两个数量级？（4）为什么 MegaKernel 需要 SM specialization 这么极端的手段？如果这几个问题你都能讲清楚，EP 的底层逻辑就已经入门了。接下来的第四章，我们把视角拉到“多维并行”，看看 EP 在千卡集群中如何与 DP、TP、PP、SP 合作。
## 四、EP 与其他并行的组合

前面三章我们一直把 Expert Parallel 当作"独立维度"在讲。但在真实的千卡乃至万卡训练中，EP 从来不是孤军奋战——它必须与 DP、TP、PP、SP/CP 一起，组成一张多维度的并行网格。本章先把六维并行空间画清楚，再讲清楚每一维到底切的是张量的哪个轴；然后落到"进程组"这个工程概念上，理解为什么 8 张卡可以同时属于多个不同的通信组；最后讨论 Megatron 在 2024 年提出的 **Parallel Folding**，为什么它能解开 EP ≤ DP 的桎梏，让 EP 真正成为一个独立的并行维度。

### 4.1 六维并行空间

要理解 EP 在大模型训练里的位置，首先要把"维度"讲清楚。一个 Transformer 模型在训练时，张量大致有这几个轴：

```text
input.shape  = [B, S, H]        # batch, sequence, hidden
weight.shape = [H_in, H_out]    # hidden, hidden
moe_expert   = [N_exp, H, 4H]   # 专家维度（MoE 独有）
模型层数      = L                 # 深度方向
```

每一维都有一种对应的并行策略：

| 维度 | 并行名称 | 切的是什么 | 通信原语 | 主要代价 |
|------|---------|-----------|---------|---------|
| Batch (B) | DP (Data Parallel) | batch 维度，每卡完整模型副本 | AllReduce 梯度 | 跨节点带宽 |
| Hidden (H) | TP (Tensor Parallel) | 矩阵乘法的 hidden 维度 | AllGather / ReduceScatter | 节点内带宽 |
| Layers (L) | PP (Pipeline Parallel) | 不同 stage 在不同 GPU | P2P send/recv | 流水线气泡 |
| Sequence (S) | SP / CP (Sequence/Context Parallel) | 序列维度 | AllGather / All-to-All | 长上下文显存 |
| Experts (N_exp) | EP (Expert Parallel) | 专家维度，MoE 独有 | All-to-All token dispatch | 路由不均 |
| Model States | ZeRO (1/2/3) | 优化器/梯度/参数 | ReduceScatter + AllGather | 间接通信 |

把上面这些"维度"乘起来，就得到一个总的 rank 数关系：

$$
W = DP \times TP \times PP \times SP \times EP
$$

这里要特别注意：**这个等式表达的是"GPU 总数 = 各并行度的乘积"，但同一张 GPU 同时属于多个不同的并行组**。也就是说，rank=0 这张卡，可能在 DP 组里编号为 0，在 TP 组里编号为 0，在 EP 组里编号为 0——三个"0"指向的是三种不同语境下的同一张物理 GPU。

我们来举一个具体例子。假设有一台 8 卡机：

```text
world_size = 8
配置：DP=2, TP=2, PP=2, EP=2

校验：DP × TP × PP × EP = 2 × 2 × 2 × 2 = 16  ❌ 等等，不对

正确理解：
  - PP=2 把模型切成 2 段，每段需要 4 张 GPU
  - 每段 4 张 GPU 内部再做 DP=2 × TP=2
  - 这 4 张里如果开 MoE，可以再做 EP=2（与 DP 复用通信组）
  - 真正的等式应该是：world = DP × TP × PP（不含 EP）
                      EP 是从 DP 维度"借"出来的子集
```

注意这里出现了第一个微妙点：**EP 通常不是独立的乘子**，而是从 DP 维度切出来的一部分。这正是下一节要解决的关键问题。

**为什么要这么多维度？一个切蛋糕的比喻**。把一个 Transformer 模型想象成一块千层蛋糕：**DP** 是把蛋糕复制几份同时上桌、各人吃一份（负责不同的 batch）；**TP** 是把同一层蛋糕横着切成几块（把 hidden 维度拆到多卡上）；**PP** 是把蛋糕从上到下切成几层、不同的人吃不同层（不同 stage 上不同卡）；**SP/CP** 是把蛋糕进一步沿 token 方向切条（处理超长 context）；**EP** 则特别一点——它只切蛋糕上“MoE 那一层”里面装的几十个小香蕉（expert），其他层不动。这个比喻随便，但能帮你记住一件事：**五个维度切的东西完全不同，它们永远可以同时存在**。一个 1024 卡集群常见的设置，就是五个维度同时被拆到不同轴上。

### 4.2 进程组划分逻辑（Process Groups）

**进程组（process group）**是 NCCL/PyTorch 分布式通信的最小通信单元。简单说：只有属于同一个 process group 的 rank，才能互相通信；不同 group 之间的通信必须显式指定。

每种并行维度都对应一个 process group：

```text
dp_group = [..]    # 同一个 dp_group 内做 AllReduce 梯度
tp_group = [..]    # 同一个 tp_group 内做 AllGather/ReduceScatter
pp_group = [..]    # 同一个 pp_group 内做 P2P send/recv
ep_group = [..]    # 同一个 ep_group 内做 All-to-All
```

我们以 8 GPU 的设置为例，看一下"组"是怎么划分的。配置 `DP=4, TP=2, EP=2`（暂时忽略 PP）：

```text
8 张 GPU 编号: rank0 rank1 rank2 rank3 rank4 rank5 rank6 rank7

—— TP 组（同节点内 NVLink）——
tp_group_0 = [rank0, rank1]    # 物理上同一台机器
tp_group_1 = [rank2, rank3]
tp_group_2 = [rank4, rank5]
tp_group_3 = [rank6, rank7]

—— DP 组（跨 TP 组复制）——
dp_group_0 = [rank0, rank2, rank4, rank6]   # 各 tp_group 的第 0 号
dp_group_1 = [rank1, rank3, rank5, rank7]   # 各 tp_group 的第 1 号

—— EP 组（从 DP 借子集）——
EP=2 的含义：把 DP=4 切成 2 个 EP 组，每组内做 All-to-All
ep_group_0 = [rank0, rank2]   # 这两个 rank 之间 dispatch token
ep_group_1 = [rank4, rank6]
ep_group_2 = [rank1, rank3]
ep_group_3 = [rank5, rank7]
```

把"组"翻译成物理直觉：

- **TP 组放在同一个 NVLink domain**：因为 TP 每层都要做 AllGather，通信频率最高、数据量大，必须用 NVLink 的 600+ GB/s 带宽，不能跨节点。
- **DP 组放在跨节点的 InfiniBand 上**：因为 DP 只在 step 末尾做一次 AllReduce，可以容忍 200 Gbps 的网卡。
- **EP 组优先放在节点内**：MoE 每层两次 All-to-All，且 token 量大；如果 EP 跨节点，单步通信延迟会成倍增长。

这里要建立一个核心认知：**"组"不是逻辑标签，而是带宽分级的物理映射**。GPU 拓扑、NVLink 域、IB 网卡，每一层带宽差 5–10 倍，组划分的本质是把通信频率最高的并行放到最快的带宽上。

**举一个反面例子**看看误配的后果：如果把 TP=2 放到两台不同节点上（跨节点 TP），每层要走两次 IB，单层延迟从 NVLink 的 2us 变成 IB 的 50us，**一个 80 层的模型一个 forward 多出几个 ms**。集群调优的魔鬼在于这里——调错一个组映射，GPU 利用率可能从 60% 跌到 30%。

**如何快速验证组划分是否合理？**用 `nccl-tests` 跑一道 AllGather/AllReduce/AlltoAll 的带宽测试，看各个组的有效带宽是否接近硬件上限。如果 TP 组跑不到 NVLink 理论带宽的 80%，那是调度问题。如果 EP 组跑不到 IB 理论带宽的 60%，那是跨节点路径问题。这是 MoE 调优的第一步、也是最常被跳过的一步。

### 4.3 Parallel Folding（EP 与 DP 解耦）

**传统约束**：在 Megatron 2023 年之前的版本里，EP 必须是 DP 的子集（即 EP ≤ DP）。原因是 EP 组在物理上"借"了 DP 组的通信资源——每个 EP 组都是某个 DP 组的子集。

```text
传统约束（pre-folding）：
  ep_group ⊆ dp_group
  即 EP_size ≤ DP_size

后果：当 N_experts 很多、希望 EP 很大时，必须先把 DP 开很大；
      但 DP 大了又会导致全局 batch 过大、收敛变慢、显存吃紧。
      这是一个"二选一"的两难。
```

举个具体例子：DeepSeek-V3 有 256 个 expert，希望开 EP=64 来分散负载。按照传统约束，必须 DP ≥ 64。但 DP=64 意味着 global batch 至少 64 × micro_batch，显存压力极大，且 micro_batch=1 时数据并行的统计意义也变弱。

**Megatron 新方案 - Parallel Folding**：在 2024 年的 Megatron-Core 中引入了 Parallel Folding，核心思想是让 EP 与 DP 完全独立，允许 EP > DP。

技术原理：把原本一维的 DP × EP 通信组重新组织成一个 **2D process mesh**。

```text
Folding 前（一维约束）：
  rank 顺序排成一列：DP_dim
  EP 必须从中切子集

Folding 后（2D mesh）：
  纵轴 = DP（梯度 AllReduce 方向）
  横轴 = EP（token All-to-All 方向）
  两个轴互不干扰，每个 rank 同时持有 (dp_idx, ep_idx) 两个坐标
```

用伪代码可视化（world_size=256, DP=8, EP=32）：

```text
配置：world_size = 256
     DP = 8
     EP = 32
     校验：DP × EP = 8 × 32 = 256 ✓

构造 2D mesh，shape = (DP=8, EP=32)：

         ep0  ep1  ep2  ...  ep31
       +----+----+----+-----+----+
  dp0  |  0 |  1 |  2 | ... | 31 |    ← dp_group_0 沿横轴聚合 32 卡的梯度
       +----+----+----+-----+----+
  dp1  | 32 | 33 | 34 | ... | 63 |
       +----+----+----+-----+----+
  dp2  | 64 | 65 | 66 | ... | 95 |
       +----+----+----+-----+----+
  ...     ...
       +----+----+----+-----+----+
  dp7  |224 |225 |226 | ... |255 |
       +----+----+----+-----+----+
        ↑
        ep_group_0 沿纵轴聚合 8 卡的 expert dispatch
```

读这张图的关键：

- **横向 32 个 rank** 构成一个 EP 组：它们之间做 All-to-All token dispatch，每行一个 EP 组，共 8 个 EP 组。
- **纵向 8 个 rank** 构成一个 DP 组：它们之间做 AllReduce 梯度，每列一个 DP 组，共 32 个 DP 组。
- 任意 rank 同时属于一个 EP 组和一个 DP 组，但这两个组完全正交、互不干涉。

这就是"折叠"的几何含义：把 DP × EP 这个原本被压扁成一维的乘积空间，重新展开成两个互相正交的轴。EP 不再被 DP 的大小所限制，可以独立扩展到非常大的值。

实际收益：对于 DeepSeek-V3 256 expert 的场景，可以选 DP=4（保持小 global batch）+ EP=64（充分分散专家）+ TP/PP 处理深度，整体显存与吞吐可以同时打到目标。

**为什么说 Parallel Folding 是突破性的？**在 Folding 出现之前，设计师要么接受"DP 越大 → EP 越大"的耦合（代价是 global batch 压不住、计算后梯度累加上不去），要么接受"EP 受限 → 专家难以充分分散"（代价是单卡变胖、显存压力分不均）。两者都是“赔錢货”。**Folding 的本质是把一个一维乘积的约束拆成了两个独立轴**，从数学上让设计空间从 $O(N)$ 变成 $O(N^2)$，在工程上意味着为 256+ expert、千卡集群训练高稀疏比 MoE 提供了一条可行路径。

这也从另一个角度说明了为什么 DeepSeek-V3 能在 2024 年下半年调出 256 expert、671B 参数这种"业界之前不敢想"的配置——不只是因为它们有足够多的卡，更是因为这样一类“解耦型并行设计”仅推出不久，填平了从“思想实验”到“实验可践”的鸿沟。

### 4.4 通信模式分析与推荐配置

把五种并行的通信特征列在一起对比：

| 并行维度 | 通信原语 | 单次通信量 | 频率 | 最佳放置 |
|---------|---------|-----------|------|---------|
| DP | AllReduce | $2\Theta_{t}$（参数总量量级） | 每 step 末尾 1 次 | 跨节点 IB |
| TP | AllGather + ReduceScatter | $\propto$ 层数 × hidden | 每层 2 次 | 节点内 NVLink |
| EP | All-to-All | $2 \times k \times B \times S \times d$ | 每 MoE 层 2 次 | 尽量节点内 NVLink |
| PP | P2P send/recv | $B \times S \times d$ | 每 stage 边界 1 次 | 跨节点可接受 |
| SP/CP | AllGather / All-to-All | $\propto S$（序列长度） | LayerNorm/Attention 边界 | 节点内 |

其中：$\Theta_{t}$ 表示参数张量元素数；$k$ 是 top-k 路由数；$B \times S$ 是 token 总数；$d$ 是 hidden dim。

**通信频率排序（高 → 低）**：TP > EP > SP > PP > DP

**推荐配置示例（大集群）**：1024 GPUs（128 节点 × 8 卡），256B MoE，$k=4$，$N_{exp}=64$：

```text
目标：
  - 训练 256B MoE 模型
  - 1024 张 GPU 分布在 128 个节点上，每节点 8 卡 NVLink
  - 节点内带宽 600 GB/s（NVLink），节点间带宽 200 Gbps（IB）

配置选择：
  DP = 4    # 每个数据副本做 4 路梯度并行
  TP = 2    # 节点内做 2 路张量并行（hidden 切半）
  PP = 8    # 模型切 8 段，跨节点流水
  EP = 16   # 64 个 expert 分到 16 个 EP rank，每 rank 4 个 expert

  校验：DP × TP × PP = 4 × 2 × 8 = 64 个数据副本
       每个数据副本占 1024/64 = 16 张卡 → 正好等于 EP=16 ✓

物理放置：
  - TP=2 落在节点内同一对 NVLink 邻居 GPU
  - EP=16 横跨 2 个节点（16 张卡 = 2 节点 × 8 卡），All-to-All 大部分走 NVLink
  - PP=8 跨节点流水，每 stage 边界一次 P2P，可走 IB
  - DP=4 跨更多节点，每 step 一次 AllReduce 梯度

通信瓶颈检查：
  - TP（节点内）✓ 利用 NVLink 满速
  - EP（16 卡内 14 张 NVLink + 2 张跨节点）✓ 主要走 NVLink
  - PP（跨节点 P2P）✓ 频率低，IB 带宽够用
  - DP（跨节点 AllReduce）✓ 每 step 一次，可 overlap
```

这套配置的核心思想是：**带宽预算分配**——把最频繁、最贵的通信（TP/EP）压到最快的链路（NVLink），把最便宜的通信（DP/PP）放到最慢的链路（IB）。

**如何在自己的集群上选配置？**不同集群规模、不同拓扑、不同模型会需要不同的定制，但可以用一个**三步决策法**快速拿到一个近似最优配置。

**第一步：选 TP**。TP 的选择主要看单卡显存能不能装下单层参数。一般 TP=2/4/8。推荐从 TP=2 起试，仅在单层装不下时打到 TP=4。不要走 TP=8——TP=8 跨 NVLink 域会造成带宽净损。

**第二步：选 PP**。PP 主要看模型总参数能不能装进一个 TP 组。如果装不下，开 PP=2/4/8 拆分不同层。PP 的代价是流水线气泡，需要 1F1B、Interleaved 1F1B 等调度优化。一般推荐 PP 不要超过 8。

**第三步：选 EP**。在 TP/PP 确定之后，剩下的并行度全部留给 DP 和 EP。对于 MoE，EP 推荐越大越好（拆散 expert 负载），但不能超过 N_exp。对于 Parallel Folding，EP 可以独立选取；对于传统架构，EP·DP 的乘积等于剩余并行度。

**一个常见的混淆点**：很多初学者会问"为什么不把 EP 调到最大？"答案是 EP 越大，all-to-all 跨节点的部分越多，带宽要求越高。**EP 的甜点是能装进一个节点（10~16 卡）或两个节点（靠 IB 补补全连接）的范围**，超过这个甜点后边际收益会迅速下降。

**三个调优踩坑**也值得提醒，这些是在多个生产集群上被反复验证过的“边界条件”：

**坑一：PP 开到 16 以上仅在超大模型上才有意义**。PP > 16 并不是不能跳，而是“气泡成本在 1F1B 调度下一般不会超过 5%，但换来的额外跨节点 P2P 却会增加 30%+”。只有 100B+ 参数、120+ 层的模型才能反转这个担忧。

**坑二：DP 开到 1。**很多人会为了调大 EP 与 PP 末尾把 DP 开到 1。这实际上取消了 ZeRO 的优势，且优化器状态全部压在单个 DP rank 上，显存压力反而变大。建议 **DP 至少保持 2**，让 ZeRO-1 能优化优化器状态。

**坑三：EP 开到 expert 总数。**看似是“最细粒度”的高效选择，但实际上 expert 总数 EP 意味着每个 EP rank 只装 1 个 expert， GroupedGEMM 的 batched dim 只能从跨 expert 里拼凑，反而失去了 GroupedGEMM 的加速优势。**实践中 EP 推荐 = N_exp / 4 到 N_exp / 2**。

**本章小结与过渡**：第四章我们把视野从“EP 独立维度”拓宽到“多维并行”。一个明确的认知是：**EP 从来不是孤军奋战**，它的性能上限完全取决于与 DP/TP/PP/SP 的协同设计。**“Parallel Folding 为什么是突破性的”**为本章提供了一个关键变量：它把 EP 从 DP 的附属中释放，为 256+ expert 的架构开辟了可行路径。以此为起点，下一章我们进入两个主流框架的具体实现——先是以“配置驱动”为哲学的 DeepSpeed-MoE，再是以“通信原语驱动”为哲学的 Megatron-LM。

## 五、DeepSpeed-MoE 实现

如果说 Megatron-LM 是"由通信原语向上构建框架"，那 DeepSpeed-MoE 就是"由配置文件向下生成系统"。DeepSpeed-MoE 由微软于 2021 年提出（论文《DeepSpeed-MoE: Advancing Mixture-of-Experts Inference and Training to Power Next-Generation AI Scale》），它的最大特色是把 MoE 的复杂度封装到一个 JSON 配置里，用户只需调几个旋钮就能跑起来；它也是最早提出 **PR-MoE（Pyramid-Residual MoE）**这种新型拓扑的工作。本章按四个维度展开：架构、并行实现、通信优化、模型创新。

### 5.1 架构设计与配置参数

**为什么先从 DeepSpeed-MoE 讲起？**在 2021–2023 这三年，DeepSpeed-MoE 几乎是外部团队能买得起的“上手门槛最低”的 MoE 训练方案。换句话说，如果你手里有一个 64–64 卡的小集群，想训一个 16x7B / 32x4B 量级的 MoE 模型，**DeepSpeed-MoE 能让你在一周内看到 loss 曲线**，而不是花三个月从头实现一套 fused permutation。这份“快起头”能力与 Megatron-Core “高手逆手可调”的设计哲学形成了互补，这也是为什么实际生产中很多公司会同时使用两者。

**DeepSpeed-MoE 适合谁？**以 2024 年的经验看，有三个场景是它的甜点：（1）中等规模（32–1024 卡）的 MoE 预训练与 SFT；（2）多领域多任务的领域适配（需要 Pyramid MoE 这种灵活拓扑）；（3）以 ZeRO 为起点、希望跳过“为每一层手写 process group”的研究型场景。走到千卡万卡的训练时，Megatron-Core / DeepEP 的优先级会高于 DeepSpeed-MoE，这个跳转点会在第六章详谈。

DeepSpeed-MoE 在系统上分三层：

```text
┌────────────────────────────────────────────────────────┐
│  Training Framework Layer                              │
│    HuggingFace Trainer / DeepSpeed config (JSON)       │
│    用户接口：模型定义 + 配置文件 → 启动训练               │
├────────────────────────────────────────────────────────┤
│  DeepSpeed Core                                        │
│    ZeRO Optimizer (1/2/3)                              │
│    Expert Parallel Manager (创建 EP 通信组)             │
│    All-to-All Communication (NCCL wrapper)             │
│    Mixed Precision (FP16/BF16/FP8)                     │
├────────────────────────────────────────────────────────┤
│  MoE Specific Kernels                                  │
│    Permutation Kernel (CUDA)                           │
│    Gating Kernel (Top-K + softmax + capacity check)    │
│    GroupedGEMM (BatchedGEMM 的变体)                    │
│    NCCL AlltoAll wrapper (异步 + overlap hook)          │
└────────────────────────────────────────────────────────┘
```

最上层是用户接触的"配置层"，最下层是真正的 CUDA 算子，中间是 DeepSpeed 自己的调度核。这种分层让 DeepSpeed-MoE 的 onboarding 成本很低——给一份 JSON 就能跑。

典型的 DeepSpeed-MoE JSON 配置：

```json
{
  "moe": {
    "num_experts": 128,
    "num_local_experts": 8,
    "expert_parallel_size": 16,
    "capacity_factor": 1.25,
    "aux_loss_coeff": 0.01,
    "top_k_routing": 2
  },
  "zero_optimization": {
    "stage": 1
  },
  "fp16": { "enabled": true }
}
```

逐项解释：

| 字段 | 含义 | 典型取值 |
|------|------|---------|
| `num_experts` | 全局 expert 总数 | 64 / 128 / 256 |
| `num_local_experts` | 每个 EP rank 上放几个 expert | num_experts / EP |
| `expert_parallel_size` | EP 组大小 | 8 / 16 / 32 |
| `capacity_factor` | 容量倍数 $C$ | 1.0 / 1.25 / 1.5 |
| `aux_loss_coeff` | 负载均衡辅助损失权重 $\alpha$ | 0.01 |
| `top_k_routing` | top-k 中的 $k$ | 1 / 2 |

注意 `num_experts = num_local_experts × expert_parallel_size` 必须严格成立，否则 DeepSpeed 启动时会报错。这一约束的本质就是"专家总数=每卡持有数×卡数"。

**配置参数的工程意义详解**：以上几个字段不是随意调的旋钮，每一个都背后隐含一个应用上的权衡。

**`num_experts`**：总专家数决定模型能存在多少参数。从 64 到 128 再到 256，是 MoE 从 2021 到 2024 年的主要演化路径。数越大，表达容量越高，但均衡、调度、通信都难度上升。

**`num_local_experts`**：每张 GPU 上要装几个专家。如果这个数 < 1，会报错；如果 >32，单卡 GroupedGEMM 会变得不高效。实践中 4~16 是个甜点区间。

**`expert_parallel_size`**：即 EP 组大小。在传统架构中需 ≤ DP；在 Parallel Folding 中可独立选。一般从 8 起、上限在一个节点内（8卡）或跨两个节点（16卡），超出后跨节点 IB 带宽会撑不住。

**`capacity_factor`**：前面详解过。反复提醒：新手推荐 1.25；走到 256 expert 以上考虑用 device-limited routing。

**`aux_loss_coeff`**：控制均衡辅助损失的权重。太小起不了均衡作用，太大占主位拽主任务。常见默认 0.01。

**`top_k_routing`**：每个 token 激活几个 expert。k=1 为 Switch，k=2 为 GShard/Mixtral，k=8 为 DeepSeek-V3。从 DeepSpeed-MoE 设计起点看，k=2 是主流选择。

### 5.2 Expert 并行实现方式

DeepSpeed-MoE 的 EP 实现走的是"轻封装"路线：直接调用 PyTorch 的 `torch.distributed.new_group` 创建 EP 通信组，并通过 hook 机制注入到 forward/backward 中。

伪代码展示 EP 组创建：

```text
def create_expert_parallel_group(world_size, ep_size):
    # 假设 world_size=64, ep_size=16
    # 划分逻辑：连续的 ep_size 张 GPU 构成一个 EP 组
    num_ep_groups = world_size // ep_size      # 4 组

    for ep_group_id in range(num_ep_groups):
        # 每个 EP 组 = [ep_group_id*16, ep_group_id*16+1, ..., +15]
        ranks = list(range(
            ep_group_id * ep_size,
            (ep_group_id + 1) * ep_size,
        ))
        group = dist.new_group(ranks=ranks)
        if my_rank in ranks:
            my_ep_group = group   # 当前 rank 记下自己所属的组

    return my_ep_group
```

Expert 在 GPU 间的分布策略采用**连续切分**：

```text
假设 num_experts=128, ep_size=16
则 num_local_experts = 128 / 16 = 8
分布方案：
  ep_rank_0 持有 expert_0  ~ expert_7
  ep_rank_1 持有 expert_8  ~ expert_15
  ep_rank_2 持有 expert_16 ~ expert_23
  ...
  ep_rank_15 持有 expert_120 ~ expert_127
```

这种连续切分的好处是 expert_id 到 (rank, local_id) 的映射可以用一个除法和取模直接算出：`rank = expert_id // 8, local_id = expert_id % 8`，路由阶段无需查表。

**使用 DeepSpeed 训练 MoE 的实践指南**：不少初学者拿到 DeepSpeed 后第一反应是“为什么启动不起来、为什么 OOM、为什么 loss 不正常”，其实都是在重复同一类问题。这里列出几条可以避踩的陷阱。

**第一**，启动前一定要验证 `num_experts == num_local_experts × expert_parallel_size`。这看似个废话，但实践中动不动就改了其中一个字段忘了同步其他两个。

**第二**，`zero_optimization.stage` 如果上到 3，MoE 会出现双重切分冲突。**DeepSpeed-MoE 官方在 expert 参数上只用 ZeRO-1/2，不用 ZeRO-3**，这个限制是硬约束，不要挑战。

**第三**，FP16 与 BF16 的选择：训练 MoE 推荐 BF16。FP16 的动态范围在路由损失加辅助损失上表现不稳定，偶发性 NaN 是常事。BF16 的 dynamic range 足够大，几乎不需要 loss scale。

**第四**，启动后第一件事是 checkpoint 一次。**DeepSpeed-MoE 的 checkpoint 格式是分散的**，每个 EP rank 只保存自己那部分 expert。恢复训练时一定要保证 EP topology 与训练时一致，否则会装错 expert，表现为 loss 猛增并难以复现。

**与 ZeRO 的兼容方式**：DeepSpeed-MoE 设计了"双轨优化器"：

- **非 expert 参数**（attention、layer norm、router）：走标准 ZeRO-1/2，跨全部 DP rank 切分。
- **Expert 参数**：只在 EP 组内切分，不走 ZeRO 的全局切分（因为 expert 已经被 EP 切过了，再切会双重切分导致冲突）。

伪代码：

```text
optimizer_groups = []
for param in model.parameters():
    if is_expert_param(param):
        optimizer_groups.append({
            "params": param,
            "scope": "ep_group",        # 只在 EP 组内做 ZeRO
        })
    else:
        optimizer_groups.append({
            "params": param,
            "scope": "dp_group",        # 在全局 DP 组做 ZeRO
        })
```

### 5.3 All-to-All 通信优化

DeepSpeed-MoE 的 All-to-All 直接 wrap NCCL，但在调度层做了几个关键优化。

**优化 1：发送/接收计数预计算**

NCCL 的 `ncclAllToAllv` 需要知道每个 peer 的发送量、接收量。DeepSpeed 在 routing 阶段就把这些量算好：

```text
# routing 完成后，立即计算 send_counts
send_counts = [0] * ep_size
for token_idx, expert_id in enumerate(routing_assignments):
    target_rank = expert_id // num_local_experts
    send_counts[target_rank] += 1

# 通过 AllToAll 让每个 rank 知道自己将收多少
recv_counts = AllToAll(send_counts)   # 这一步通信量极小（仅 ep_size 个 int）

# 然后才执行真正的 token AllToAllv
token_chunks = AllToAllv(
    sendbuf=permuted_tokens,
    sendcounts=send_counts,
    recvcounts=recv_counts,
)
```

为什么要"先 AllToAll 计数、再 AllToAllv 数据"？因为 NCCL 不支持动态 size 的 AllToAll——必须提前知道每个 peer 收发多少元素。这个"两段式 AllToAll"是 DeepSpeed 的标准做法。

**优化 2：内存对齐**

permutation 后的 token buffer 按 16-byte 对齐，避免 NCCL 内部为了对齐做 padding；同时使用 pinned memory + CUDA stream，让 H2D 与计算 overlap。

**优化 3：与 ZeRO 梯度通信的调度协调**

在 backward 阶段，expert 参数的梯度 ReduceScatter 与 attention 参数的 AllReduce 会同时发生。DeepSpeed 的调度器做了如下排序：

```text
backward 顺序（从最后一层往前）：
  1. expert backward → 触发 EP 组内 AllToAll（combine 反向）
  2. attention backward → 触发 DP 组 AllReduce
  3. 优化器更新前，AllReduce 与 AllToAll 必须都完成

调度技巧：
  - 把 EP AllToAll（节点内 NVLink）放在前
  - 把 DP AllReduce（跨节点 IB）放在后
  - 二者互不干扰，可在不同 stream 上并行
```

**通信优化的效果量化**：以上三条优化看起来不起眼，但加起来能在典型 8台节点 × 8卡 EP 训练上拿下近 1.5× 的 step 吞吐。具体拆分：

- **优化 1（两段式 AllToAll）**：避免了为了应对 worst case 预留超大 buffer 的损失，实际吞吐 提升 ~10%。
- **优化 2（16-byte 对齐 + pinned memory）**：让 NCCL 不需要做 internal padding，减少了 ~5% 的传输量；H2D 与计算 overlap 的效果是另外 ~15%。
- **优化 3（EP/DP 调度协调）**：让 backward 阶段 EP All-to-All 与 DP AllReduce 完全重叠，净节省 ~20% 的 backward 总时间。

**三条加起来，在中等规模 MoE 训练上能追上 Megatron-Core 的 fused-permute 方案 70%~80% 的效果**，代价是代码保持在“可读可调试”的状态。对于不能忍受 DeepEP MegaKernel 调试难度的中小团队，DeepSpeed-MoE 的这套优化是性价比最高的选择之一。

### 5.4 PR-MoE 与金字塔型 MoE

这是 DeepSpeed-MoE 论文的两个原创贡献，直到今天仍在被广泛使用。

**PR-MoE（Pyramid-Residual MoE）**

朴素 MoE 的 FFN 层输出是：

$$
y = \sum_{e \in \text{TopK}} g_{e}(x) \cdot \text{Expert}_{e}(x)
$$

只有被路由到的 expert 输出会反向传播。这有两个问题：

1. 没被选中的 token 的 FFN 信号"断流"，损失了 dense 网络的稳定性。
2. Expert 容易陷入 collapse——某几个 expert 永远不被选中。

PR-MoE 的修改：在 expert 输出之外加一条**残差路径**：

$$
y = \underbrace{\text{FFN}_{\text{shared}}(x)}_{\text{残差/共享专家}} + \underbrace{\sum_{e \in \text{TopK}} g_{e}(x) \cdot \text{Expert}_{e}(x)}_{\text{routed experts}}
$$

伪代码视角：

```text
def pr_moe_layer(x):
    # 1. 共享路径：所有 token 都过这个小 FFN（dense）
    shared_out = shared_ffn(x)        # 标准 FFN，所有 token 都过

    # 2. 路由路径：只有被选中的 token 过对应 expert
    routed_out = standard_moe(x)       # top-k expert + dispatch

    # 3. 残差合并
    return shared_out + routed_out
```

好处：

- 保留了原始信号流动（dense 的稳定性）。
- 即使某个 token 的 routing 完全跑偏，shared 路径还能给出基础输出。
- 训练前期尤其重要——early stage routing 还没学好时，shared 路径担当主力。

DeepSeek-V2/V3 的 "shared experts" 设计就是 PR-MoE 思想的延续：每层固定有 1–2 个 shared expert（所有 token 都过），加上 top-k routed expert。

**PR-MoE 为什么是重要创新？**最主要的价值在于它 将"MoE 的集成与稳定性"这个在论文里常被忽略的工程问题提上了台面。原生 MoE 的一个隐疼是：**路由一旦出现偏差（某些 token 被全部路由到烂 expert，或者被 drop），这些 token 的信号就丢了**。加了共享路径之后，即使路由这边"崩塌"了，shared FFN 依然能保证一个合理的输出。这让 MoE 的训练曲线变得与 Dense 类似的平滑，是看似微小、实际重要的改进。

**Shared Expert 在训练与推理中的不同意义**：训练阶段，shared expert 主要偏重“隔离路由噪声、保护梯度流动”；推理阶段，shared expert 能提供"默认输出保底”，让模型在未曾见过的 token 上也具有一定的 fallback 能力。这也是为什么 DeepSeek-V2/V3 都采用了 shared expert——不是炫技，而是被必需驱动的工程选择。

**金字塔型 MoE（Pyramid MoE）**

观察发现：Transformer 的不同层对"专家容量"的需求不同。

- **底层**（接近输入）：处理 token 级别的语法/形态信息，需要大量细粒度专家。
- **顶层**（接近输出）：做高层语义聚合，需要稳定的 dense 表示。

朴素 MoE 让每层用相同的 $N_{exp}$ 和 $k$，浪费了顶层的 capacity。Pyramid MoE 的方案：**层数越高，expert 数越少、k 越小**。

典型配置（24 层 MoE）：

```text
layer  0–7   (底层): N_exp=256, top_k=8   → 高 capacity，捕获细节
layer  8–15  (中层): N_exp=128, top_k=4   → 中等 capacity
layer  16–23 (顶层): N_exp=32,  top_k=1   → 低 capacity，趋近 dense
```

效果：

- 总参数量与等量纯 MoE 相比下降 ~30%。
- 同等参数下 perplexity 更低（顶层稀疏度低，收敛更稳定）。
- 训练 wall-clock 缩短（顶层 routing 计算量小）。

DeepSpeed-MoE 的实现方式是允许 `num_experts` 是一个 list 而不是单个 int：

```json
{
  "moe": {
    "num_experts": [256, 256, 128, 128, 64, 32],
    "top_k_routing": [8, 8, 4, 4, 2, 1]
  }
}
```

每个位置对应一组层，框架自动 broadcast 到对应层数。这是 DeepSpeed-MoE 在工程上对"非均匀 MoE"的优雅支持。

**Pyramid MoE 背后的认知**：为什么“顶层应该趋近 dense”？这是一个在 NLP 领域重复被验证过的现象：Transformer 的不同层学到的东西不一样。**底层学到的是字词形态与词法**（POS 标注、子词拼接、实体边界），这些任务需要大量细粒度特征；**中层学到的是句法与含义关系**（主谓宾、从属、什么修饰什么）；**顶层学到的是高阶语义与任务抽象**（意图、意义联动、抽象推理），这些任务越偏向 dense 表示。Pyramid MoE 的设计哲学与由低阶到高阶逐渐“收敛表示”的词法学习理论是一致的。

**与传统 MoE 的量化对比**：举一个 24 层 MoE 的例子，如果所有层都用 N_exp=128, k=4，则整个模型存储参数量为 24×128×Θ_expert；Pyramid MoE 的配置 下，总参数只有 8×256 + 8×128 + 8×32 = 3296×Θ_expert，与原来的 24×128 = 3072×Θ_expert 几乎持平，但底层专家数提高了一倍，顶层 routing 开销下降了 4 倍。这个“同参数量、不同质量”的交换在 NLP 任务上能带来 1~2 个 perplexity 点的提升。

**实际用法上的限制**：Pyramid MoE 近两年并未被主流 MoE 模型采纳（DeepSeek-V3、Qwen3 都是“均匀 MoE”）。主要原因不是它不 work，而是**调优难度太大**：需要为每一组层独立调 N_exp、k、CF、负载均衡等参数，调参空间指数级扩张。在备受关注的大模型训练里，大家更偏好“均匀不花哨但能快速交付”的设计。Pyramid MoE 更多在小规模实验或领域适配场景下被采用。

### 5.5 本章小结

DeepSpeed-MoE 的价值在于它是**第一个让 MoE 上手门槛变低的平台**。一份 JSON 配置、一个 Trainer 调用，中小团队也能起手训 MoE。它的 PR-MoE 设计启发了后来 DeepSeek-V2/V3 的 shared expert；它的 "两段式 AllToAll" 被后面所有主流 MoE 框架采纳；它的 ZeRO 与 EP 的双轨优化器设计，几乎是后续所有 MoE 训练框架的标配。

但 DeepSpeed-MoE 也有它的限制：**它是"中等规模友好"的，并不适合走到千卡或万卡集群**。在千卡以上的训练上，Megatron-Core 的多维度并行、DeepEP 的 MegaKernel、DeepSeek-V3 的 device-limited routing 都表现更优。下一章我们将带你进入 Megatron-LM MoE 的世界，看看"通信原语驱动"的路线如何与 DeepSpeed 的"配置驱动"路线形成互补。

**三个心得作为全章总结**。第一，**DeepSpeed-MoE 代表了“能用”比“性能极致”更重要的产品哲学**，在中小团队看来，能一周内跑起来的 0.7× 于一个多个月调出来的 1.0×，它们能让重点回到“模型质量”这个本质问题上。第二，**JSON 配置不是劣势限制，而是一份完整的设计语言**——你能从配置文件反推出一份设计意图，这是很多“代码驱动”框架做不到的。第三，**PR-MoE 与 Pyramid MoE 在论文发表 5 年后仍被讨论**，这件事本身说明了 DeepSpeed-MoE 的贡献不只是工程实现，还包含了对“MoE 架构应该怎么长”这个问题的哲学思考。

## 六、Megatron-LM MoE 实现

如果 DeepSpeed-MoE 是"配置驱动"的工程派代表，那 Megatron-LM MoE 就是"通信原语驱动"的研究派代表。Megatron 由 NVIDIA 维护，所有的 MoE 优化都直接体现在 process group 划分、CUDA kernel 选型、与 TP/PP/SP 的深度耦合上。本章先讲 Megatron 的 Token Dispatcher 6 步流程，再深入 Parallel Folding 的 process group 创新，然后展示功能矩阵和 GroupedGEMM kernel。

### 6.1 Token Dispatcher 机制

在深入 6 步流程之前，先想清楚为什么需要一个叫 **Dispatcher** 的东西。MoE 与 Dense 在执行模型上的根本区别就在于：Dense 层里每个 token 走的路径是完全一样的——一条直筒子 GEMM；而 MoE 层里每个 token 都要根据 router 的判决，被"派遣"到不同的 expert，再"取回"结果。**这件事本身就是一次跨设备的、不规则的、动态的数据搬运**——它既不像 AllReduce 那样规则、也不像 broadcast 那样固定方向，需要一个专门的抽象来统一管理"哪个 token 该去哪个 GPU、要传多少字节、回来后怎么拼回去"。Megatron 把这一整套调度统一封装成 Token Dispatcher，外部调用者只需要传入 `tokens` 和 `router_logits`，里面 6 个步骤一气呵成，对上层 Transformer 完全透明。

这种抽象的好处是：Dispatcher 的 6 步流程一旦固定，**底层通信用什么后端（NCCL、DeepEP、MSCCL++）、计算用什么 kernel（朴素循环、GroupedGEMM、Fused MoE）都可以独立替换**，而不影响 Transformer 主干。这也是 Megatron 能在两年内快速吸收 DeepSeek、Qwen 等新模型路由设计的工程基础。

Megatron-LM 把 MoE 的执行抽象成一个 6 步的 **Token Dispatcher** 流程：

```text
┌────────────────────────────────────────────────────────┐
│  Token Dispatcher 6 步流程                              │
├────────────────────────────────────────────────────────┤
│                                                        │
│  Step 1: Routing Decision                              │
│    Input  : tokens [B*S, H]                            │
│    Logits : router(tokens) = [B*S, N_exp]              │
│    Softmax / Sigmoid + TopK                            │
│    Output : assignments [B*S, k] (expert_id, weight)   │
│                                                        │
│  Step 2: Compute Send/Recv Counts                      │
│    For each token, expert_id // num_local = target_rank│
│    send_counts[target_rank] += 1                       │
│    AllToAll(send_counts) → recv_counts                 │
│                                                        │
│  Step 3: Prepare Permutation Index                     │
│    按 expert_id 排序生成索引：                          │
│    perm_idx = argsort(assignments.expert_id)           │
│    permuted_tokens = tokens[perm_idx]                  │
│    （所有发往同一个 rank 的 token 紧密排在一起）         │
│                                                        │
│  Step 4: Execute All-to-All                            │
│    NCCL AllToAllv(                                     │
│      send=permuted_tokens, send_counts,                │
│      recv=local_tokens,    recv_counts,                │
│    )                                                   │
│                                                        │
│  Step 5: Expert Forward (本地)                         │
│    for each local_expert_id in [0, num_local_experts): │
│        slice = local_tokens[expert_id == ...]          │
│        out_slice = expert[local_expert_id](slice)      │
│    或用 GroupedGEMM 一次完成（见 6.4）                  │
│                                                        │
│  Step 6: Combine (反向 dispatch)                       │
│    AllToAllv(out_local) → out_permuted                 │
│    Unpermute: out = out_permuted[inv_perm_idx]         │
│    Gate combine: y = Σ gate_weight × out               │
└────────────────────────────────────────────────────────┘
```

关键点：

- Step 2 的"先 count、再 data"是为了 NCCL 的 size 要求。
- Step 3 的 permutation 让 token 在显存里按 expert_id 紧密排列，这是 GroupedGEMM 能加速的前提。
- Step 6 的 unpermute 必须用 Step 3 保存的 `perm_idx` 来反算，逻辑上是"先按 expert 重排去算，再按原 token 顺序拼回去"。

**为什么这 6 步的顺序不能颠倒？** 这其实不是工程随意排列的结果，而是被 NCCL/CUDA 的硬约束逼出来的。Step 1（routing）必须最先，因为没有 routing 决策就根本不知道要把谁发给谁；Step 2（count 交换）必须在 Step 4（data 交换）之前，因为 NCCL 的 `AllToAllv` 需要双方都先知道 send/recv 的精确字节数，否则缓冲区无法预分配；Step 3（permutation）必须在 Step 4 之前，因为 NCCL 要求"发往同一 rank 的数据在内存里连续"——这是底层 RDMA 写入对地址连续性的要求；Step 5（expert forward）必须在 Step 4 之后，因为没有真正接收到 token 就没法算；Step 6（combine）必须在最后，因为只有重新按原 token 顺序拼回去，下一个 attention 层才能正确读取。**任何一步打乱次序，要么 NCCL 调用直接报错、要么张量形状对不上、要么 attention 看到错位的语义**。

从工程实现上看，这 6 步还隐含着两次 **All-to-All**（Step 4 forward、Step 6 backward 各一次），加起来是 MoE 通信开销的主要来源。后面 6.4 节讨论的 GroupedGEMM 和 fusion kernel 优化，本质上就是在不动这 6 步骨架的前提下，把每一步的 latency 压到极致。

这里还要再多说一句**Dispatcher 的边界条件处理**——这往往是新人最容易忽视、却最容易把训练打挂的地方。比如某个 batch 里恰好有一个 expert 收到 0 个 token，朴素实现里 GroupedGEMM 的 group_sizes 中会出现 0，许多旧版 CUTLASS kernel 在这种情况下会触发 illegal memory access；又比如 capacity factor 设得过紧，部分 token 会被 drop，drop 掉的 token 在 Step 6 combine 时如果没有正确处理（直接置零 vs 用残差兜底），梯度回传就会出现 NaN。Megatron 的 Dispatcher 在每一步都做了**显式的 0-token 边界处理与 dropped-token 占位填充**，这些细节在论文里完全看不到，但工程师只要写过一次自己的 MoE，就一定踩过这些坑。

另外一个值得记住的设计原则是**Dispatcher 的所有中间状态都必须是显式可重入的**。MoE 训练经常需要 activation recomputation（把 Step 5 的中间输出丢掉、反向时重算），如果 Step 3 的 perm_idx、Step 2 的 send_counts/recv_counts 任何一个没被存下来，反向 recompute 的时候就只能从 routing 重新跑一遍——这意味着反向比前向还要再多一次 routing 决策，浪费整整一个 step 的开销。Megatron 的实现显式地把 perm_idx 和 counts 列入"必须保留的小型状态"，只丢掉真正占大头的 activation 张量，这样 recompute 才划算。

### 6.2 Parallel Folding 与 Process Group 创新

在 4.3 节我们讲了 Parallel Folding 的几何含义。现在从代码视角看 Megatron 是怎么实现的。在进入代码之前，先理解一下**这个设计为什么是突破性的**——只有亲身被旧约束坑过的人，才能明白 Folding 有多解放生产力。

传统约束之痛：在 Folding 出现之前，工程师如果想"扩大 expert 数量却保持 DP 不变"，就只能把 DP 砍小、把 EP 加大——因为 EP 是从 DP 里"借"卡。比如原本 DP=64、EP=8（每 expert 复制 8 份），如果想让 EP=64（专家完全分布、不冗余），就只能 DP=8。但 DP 一旦砍小，batch 规模也跟着掉，梯度噪声变大、训练收敛变慢；反过来如果坚持 DP=64，就只能少装 expert，模型容量天花板被限制。**这就是为什么早期 MoE 论文里 expert 数往往只有 8、16、32——不是研究者不想多，而是框架不允许多。**

Folding 解决之后带来了什么可能性：DeepSeek-V3 能跑 256 个 routed expert，DP 仍可以保持 32~64 的规模，全靠 EP 与 DP 在 2D mesh 上正交切分。换句话说，**"扩大模型容量"和"扩大数据并行规模"第一次被解耦**，研究者可以独立调整两个轴。这就像从单变量优化变成了二维空间搜索，模型架构的设计自由度直接翻了一个数量级。

**传统约束的来源**：早期 Megatron 维护一个一维的 rank 序列，DP 组 = 整个序列，EP 组 = DP 组的连续子集：

```text
所有 rank = [0, 1, 2, ..., 255]
DP_size = 256
EP_size 必须能整除 DP_size，且 ep_group 是 dp_group 的连续 chunk
```

这种实现里，`dp_group` 和 `ep_group` 共享一组 rank 列表，导致它们的通信不能完全解耦。

**Folding 的数学原理**：把一维序列 reshape 成 2D mesh：

$$
\text{rank}_{1D}[i] \;\;\Longleftrightarrow\;\; \text{rank}_{2D}[\,i \div EP,\; i \bmod EP\,]
$$

也就是说，rank=33 在 2D mesh 中坐标为 (33 // 32, 33 % 32) = (1, 1)。

**代码级别的组创建**（伪代码）：

```text
def build_process_groups(world_size, dp, tp, pp, ep):
    # 假设 world_size=256, dp=8, ep=32, tp=1, pp=1
    # 构造 2D mesh，shape = (dp, ep)
    mesh = torch.arange(world_size).reshape(dp, ep)
    # mesh[0] = [  0,   1, ...,  31]
    # mesh[1] = [ 32,  33, ...,  63]
    # ...
    # mesh[7] = [224, 225, ..., 255]

    # DP 组 = mesh 的每一列（沿 dp 轴聚合）
    for ep_idx in range(ep):
        ranks = mesh[:, ep_idx].tolist()    # [ep_idx, ep_idx+32, ...]
        dp_group_e = dist.new_group(ranks=ranks)

    # EP 组 = mesh 的每一行（沿 ep 轴聚合）
    for dp_idx in range(dp):
        ranks = mesh[dp_idx, :].tolist()    # [dp_idx*32, ..., dp_idx*32+31]
        ep_group_d = dist.new_group(ranks=ranks)

    return dp_group_e, ep_group_d
```

**实例：world_size=256, DP=8, EP=32 的具体 rank 映射**

```text
2D mesh（DP=8 行 × EP=32 列）：

         ep=0  ep=1   ...  ep=31
       +-----+-----+------+------+
  dp=0 |   0 |   1 | ...  |  31  |   ← EP 组_0 = [0, 1, ..., 31]
  dp=1 |  32 |  33 | ...  |  63  |   ← EP 组_1 = [32, ..., 63]
  dp=2 |  64 |  65 | ...  |  95  |
  ...
  dp=7 | 224 | 225 | ...  | 255  |   ← EP 组_7 = [224, ..., 255]
       +-----+-----+------+------+
       ↑
       DP 组_0 = [0, 32, 64, 96, 128, 160, 192, 224]
       DP 组_1 = [1, 33, 65, ..., 225]
       ...

rank=33 的归属：
  → 属于 DP 组_1 = [1, 33, 65, 97, ..., 225]   做梯度 AllReduce
  → 属于 EP 组_1 = [32, 33, 34, ..., 63]        做 token All-to-All

rank=33 同时是 (dp=1, ep=1) 这个二维坐标。
两个组的成员完全不重叠（除了它自己），所以两个通信完全独立。
```

这就是 Folding 的工程价值：**两个进程组在 rank 集合上互不相交（除自身），所以两条通信链路可以同时跑而不会互相竞争 NCCL channel**。在 Megatron 2024 的 benchmark 中，Folding 让 DeepSeek-V2 规模模型的训练吞吐提升约 18%。

更深一层看，Folding 之所以重要，是因为它让 Megatron 的 process group 抽象从"一维 rank 列表"升级到了"N 维 mesh"。在新的抽象里，DP/EP/TP/PP/SP/CP 每个维度都对应 mesh 的一个轴，任意两轴互相正交、可独立 reshape。这件事的工程意义远超 MoE 本身——它意味着未来加入新的并行维度（比如 Expert Sharding、Adapter Parallelism）只需要在 mesh 上多开一个轴，不需要重写底层通信逻辑。Megatron 在 2024 年的重构基本都是围绕这一点展开的。

### 6.4 GroupedGEMM 与 Fusion Kernel

**为什么 MoE 需要 GroupedGEMM**

要理解 GroupedGEMM 为什么不可或缺，先得回答一个更基础的问题——**为什么朴素循环 GEMM 在 MoE 场景下慢得"令人发指"**。原因可以拆成两层。第一层是 CPU 侧的 kernel launch 开销：CUDA 每次 launch 一个 kernel 都有约 5–10μs 的固定成本，包含驱动调度、参数拷贝、stream 同步等，单看这个数字很小，但 MoE 一层有几十个 expert、网络有几十层，乘起来就是几十毫秒纯 launch 开销，把训练吞吐直接腰斩。第二层更致命，是 GPU 侧的硬件利用率：H100 有 132 个 SM，一次 GEMM 必须把任务切成大量 tile 才能填满 SM；而单 expert 拿到的 token 数量往往只有几百，对应的 GEMM 维度 `[token, hidden] × [hidden, 4*hidden]` 在 token 维度上太小，分块后只能占用一小部分 SM，剩下的 SM 全部空闲，**SM 利用率往往不到 20%**。

在 Step 5（expert forward）中，每个 expert 收到的 token 数是不同的。朴素实现是循环 GEMM：

```text
朴素循环实现：
for local_expert_id in range(num_local_experts):
    tokens = local_buffer[expert_assignment == local_expert_id]
    out    = tokens @ expert_weight[local_expert_id]
    # 这里是一次独立的 GEMM kernel launch

问题：
  1. kernel launch 开销：每次 launch 几微秒，8 个 expert 就是几十微秒
  2. SM 利用率低：每个 GEMM 用不满全部 SM
  3. 不规则 size：每个 expert 的 token 数不同，无法 batched
```

再补一刀：GPU 还有"wave quantization"的隐藏成本。GEMM 的 tile 数必须是 SM 数的整数倍，否则最后一波（wave）会只用部分 SM，造成"长尾"。MoE 里的 GEMM 形状本身就不规则，几乎每次循环都触发 wave 长尾，叠加在一起就是雪上加霜。

**GroupedGEMM 的方案**：把多个不等大小的矩阵乘法打包成一次 kernel launch，每个 SM 处理其中一个子任务。

```text
GroupedGEMM 调用：
out_grouped = grouped_gemm(
    tokens_concat,                # [total_tokens, H]，所有 expert 的 token 拼一起
    weight_grouped,               # [num_local_experts, H, 4H]
    group_sizes=[127, 64, 200, 89, 156, 72, 33, 167],   # 每个 expert 的 token 数
)
# 内部：CUTLASS 把这 8 个不等大小的 GEMM 打包到 1 次 kernel launch
# 每个 SM 拿到一个子任务，自适应调度

性能对比（典型场景，A100 80G，hidden=4096）：
  循环 GEMM   ：~2.4 ms
  GroupedGEMM ：~0.8 ms        加速 3×
```

**与 FlashAttention 的 fusion**

更激进的优化是把 router、permutation、GroupedGEMM、unpermute 全部 fuse 成一个 mega-kernel：

```text
朴素 MoE 单层（5 个 kernel launch）：
  router_kernel → permute_kernel → grouped_gemm
  → unpermute_kernel → gate_combine_kernel

Fusion MoE 单层（1 个 kernel launch）：
  fused_moe_kernel(tokens, router_w, expert_w, gate_w)
    内部完成：router → permute → gemm → unpermute → combine
    无中间显存写回

收益：
  - kernel launch 开销 5× → 1×
  - 中间张量不写显存（permuted tokens 全留在 SMEM/寄存器）
  - 端到端加速 1.5–2×（视场景）
```

DeepSeek 的 DeepEP 库、SGLang 的 fused_moe kernel 都是这种思路。Megatron 在 2025 年的版本中也集成了类似的 fusion 优化。

这里有一个工程经验值得提醒：**fusion 不是越激进越好**。把 5 个 kernel 揉成 1 个 mega-kernel 之后，调试难度急剧上升——任何一步出错都很难单独打印中间张量；而且 mega-kernel 对 SMEM 容量的需求飙升，一旦超过 H100 的 228KB SMEM 上限就 spill 到 L1，性能反而下降。所以生产中常见的折中是"两段 fusion"：router+permute 一段，gemm+unpermute+combine 一段，这样既拿到大部分收益，又保留可调试性。

再补充一个常被忽略的硬件视角：**GroupedGEMM 的真实瓶颈往往不是计算而是 SMEM 与 register 的容量**。H100 一个 SM 只有 256KB 寄存器和 228KB SMEM，分给 warp 后人均很少。当单 expert 矩阵很小时，每个 tile 的工作量装不满 SM 的资源，多余的容量是浪费的；当矩阵略大时又装不下、要 spill 到 L1。所以 GroupedGEMM 实际加速比与 expert size、token count 的乘积分布**强相关**——同一个 kernel 在 hidden=4096 时加速 3×、在 hidden=2048 时可能只加速 1.5×。这也解释了为什么 DeepSeek-V3 把 expert hidden 设计成 2048（细粒度方向）的同时必须配合自研的 DeepGEMM kernel：不重写 kernel，硬件吃不饱。

还有一条容易混淆的概念辨析值得讲清楚：**GroupedGEMM 与 BatchedGEMM 不是一回事**。BatchedGEMM 要求所有子矩阵 shape 完全一致（典型场景是多头注意力里的 N 个 head），底层实现是把 batch 维度直接当成额外的 stride，kernel launch 仍然是一次但 tile schedule 是规则的；而 GroupedGEMM 允许每个子矩阵独立的 (M, N, K)，必须用 indirect addressing 才能让一个 kernel 处理不规则形状。两者的硬件路径完全不同，性能特性也不同——BatchedGEMM 接近原生 GEMM 的峰值效率，GroupedGEMM 因为间接寻址会损失 10–20% 峰值。MoE 必须用 GroupedGEMM 而不能退化成 BatchedGEMM，因为 expert 之间的 token 数本来就不等。

**性能对比小结**：

| 方案 | 单 MoE 层 forward | 加速比 | 实现复杂度 |
|------|------------------|--------|-----------|
| 循环 GEMM（朴素） | 2.4 ms | 1.0× | 低 |
| GroupedGEMM | 0.8 ms | 3.0× | 中 |
| Fused MoE Kernel | 0.5 ms | 4.8× | 高 |

至此，Megatron-LM MoE 的实现栈就讲清楚了：上层是 6 步 Token Dispatcher 把 routing/permute/dispatch/compute/combine 串成流程；中层是 Parallel Folding 让 EP 与 DP 真正解耦；下层是 GroupedGEMM 与 Fusion Kernel 把通信间隙塞满。三层加在一起，才是今天能撑起千亿参数 MoE 训练的工程底座。

**本章小结**：如果说 DeepSpeed-MoE 把"用得起 MoE"这件事普及了，那 Megatron-LM 就把"用得好 MoE"这件事工业化了。它的三层架构对应着三种不同维度的工程能力：Dispatcher 是**接口抽象**（把不规则通信封装成稳定 API）、Folding 是**并行设计**（让多维度切分正交）、GroupedGEMM/Fusion 是**底层性能**（把硬件榨干）。后面要讨论的 DeepSeek-V3 创新，本质上都是在 Megatron 这套底座之上往前再走一步——比如 Node-Limited Routing 是在 Dispatcher 的 routing 决策上加约束、FP8 训练是在 GEMM 计算路径上换精度。只有先看清这套底座，才能理解后续创新的"接口位置"在哪里。

## 七、前沿创新：DeepSeek-V3 的 MoE 设计

讲完通用框架，本章把镜头拉近到 2025 年最受关注的开源 MoE 模型——**DeepSeek-V3（671B 总参/37B 激活）**。它在多个维度上对"MoE 训练范式"做了重新设计：负载均衡不再依赖 auxiliary loss、路由被限制在少量节点内、shared expert + 细粒度 routed expert 的混合架构、FP8 全链路训练。下面我们一项一项拆开看。

### 7.1 无辅助损失负载均衡

#### 7.1.1 传统 auxiliary loss 的矛盾

在动手分析 DeepSeek 的新方案之前，先把传统 auxiliary loss 为什么"本质上是在和模型对着干"想清楚。语言建模损失 $\mathcal{L}_{\text{LM}}$ 教 router "哪个 expert 真正擅长处理这个 token"，这是模型应该学到的；而辅助损失 $\mathcal{L}_{\text{aux}}$ 教 router "不要老把 token 送到同一个 expert"，这是为了系统并行效率而引入的人为约束。这两件事的目标**根本不一致**——一个追求质量、一个追求均衡。把它们加在同一个 loss 里、用同一个反向传播一起优化，相当于让模型在每一步都要做妥协，**router 的参数永远在两股相反的拉力之间摇摆**。

更糟糕的是，aux loss 的存在让 router 学到的"亲和度"不再是纯粹的语义偏好，而是"语义偏好 - 负载惩罚"的混合信号。当后续推理时如果想做某种基于亲和度的可解释性分析，会发现 router 输出的数字根本说不清楚到底反映了什么。这也是为什么社区里很多 MoE 模型的 router 行为看起来"难以解释"。

在第三章我们看到，主流 MoE（Switch Transformer、GShard、Mixtral）都需要给总 loss 加一项辅助损失（auxiliary loss）来强制负载均衡：

$$
\mathcal{L}_{\text{total}} = \mathcal{L}_{\text{LM}} + \alpha \cdot \mathcal{L}_{\text{aux}}
$$

其中 $\alpha$ 是一个全局超参数。问题在于 $\alpha$ 的取值非常微妙：

- $\alpha$ **太小**（如 $10^{-3}$ 以下）→ 均衡项的梯度被语言建模梯度淹没 → router 仍然倾向于把 token 集中送到少数 expert → **routing collapse**；
- $\alpha$ **太大**（如 $10^{-1}$ 以上）→ 均衡项主导优化 → router 为了均衡而把 token 随机分发 → 语言建模能力下降 → **perplexity 上升**；
- 不同训练阶段（early/late）、不同 batch 分布、不同模型规模，最佳 $\alpha$ 都不一样，调参极其困难。

更糟糕的是，aux loss 是个"虚假"的损失项——它并不直接服务于语言建模，但又必须存在。在大规模训练中，aux loss 还可能引发 **loss spike**：某个 step 由于路由不均，aux 项突然变大，触发梯度爆炸。

DeepSeek-V3 的设计哲学是：**彻底丢弃 aux loss**，改用一种"训练时动态调整 bias"的方法。

#### 7.1.2 算法伪代码

```text
# 初始化（在模型构造时）
bias = zeros(N_routed_experts)   # 形状 [256]，跟着模型一起保存但不参与梯度
gamma = 0.001                    # bias 更新步长，超参

# 每个 training iteration 的 routing 过程
for each token u_t in batch:
    # 1) 计算原始 affinity（不含 bias）
    for i in range(N_routed_experts):
        s_i = sigmoid(u_t · e_i)        # 论文用 sigmoid 而非 softmax

    # 2) 加 bias 仅用于 TopK 选择
    s_biased = s_i + bias               # element-wise

    # 3) 基于 biased scores 选 TopK
    topk_indices = argtopk(s_biased, k=8)

    # 4) ★关键：最终 gating 值用原始 s_i（不含 bias！）
    for i in topk_indices:
        g_i = s_i / sum_{j in topk} s_j  # 归一化原始亲和度
        output += g_i * expert_i(u_t)

# 每个 step 末尾：根据本 step 各 expert 的实际 load 更新 bias
load = count_tokens_per_expert(batch)   # 形状 [256]
mean_load = mean(load)
for i in range(N_routed_experts):
    if load[i] > mean_load:             # 过载
        bias[i] -= gamma
    elif load[i] < mean_load:           # 欠载
        bias[i] += gamma
```

这段伪代码看起来简单，但其中蕴含了三个关键设计。

#### 7.1.3 三个关键特性

**① gating 与 routing 的分离（学习信号不失真）**

传统方案中，aux loss 的梯度会回流到 router 参数，扭曲 router 的学习信号。DeepSeek-V3 把"选哪几个 expert"（routing）和"以多大权重融合 expert 输出"（gating）拆开：

- routing 用 $s_{i} + b_{i}$，由 bias 动态调节，确保均衡；
- gating 用纯净的 $s_{i}$，router 参数的梯度仍然由 LM loss 提供，不会被均衡需求污染。

这意味着 router 学到的是"哪个 expert 真正适合这个 token"，而不是"哪个 expert 当前负载轻"。

**② 动态自适应（不同 batch 分布自动调节）**

bias 是按 step 在线更新的。如果某个 batch 的语义偏 code，对应的 expert 自然过载，bias 就会下调；下一个 batch 偏数学，bias 又自动调整。这是一个**反馈控制系统**，无需手工调参：

```text
bias_i 在训练过程中的典型轨迹：
  step 0:     0.000
  step 100:  -0.012   (expert i 持续过载，bias 被压低)
  step 500:  -0.045
  step 1000: -0.038   (其他 token 不再选 i，load 回落，bias 回升)
  step 2000: -0.041   (稳态：动态平衡)
```

**③ 连续性（没有虚假 loss 项，无 loss spike 风险）**

aux loss 是离散加到总 loss 上的，一旦 routing 突变就可能 spike。bias 调整是**平滑且有界**的——$\gamma = 0.001$ 决定了每步 bias 最多变化 0.001，永远不会引发突变。

DeepSeek-V3 论文报告：在 14.8T tokens 的训练全程，**没有出现任何 loss spike**，也无需任何 rollback。这在千亿规模 MoE 训练中是非常罕见的稳定性表现。

这里再补充一个实践中极其关键但论文里几乎不提的问题：**bias 的初始化策略和冷启动行为**。训练刚开始时所有 expert 的权重都是随机初始化的，router 的亲和度输出几乎是均匀噪声，此时 bias 机制其实不需要工作（因为 load 本来就接近均匀）。但随着训练进行几百步，router 逐渐学出偏好，某些 expert 开始被大量选中，bias 才开始真正发挥作用。**这意味着 $\gamma$ 不能设太大**——前几百步里如果 bias 波动过大，会干扰 router 学习初期的"探索期"，导致 router 看到的亲和度信号一直被 bias 扭来扭去、学不出稳定的语义偏好。生产中的常见做法是：前 1000 步 $\gamma=0$（不更新 bias），之后再 warmup 到正常值 $\gamma=0.001$，这样给 router 一个干净的初始化窗口。

还有一个细节：当从 checkpoint 恢复训练时，bias 数组也必须被完整保存和恢复。虽然 bias 本身不参与梯度计算，但如果恢复后 bias 被清零，相当于突然把所有负载约束都撤掉，router 会在恢复后的第一个 step 就出现 load 突变——这在实践中出过不止一次事故。

这一点的工程意义怎么强调都不为过。千亿规模 MoE 训练一次完整跑要几十天，单次 loss spike 可能造成几个小时的 rollback、甚至触发整批权重作废。Meta 的 OPT-175B 论文里详细记录过他们遇到几十次 spike、每次都要回滚的痛苦经历。而 DeepSeek-V3 用 14.8T tokens 训完**零 spike**，意味着工程团队不需要 24 小时蹲守监控大盘、不需要准备 rollback 脚本、不需要解释"为什么这一段训练浪费了 8 小时算力"。**对一个上百人参与、动辄数千万美元的训练任务来说，稳定性本身就是巨大的成本节省**。

### 7.2 Node-Limited（设备受限）路由

#### 7.2.1 问题：跨节点 All-to-All 的灾难

要理解 Node-Limited Routing 为什么是"用质量换带宽"的精明权衡，先要直观感受**跨节点通信有多贵**。一张 H100 在节点内通过 NVLink 4 代互联，双向带宽高达 **900 GB/s**；而跨节点最快的 InfiniBand NDR（400Gb/s）双向也只有 **50 GB/s**。**节点内比跨节点快约 18 倍**。这不仅仅是数字差异——它意味着如果一次 All-to-All 全部走 IB，需要花的时间相当于节点内 NVLink 的 18 倍；如果一次 All-to-All 既有节点内又有跨节点，整体延迟会被跨节点部分拖死，因为 NCCL 的同步语义要求所有 rank 都完成才能继续。

所以"让 token 尽量留在节点内"不是一个可选的优化，而是 MoE 训练必须遵守的**第一原则**。Device-Limited Routing 的本质就是：宁可让 router 做次优选择（在 M 个节点的 4×8=32 个 expert 里挑，而不是在全部 256 个 expert 里挑），也要把跨节点流量压下去。这看似牺牲了路由质量，但实测下来对 perplexity 的影响小于 0.3%——而通信时间节省 40%+，**整体训练效率净赚**。这就是"用质量换带宽"的精明权衡。

DeepSeek-V3 把 256 个 routed expert 分布在 32 个节点上（每节点 8 个 expert）。如果不加限制，一个 token 在 TopK=8 的情况下，最坏情况可能选中分布在 **8 个不同节点**上的 expert，整个 batch 累计起来就需要跨 32 节点的 All-to-All。

```text
不加限制的最坏情况：
  Node 0  ──┐                       ┌── Node 0
  Node 1  ──┤                       ├── Node 1
  Node 2  ──┤   全互联 All-to-All     ├── Node 2
   ...     ──┤    跨 IB 网络          ├──  ...
  Node 31 ──┘                       └── Node 31

跨节点带宽：~50 GB/s (IB 400Gb/s 双向)
节点内带宽：~900 GB/s (NVLink)
差距：18×
```

跨节点 All-to-All 不仅吃光 IB 带宽，还会因为消息粒度太碎（每 token 只有 7K 字节）导致网络利用率严重下降。

#### 7.2.2 DeepSeek-V3 方案：M=4 节点级 TopK

DeepSeek-V3 在 token 级 TopK 之前先做一次**节点级 TopM**（M=4），把每个 token 的 routing 范围限制在 4 个节点内：

```text
# 每个 token 的两级路由
for each token u_t:
    # 1) 计算 256 个 expert 的原始亲和度
    s = [sigmoid(u_t · e_i) for i in range(256)]

    # 2) 按节点聚合（每节点 8 个 expert）
    for j in range(N_nodes):                   # N_nodes = 32
        node_affinity[j] = sum(s[i] for i in experts_on_node[j])
        # 也可以用 max 或 topk 内求和，论文用"节点内 top-(K/M) 求和"

    # 3) 节点级 TopM 选择（M = 4）
    selected_nodes = argtopk(node_affinity, k=4)

    # 4) 限定在 selected_nodes 范围内做 expert 级 TopK
    candidate_experts = [i for j in selected_nodes for i in experts_on_node[j]]
                        # 候选数：4 × 8 = 32 个 expert
    topk_indices = argtopk(s[candidate_experts], k=8)
```

这样每个 token 最多只会跨 4 个节点通信。

#### 7.2.3 通信优化效果

```text
Before (无限制)：
  每 token 可达 32 节点
  → 跨节点 All-to-All（32 路）
  → 通信打 IB，900 GB/s 的 NVLink 闲置

After (M=4)：
  每 token 最多跨 4 节点
  → 跨节点 All-to-All（仅 4 路）
  → 同节点内 expert 通过 NVLink 通信
  → 跨节点流量减少约 8 倍
```

更妙的是，由于跨节点流量大幅减少，**跨节点通信可以与节点内计算完全重叠**——节点内 NVLink 通信很快（~1ms 量级），节点间 IB 通信慢（~5ms 量级），两者并行进行时 IB 的延迟被 NVLink 计算+通信掩盖。

用一个具体的数值推演来量化这种收益。假设一个 MoE 层需要 dispatch 32K tokens（每 token 7168 维 BF16），总 payload = 32K × 7168 × 2B = 430 MB。如果不加 Node-Limited 约束，这 430MB 中约 87.5%（= 28/32 节点外） 需要走 IB，即 ~376MB / 50 GB/s ≈ 7.5ms；加上 Node-Limited M=4 后，只有约 9.4%（= 3/32 节点外）走 IB，即 ~40MB / 50 GB/s ≈ 0.8ms。节点内的 390MB 走 NVLink 只需 390MB / 900 GB/s ≈ 0.43ms，远小于计算时间。**这就是为什么 Node-Limited Routing 号称"通信减少 8 倍"——从 7.5ms 到 0.8ms，几乎就是这个倍数关系。**

还有一层容易忽视的工程细节：Node-Limited Routing 不仅减少了带宽消耗，还减少了 **IB 网络上的消息条数**。IB 网络的延迟不仅跟带宽有关，还跟"有多少条 outstanding message"有关——消息条数太多时，交换机的调度开销、credit-based flow control 的反压都会让有效带宽远低于理论峰值。把每 token 的跨节点目的地从 8 个压缩到最多 4 个，消息数几乎减半，IB 网络的实际利用率反而上升。

实测数据（DeepSeek-V3 论文）：开启 Node-Limited Routing 后，单步 MoE 层时间从 12ms 降到 6ms，全模型训练吞吐提升 35%。

### 7.3 Shared Expert 架构

#### 7.3.1 设计动机

要直观理解为什么需要 shared expert，可以用一个医学类比：**routed expert 像专科医生，shared expert 像全科医生**。一个综合医院如果只有专科医生，那患者无论得了什么病都要先做一堆检查找对科室；而有了全科医生，常见病可以直接处理掉、疑难杂症再分流到专科。MoE 也是一样——如果只有 routed expert，那"基本句法理解""常用词汇""标点处理"这些通用任务也要被 router 反复决策、反复分发，既浪费 routing 算力，又让 routed expert 内部承载了一堆"通用知识"的重复学习。引入一个永远激活的 shared expert，等于在每一层都铺一个**语义底座**，把通用知识固化下来，让 routed expert 可以更彻底地分化成"真正的专科"。

这种设计本质上是在做"知识分层"——通用知识由 shared expert 保底，专门知识由 routed expert 承担。从信息论角度看，它把一个高熵分布（256 个 routed expert 都要学一点通用知识）拆成了一个低熵分布（shared expert 学通用 + routed expert 学专门），**参数利用率显著提升**。

如果你仔细观察 routed expert 的行为，会发现一个有趣现象：**很多 expert 都在重复学习一些"通用知识"**，比如基本的句法理解、常用词汇、低频但跨领域的概念。这造成了参数浪费——理论上这些知识只需要一份。

DeepSeek-V3（以及更早的 DeepSeekMoE）引入了 **Shared Expert** 架构：拆分出 1 个（或几个）始终被所有 token 激活的 expert，让它专门承载"通用知识"；routed expert 则可以更彻底地专门化。

#### 7.3.2 数学表达

每个 MoE 层的输出由两部分相加：

$$
\text{FFN\_output}(x) = \underbrace{\text{shared\_expert}(x)}_{\text{always active}} + \sum_{i \in \text{TopK}} g_{i} \cdot \text{routed\_expert}_{i}(x)
$$

伪代码：

```text
class DeepSeekMoELayer:
    def forward(x):
        # 1) Shared expert：所有 token 必经
        shared_out = shared_expert(x)        # 1 个 expert，全 batch 都过

        # 2) Routed experts：TopK 选 8 个
        scores = router(x)                   # [B, 256]
        topk_idx, topk_g = topk_select(scores, k=8, bias=bias)
        routed_out = zeros_like(x)
        for i in topk_idx:
            routed_out += topk_g[i] * routed_experts[i](x)

        # 3) 相加，残差，输出
        return x + shared_out + routed_out
```

#### 7.3.3 效果

- **perplexity 降低 ~0.5%**：相同激活参数下，shared+routed 比纯 routed 效果更好；
- **训练更稳定**：shared expert 提供了"语义底座"，routed expert 不需要承担通用任务，方差降低；
- **expert collapse 概率下降**：通用知识由 shared expert 承担，routed expert 不会出现"什么 token 都想抢"的退化。

代价：shared expert 始终激活，单 token 的激活参数增加了一个 expert 的量。但因为 shared expert 通常做得不大（~300M），开销可控。

### 7.4 Fine-Grained Expert（细粒度专家）

#### 7.4.1 传统 vs 细粒度

要理解细粒度专家为什么能带来表达能力的飞跃，关键是看**组合空间的爆炸式增长**意味着什么。Mixtral 8×7B 是 8 选 2，组合数 $C(8,2)=28$——也就是说，整个模型最多能形成 28 种不同的"知识子集激活模式"。这听起来不少，但跟语言学里需要表达的细粒度概念相比远远不够：英文里的语法范畴就有几十种、专业领域的小众术语成千上万种、同义概念的细微语义差别更是难以穷尽。28 种组合显然不够用，模型只能让每个 expert 承担一个"宽泛主题"，专门化程度有限。

而 DeepSeek-V3 是 256 选 8，组合数 $C(256,8) \approx 4 \times 10^{14}$——**400 万亿种可能的激活模式**。这个数字已经超过了语言学里所有可能的细粒度概念组合，意味着模型对每一个 token 都可以"精心定制"一个独特的 expert 子集去处理。这种细粒度的代价是单 expert 变小（从 7B 到 230M），但因为每个 expert 只需要处理极窄的一类模式，它不需要那么大也能学得很尖锐。**这就像把一支 8 人的全科团队拆成 256 人的专项小组——每个人技能更窄，但整个团队能覆盖的场景几何级增加**。

经典 MoE（如 Mixtral 8×7B）的设计是"少而大"——8 个 expert，每个 7B，TopK=2。DeepSeek 走的是相反方向："多而小"——256 个 expert，每个 ~230M，TopK=8。

| 架构 | $N_{\text{exp}}$ | $k$ | $k / N_{\text{exp}}$ | 单 expert 参数 | 激活参数 | 组合空间 |
|------|---|---|---|---|---|---|
| 标准（Mixtral 8×7B） | 8 | 2 | 25% | 7B | ~14B | $C(8,2)=28$ |
| 细粒度（Qwen1.5-MoE） | 64 | 8 | 12.5% | 900M | ~7.2B | $C(64,8)\approx 4{\times}10^{9}$ |
| 超细粒度（DeepSeek-V3） | 256 | 8 | 3.1% | 230M | ~1.8B | $C(256,8)\approx 4{\times}10^{14}$ |

组合空间的爆炸式增长意味着：**模型可以表达的"专门化模式"数量呈天文数字增长**。

#### 7.4.2 细粒度的三大好处

**① 冗余度更高**：单个 expert 失活影响有限。8 个 expert 时丢失 1 个 = 损失 12.5% 容量；256 个时 = 损失 0.4%。

**② 负载均衡更容易**：256 个 expert 比 8 个 expert 更容易"均匀分布到 token 上"。极端例子：如果某个 batch 只有 100 个 token，8 个 expert 必然有 expert 收到 0 个；256 个 expert 下，每个 expert 平均收到 100×8/256 ≈ 3 个 token，分布更平滑。

**③ 特殊化程度更高**：每个 small expert 只需关注一类极窄的模式（比如"Python 语法 + 数学等式 + 英文标点"的某种组合），学到的特征更尖锐。论文实验显示，细粒度 expert 的 routing entropy 显著降低（更确定地选择特定 expert），说明每个 expert 真的找到了自己的"领域"。

#### 7.4.3 Trade-off

这里再展开一下“细粒度 + TopK ”带来的一个隐形好处：**容错性与可缩放性**。追求全局最优 routing 的关键困难是“单点故障”——如果某个 expert 的权重因为某次 loss spike 被損坏，对 8-expert 模型来说就是 12.5% 容量直接作废，训练很难自愈；而对 256-expert 模型而言只是 0.4% 容量损失，其它 expert 完全可以补位。另外，当要在训练过程中动态加入新的 expert（比如训练到一半发现数学能力不足，想加几个专治数学的 expert），细粒度架构只需要在 256 expert 后追加少量小 expert、重新初始化 router 输出层（只多几列），音体改动远小于粗粒度架构。**这种“多而小”的设计让模型在生命周期内的可扩展性显著提升**。

细粒度并非没有代价：

- **通信总量**：基本不变。256 expert × 8 选 1 与 8 expert × 2 选 1，token 总数和激活的 expert 总数相近，All-to-All 总流量量级相同；
- **permutation 成本微增**：256 个分组比 8 个分组的 grouping 操作开销略大，但相比通信和计算可忽略；
- **GroupedGEMM 友好性**：256 个小矩阵相比 8 个大矩阵，对硬件 utilization 略不友好——这就是为什么 DeepSeek 必须配合自研的 DeepEP/DeepGEMM kernel 才能跑到峰值性能。

### 7.5 FP8 混合精度训练与通信压缩

#### 7.5.1 为什么需要 FP8

看到"FP8 训练"很多工程师的第一反应是：**精度真的够用吗？** 这种担心在 2022 年之前是完全合理的——早期 NVIDIA Volta/Ampere 架构上的 FP16 训练就经常发散，需要 loss scaling、动态调整等一堆 trick 才能稳住。FP8 把每个数字砍到 8 bit，动态范围只有 $\pm 2^{8}$ 量级，正常 GEMM 累加几下就 overflow 或 underflow，怎么看都像是"为了省带宽而牺牲收敛性"的危险方案。

但 DeepSeek-V3 用 14.8T tokens 的实测证明了：**只要分层混合精度策略设计得当，FP8 训练和 BF16 训练的 loss 曲线可以做到几乎完全重合**。关键的几个设计点是：① master weights 永远保持 FP32（优化器主权重不变）；② GEMM 累加器用 FP32（防止累加精度损失）；③ 权重存储用 FP8 但带 per-tile scale（128×128 一个 scale，规避 outlier）；④ 梯度用 BF16（保留小梯度信号）。**这套组合拳的核心思想是：FP8 只用在"能承受"的地方（前向 activation、forward GEMM 输入、通信传输），FP32/BF16 保留在"必须精确"的地方（master weights、累加器、梯度）**。

BF16 已经是大模型训练的标配，但 MoE 训练中通信占比高达 30–43%，BF16 通信仍然是瓶颈。FP8 把每个数字从 16 bit 砍到 8 bit，通信量减半，显存也减半——但有一个核心问题：**FP8 表示范围窄、精度低，直接用会发散**。

DeepSeek-V3 设计了一套**分层混合精度**策略，把 FP8 用在"能承受"的地方，BF16/FP32 保留在"必须精确"的地方。

#### 7.5.2 DeepSeek-V3 的混合精度方案

```text
# 显存中的存储格式
master weights:   FP32  (优化器主权重，always保留)
working weights:  FP8   (训练时用的副本，每步从 FP32 cast)
gradients:        BF16  (反向传播累积精度)
optimizer state:  FP32  (AdamW 的 m/v)

# 计算路径
forward:
  - GEMM 输入：FP8 (activation) × FP8 (weight)
  - GEMM 累加器：FP32  ← 关键，防止累加精度损失
  - GEMM 输出：cast 回 BF16

backward:
  - 梯度累积：FP32 累加器，BF16 存储

# 通信路径
AllGather (权重 / activation)：用 FP8（带 per-tensor scale）
All-to-All (MoE token dispatch)：用 FP8（通信带 scale 元数据）
AllReduce (梯度 sync)：用 FP32 累积，每 rank 间 BF16 传输
```

关键技术点：

- **per-tile scaling**：FP8 的动态范围只有 ~10 量级，DeepSeek 把权重分成 128×128 的 tile，每个 tile 独立 scale，避免 outlier 把整个张量 overflow；
- **GEMM 累加器 FP32**：H100 的 FP8 Tensor Core 支持"FP8 输入 + FP32 累加"，这是 FP8 训练能收敛的核心硬件能力；
- **梯度仍用 BF16**：梯度的数值范围波动大，强行 FP8 会丢失小梯度信号。

#### 7.5.3 收益与代价

```text
收益：
  - 通信量减半                  →  AllGather/All-to-All 时间 -50%
  - 权重显存减半                →  单卡可放更大模型
  - GEMM 计算吞吐 +60–80%       →  H100 FP8 算力是 BF16 的 2×
  - MegaScale 1440 H100 训 352B MoE：相对 BF16 训练效率 ×1.88

代价：
  - per-tile scale 额外存储 ~0.1% 显存
  - cast 操作引入少量额外 kernel（已被 fusion 优化）
  - 调试难度上升（FP8 行为远比 BF16 复杂）
```

DeepSeek-V3 论文做了同模型的 FP8 vs BF16 收敛对比实验：在 14.8T tokens 训练全程，两者 loss 曲线几乎重合，**无 convergence 差异，无 loss spike**。这意味着 FP8 训练已经从"研究阶段"进入了"生产阶段"。

但必须强调：**这套 FP8 方案对硬件有硬性要求**。“FP8 输入 + FP32 累加器”是 H100 Tensor Core 的专用硬件路径（SM90 架构新增），在 A100（SM80）上无法复现。如果你的集群还是 A100，则只能用 FP16/BF16 训练，FP8 的所有收益都拿不到。这也解释了为什么 DeepSeek-V3 的训练集群必须是 H100/H800——不仅因为算力大，更因为 FP8 支持是架构级别的。

另外，当前社区在 FP8 训练上还存在一个认知误区：“FP8 就是无脑把 BF16 截断到 8 bit”。实际上 FP8 有两种格式——E4M3（动态范围大、精度低）和 E5M2（动态范围更大、精度更低）。DeepSeek-V3 在前向用 E4M3（因为 activation 数值范围可控）、在反向的部分步骤用 E5M2（因为梯度数值范围波动更大）。这种“前向/反向用不同 FP8 子格式”的实践在论文里不显眼，但是训练能收敛的关键细节之一。

至此，DeepSeek-V3 在 MoE 设计上的四大创新——**无 aux loss 均衡、Node-Limited 路由、Shared + Fine-Grained 架构、FP8 全链路**——构成了 2025 年 MoE 训练的"新基线"。后来的 Qwen3-MoE、Llama-MoE 等模型在不同程度上都吸收了这些设计。

**本章小结**：DeepSeek-V3 的四大创新看似各自独立，其实背后有一条统一的设计哲学——**所有看似"为了均衡"的人为设计都是有代价的，能让模型自然均衡的方案才是好方案**。无 aux loss 是把均衡从 loss 项里剥离出来、改用动态 bias 反馈控制；Node-Limited 是把均衡的代价从"router 学得别扭"转为"路由空间显式约束"；Shared+Fine-Grained 是通过架构分层让通用知识不需要被反复均衡；FP8 则是承认"在能承受的地方就用低精度"而不是处处保守。这四点合起来，让 DeepSeek-V3 在 671B 总参规模上实现了"训练全程零 spike + FP8 吞吐 1.88×"的工程奇迹。这也给后续 MoE 模型设计指明了方向——**好的设计不是叠加更多约束，而是发现哪些约束其实可以拿掉**。

## 八、工程挑战与解决方案

理论再优雅，最终都要落到工程上。MoE 训练在工程上有四座大山：**显存**、**checkpoint**、**负载均衡**、**通信**。本章把每一座山的实战处理方案讲清楚。

### 8.1 显存管理

#### 8.1.1 MoE 显存账本

在拆解 671B 模型的账本之前，先回答一个很多人问过的问题：**一张卡 80GB 够不够跑 MoE？** 答案是：跑推理勉强可以，跑训练远远不够。原因是 Dense 和 MoE 在训练期间的显存需求有本质差异。Dense 模型里的参数可以通过 TP+PP+ZeRO 被均匀切到所有卡上，每张卡只存总参数的 1/N；但 MoE 里的 expert 参数占总参数的 80%+，这部分要么不切分（全复制）、要么不能继续用 TP 切分（因为 TP 切分需要所有 token 都走同一条 GEMM，与 MoE 的稀疏激活冲突），只能用 EP 按 expert 粒度切到不同卡上。而 EP 切分后，每卡仍然要存一部分完整 expert 参数，所以总参数不能被“压缩”到任意小。

更重要的是**MoE 比 Dense 更吃显存的根本原因**：优化器状态 (AdamW 的 m/v) 是按总参数 $\Theta_{t}$ 计算的，不是按激活参数 $\Theta_{a}$。Dense 模型可以用 ZeRO-2/3 把这些优化器状态均匀切给所有 rank；但 MoE 里的 expert 优化器状态只能跟随 expert 本身被切到有限的 EP rank 上，而不能别出心裁。**这就是为什么 671B Dense 可以用 80 张卡勉强装下，而 671B MoE 需要 150+ 张卡才能跑起来**——实际上生产部署中常用到 256–2048 张。

我们先把一个 MoE 模型的显存清清楚楚列出来。以 DeepSeek-V3（$\Theta_{t}=671B$ 总参，$\Theta_{a}=37B$ 激活）为例：

```text
# 静态显存（不随 batch 变化）
model_params  = Theta_total × bytes_per_param      # 671B × 2 (BF16) = 1.342 TB
optimizer     = Theta_total × bytes_per_optstate   # 671B × 16 (AdamW FP32) = 10.736 TB
              # AdamW: master weights(4) + m(4) + v(4) + grad buffer(4) = 16
static_total  ≈ 12 TB
              # 单卡 H100 80GB 显存 → 必须分到 ≥150 张卡

# 动态显存（每 step）
activations   = depends_on(batch, seq_len, recompute_policy)  # ~数百 GB/卡
gradients     = Theta_total × bytes_per_grad                  # 1.34 TB（与权重同形）
a2a_buffers   = 2 × num_tokens × hidden × bytes  
              # All-to-All 临时缓冲，前向/反向各一份
              # 32K tokens × 7168 hidden × 2 BF16 = ~470 MB/卡

# 总账
total_per_GPU ≈ 60–80 GB (装满)
```

一句话总结：**671B MoE 模型至少需要 150 张 H100 才能装得下，且每张卡都贴近 80GB 容量上限**。

#### 8.1.2 降低显存的四把斧头

**① ZeRO-1 配合 EP：optimizer 状态分片**

最大的杀手——10.7TB 的 optimizer 状态——可以通过 ZeRO-1 分片到 DP rank 上。如果 EP=64，配合 DP=64，optimizer 显存 ÷ DP_size：

```text
ZeRO-1 + EP=64 + DP=64 (共 4096 卡):
  per-GPU optimizer 显存 = 10.7 TB / 4096 = 2.6 GB  ✓ 完全可接受
```

注意：optimizer 分片不影响 expert 参数的 EP 切分逻辑（参考第六章 Parallel Folding）。两者是正交的。

**② Activation Recomputation：用算力换显存**

激活值占用通常是显存大头。Selective Recomputation（只 recompute attention）开销 +5% FLOPs，激活显存减少约 30%；Full Recomputation 开销 +33% FLOPs，激活显存减少 10–20 倍。

```text
recompute 策略选择：
  显存压力小 → 不开 recompute（最快）
  中等压力   → selective recompute attention（+5% FLOPs，节省 30% 激活）
  极度紧张   → full recompute（+33% FLOPs，节省 90%+ 激活）
```

DeepSeek-V3 训练实际只对 attention 做 selective recompute，MoE 层不 recompute（因为 expert 计算量太大，recompute 太贵）。

**③ Expert Offloading：卸到 CPU/NVMe**

把一部分 expert 权重放到 CPU 内存，需要时按需 swap 到 GPU。原理简单但效果差——expert 切换需要走 PCIe（~50 GB/s），跟 GPU 显存（~3 TB/s）差 60 倍。

```text
expert offloading 性价比：
  显存收益：可省 30–50%
  时间代价：每 step +30–80%（PCIe 是瓶颈）
  结论：不推荐，除非显存极度紧张且能用流水线掩盖 PCIe 传输
```

**④ FP8 混合精度：权重 + 通信减半**

如 7.5 节所述，FP8 把权重和通信显存砍半。对 MoE 模型尤为关键，因为 expert 权重占总参数 80%+。

```text
单卡显存对比（同等 batch、seq）：
  BF16: ~78 GB（接近爆炸）
  FP8 :  ~42 GB（轻松运行，剩余空间可加大 batch）
```

### 8.2 Checkpoint 管理

#### 8.2.1 规模挑战

在进入具体数字之前，先想一个实际场景：**训练到一半挂了怎么恢复？** 这不是假设性问题——千卡训练任务连续跑几十天，其间可能仅仅是**某个 NIC 错误、某个 NVLink 闪断、某个机柜温度告警重启**，都可能让某张卡挂掉，进而拖累整个训练作业。面对这种现实，checkpoint 不是可选项，而是训练过程中唯一的“退路”。

生产中的典型恢复流程是：① 调度器（SLURM/K8s）检测到某节点挂掉，自动重启所有 worker；② 每个 worker 从共享文件系统 load 最近一份 ckpt（包含权重 + 优化器状态 + RNG seed + step 号）；③ dataloader 根据 step 号 fast-forward 到对应位置（不重算数据，只跳过已访问过的 sample）；④ NCCL 重新建立所有 process group；⑤ 从上一个 ckpt step 继续训练。**整个恢复过程要求 ckpt 被多个 rank 同时读到，这就要求底层文件系统能以 100+ GB/s 的带宽同时服务上千个读请求**，这是底层设计必须考虑的。

671B 模型保存一次完整 checkpoint 需要多大？

```text
模型权重 (BF16):       671B × 2 = 1.34 TB
optimizer 状态 (FP32): 671B × 12 = 8.05 TB
        # master_weights(4) + m(4) + v(4)
合计单 checkpoint:     ~9.4 TB
向上取整保留:          ~7 TB（实际生产中常用 BF16 保存 optimizer 节省一半）
保留 3–5 份历史版本:    21–35 TB
```

写入速度更是噩梦：

```text
网络文件系统带宽:
  传统 Lustre:     ~30–80 GB/s
  WekaFS/DAOS:    ~100–300 GB/s
  本地 NVMe RAID: ~10–20 GB/s

写一次 7TB checkpoint:
  Lustre:    ~150 秒（训练完全 stall）
  WekaFS:    ~30–70 秒
  本地 NVMe: ~400 秒
```

如果按"每 1000 步保存一次"的常规策略，每次 stall 70 秒，单次实验丢掉的训练时间累计上百小时。

#### 8.2.2 三种优化方案

**① 分片写（Sharded Checkpoint）**

每个 rank 只保存自己负责的参数分片，不集中到 rank 0：

```text
传统方案（gathered）：
  all rank → AllGather → rank 0 → 写 1 个大文件
  写入串行，rank 0 卡死，其他 rank 空等

分片方案（sharded）：
  rank 0 → 写 rank0.pt
  rank 1 → 写 rank1.pt    （所有 rank 并行写）
  ...
  rank N → 写 rankN.pt

性能提升：写入时间 ÷ N_ranks（理想情况），实际 ÷ 数十倍
```

PyTorch 的 `torch.distributed.checkpoint`、Megatron 的 `dist_checkpointing` 都是这个思路。

**② 异步 checkpoint：后台线程写**

```text
class AsyncCheckpointer:
    def save(state):
        # 1) 立刻把 state 拷到 pinned CPU 内存（GPU→CPU 很快，~1 秒）
        cpu_state = pin_memory_copy(state)
        
        # 2) 训练继续，主线程立即返回
        training_loop.resume()
        
        # 3) 后台线程把 cpu_state 写到磁盘（耗时 30 秒）
        background_thread.submit(write_to_disk, cpu_state)
```

主线程只 stall 拷贝的 ~1 秒，剩下 30 秒磁盘写在后台进行。生产中可以每 100 步保存一次而几乎无感。

**③ 分布式文件系统选型**

| 文件系统 | 写入带宽 | 适用场景 |
|----|----|----|
| 本地 NVMe RAID | 10–20 GB/s | 实验环境，单机训练 |
| Lustre（传统 HPC） | 30–80 GB/s | 通用大规模训练，成熟稳定 |
| WekaFS | 100–300 GB/s | 大模型训练，需要高吞吐 checkpoint |
| DAOS（基于 PMEM） | 200–500 GB/s | 超大规模，预算充足 |
| S3 / OSS | 5–20 GB/s | 长期归档，不适合频繁 ckpt |

实践建议：训练目录用 WekaFS/DAOS（高吞吐），归档目录用 S3（低成本）。每 N 步保存到 WekaFS，每 M×N 步同步一份到 S3。

实际生产中的一些踩坑经验值得记录。**坑一：分片 checkpoint 的 rank 映射错位**：如果训练复用时改变了 EP/DP 配置（比如从 EP=32 改为 EP=64），各 rank 保存的参数分片不再一一对应，必须有“重分片加载”逻辑（Megatron 叫 `dist_checkpointing.load_with_resharding`）。忽略这一步会导致不同 expert 读到错位的权重，等价于“用别人的脑子替换自己的脑子”，训练立刻发散。**坑二：异步 checkpoint 与主训练争显存**：异步写盘期间，CPU 侧的 pinned memory 占用不小（每卡 2–5 GB），如果同时的主循环还在做 GPU→CPU 的梯度 offload，两者会争 PCIe 带宽和 pinned buffer。生产中的解法是“checkpoint 步和 offload 步错开”——比如 step%200==0 存 ckpt，step%200==100 做 offload 整理。**坑三：checkpoint 超大导致 load 时间太长影响恢复速度**：7TB ckpt 从 Lustre 加载需要 2–3 分钟，当你有千张卡同时读时 Lustre 的 MDS（元数据服务）可能被打爆。解法是“分层 load”：先让每节点的 rank 0 从并行文件系统读入本节点的全部分片，再通过 NVLink 广播到节点内的其他卡，这样对文件系统的并发读压力降低 8 倍（节点内 8 卡只产生 1 次读）。

### 8.3 负载均衡实战

#### 8.3.1 Expert Collapse 的级联效应

在看报警阈值之前，先谈一下实际生产中的**监控 dashboard 该怎么设计**。一个成熟的 MoE 训练 dashboard 至少要包含三层面板：第一层是**实时总览**，显示当前 step 的 loss、grad norm、总 token 吞吐、expert load 直方图（256 个柱子一眼看清是否均匀）、routing_entropy 曲线；第二层是**时间序列**，过去 24/72 小时的 load_balance_ratio、dead_expert_count、top-1 expert frequency、per-layer routing entropy 的变化趋势；第三层是**告警面板**，列出过去 1 小时中触发的阈值事件、备注是否已被手动确认。

报警阈值怎么设是门艺术。一般推荐的实践是：“温和告警”阈值设为历史基线的 1.5–2倍（比如历史 ratio 平均 1.2，告警阈值设 2.5），触发后发 Slack 通知但不停训练；“严重告警”阈值设为历史基线的 3–5 倍，触发后发电话 / 值班同事唤起，同时自动 dump 当前详细状态以供调查。需要强调的是：**一定要遵循历史基线而不是全局常量**，因为不同模型架构、不同训练阶段的“正常 ratio”差别很大。初期训练 ratio 可能 2–3是正常的，后期稳定后该降到 1.2–1.5，如果用同一个阈值要么初期误报、要么后期漏报。

负载不均不是一个静态问题，而是一个**正反馈放大**的过程。下面这段伪代码展示了一个典型的 collapse 轨迹：

```text
# Expert Collapse 正反馈循环
step T:
  expert_5 激活频率 = 30%（已经偏高，但不至于灾难）
  → expert_5 在本 step 获得更多梯度更新
  → expert_5 的参数表达能力略微增强

step T+1:
  router 计算亲和度时，expert_5 的 e_5 向量被更新得更"通用"
  → 更多 token 对 expert_5 的亲和度上升
  → expert_5 激活频率 = 31%

step T+10:
  expert_5 频率 = 45%
  其他 expert 因为接收 token 少，更新慢，能力相对退化
  → 进一步降低被选中概率

step T+100:
  expert_5 频率 = 80%
  正反馈已经稳态化

step T+1000:
  expert_5 频率 = 99%
  其他 255 个 expert 各自 <1%
  ★ 模型实际已经退化成"Dense + 255 个死 expert"
```

一旦进入这个循环，模型相当于浪费了 99% 的参数。这就是为什么 aux loss / bias-based balancing 这么重要。

#### 8.3.2 四个必须监控的指标

```text
# 训练过程中要打到日志/dashboard 的关键 metric
expert_load[i]       = token_count_to_expert_i / total_tokens
                       # 理想值: 1 / N_routed_experts ≈ 0.39% (256 expert)

load_balance_ratio   = max(expert_load) / mean(expert_load)
                       # 理想值: 1.0    良好: < 1.5    警戒: > 3.0

dead_expert_count    = #{i : expert_load[i] == 0 over last 100 steps}
                       # 理想值: 0      警戒: > 5

routing_entropy      = -sum(p_i * log(p_i)) for p_i = expert_load[i]
                       # 理想值: log(N_routed_experts) = log(256) ≈ 5.55
                       # 越接近上限越均匀
```

实战中：

- `load_balance_ratio > 3.0` 持续 1000 步 → 调大 bias 更新步长 $\gamma$，或开启 sequence-level aux loss 兜底；
- `dead_expert_count > 0` → 一定要在下一次 ckpt restore 时手动 reset 这些 expert 的 router 偏置，否则它们永远死亡；
- `routing_entropy < 4.0`（针对 256 expert）→ 立即报警，可能进入 collapse 早期阶段。

### 8.4 通信瓶颈与优化

#### 8.4.1 通信占比分析

在看占比数字之前，先解决一个更实际的问题：**怎么判断你的训练是计算 bound 还是通信 bound？** 这是优化之前必须回答的问题，否则会出现“拼命优化 kernel 但吞吐没动”的尴尬场面——因为真正的瓶颈在别处。

诊断方法有三招。**第一招是单点 profiling**：用 NVIDIA Nsight Systems 抓 100 个 step 的 timeline，看 NCCL kernel 的总时间 / step 总时间 = 通信占比。如果这个比例 > 30%，你是通信 bound；< 15%，你是计算 bound；介于两者之间则是混合型，优化任一侧都能拿到收益但不会很大。**第二招是理论计算**：用模型 FLOPs 除以 GPU 的峰值 FLOPS（比如 H100 BF16 是 989 TFLOPS）得到“计算下限时间”；用总通信量除以带宽得到“通信下限时间”。谁大谁就是 bound。**第三招是扰动实验**：把 EP 翻倍试一次，如果吞吐明显下降（说明跨节点通信变多），你是通信 bound；如果吞吐几乎不变（说明通信被计算掩盖），你是计算 bound。

诊断出结果后才能选优化策略：通信 bound 优先上 Node-Limited Routing 与 FP8 压缩，计算 bound 优先上 GroupedGEMM 与 Fusion Kernel，两者都不例外才上 Tiled Overlap。**盲目并行调优是 MoE 训练优化的头号坑**。

不同规模下，MoE 训练的通信占比差异很大：

```text
模型规模           |  通信占总时间
70B MoE，64 GPU    |   18–25%
350B MoE，512 GPU  |   30–35%
671B MoE，2048 GPU |   38–43%  ← 通信成为绝对瓶颈
```

这个比例在没有专门优化的情况下，会随规模线性恶化。

#### 8.4.2 带宽差异：NVLink vs IB

```text
单节点内 (8 张 H100 通过 NVLink + NVSwitch 互联):
  双向带宽 ≈ 900 GB/s/卡    （SXM5 NVLink 4代）

跨节点 (通过 IB 网络互联):
  IB NDR (400 Gb/s) ≈ 50 GB/s/卡
  IB HDR (200 Gb/s) ≈ 25 GB/s/卡

差距：节点内 / 跨节点 ≈ 18 倍

这就是为什么"尽量把流量留在节点内"是 MoE 训练的核心优化原则。
```

#### 8.4.3 优化策略汇总

```text
策略 1：Device-Limited Routing（DeepSeek-V3）
  - 限制每 token 跨节点数 ≤ M
  - 节点内通信走 NVLink，跨节点流量 ÷ M
  - 收益：通信时间 -40~60%

策略 2：FP8 通信压缩
  - All-to-All / AllGather 用 FP8 + per-tile scale
  - 通信量减半
  - 收益：通信时间 -45%

策略 3：Tiled Overlap（第六章已讲）
  - All-to-All 与 GEMM 切 tile 并发
  - 收益：通信被计算掩盖 80%

策略 4：MegaKernel / DeepEP
  - 把 router/permute/gemm/unpermute 融合
  - kernel launch 减少 80%，中间张量不落显存
  - 收益：单层 forward 时间 -50%

策略 5：拓扑感知 Process Group
  - 把 EP rank 优先排在同节点
  - All-to-All 自动走 NVLink
  - 收益：跨节点流量大幅下降，依赖运维配置

综合应用上述策略后，671B 模型在 2048 H100 上的通信占比
可从原始的 ~43% 降到 ~18%，整体吞吐提升 ~2.1×。
```

至此，工程四大山——显存、checkpoint、负载均衡、通信——都有了系统性的应对方案。这些方案不是"调一调超参就能上"的小技巧，而是必须从框架层、调度层、网络层同时配合的系统工程。

最后再讨论一个在四座山之间横切的工程决策：**怎么判断什么时候工程上的投入已经“够用”了，不需要继续优化？** 答案是看 **Model FLOPs Utilization (MFU)**。MFU = 实际每秒计算量 / 理论峰值计算量。当前千亿规模 MoE 训练的 MFU 典型值在 35–45%（在 H100 集群上），而 Dense 模型通常能做到 50–60%。这个差距恰好对应着 MoE 的 All-to-All 通信开销。如果你已经通过上述优化把 MFU 做到 40%+，说明工程侧已经没有太大的空间了，剩下的提升只能靠算法创新（比如减少激活参数比、用更少的 TopK）。如果 MFU < 30%，则工程侧一定还有大量可优化空间——先做 profile、找到具体的 bound，再按本章的策略对症下药。

**本章小结**：工程四大山看似独立，其实背后是深度耦合的。显存不够造成你要加大 EP，EP 一大就造成跨节点通信变多，通信压力反过来又逐出了 Node-Limited Routing 这种设计；checkpoint 太大让你不敢频繁存，但不频繁存一旦出事就会丢上百亿 token 的训练结果，于是又逐出了异步 ckpt 与分片写入的优化；负载不均衡表面上是 routing 问题，实际上会造成部分 GPU 闲置、那就是变相的显存浪费。所以理解这四座山不能孤立看，而要看它们是怎么互相传递压力、又是怎么被一套组合拳（EP+ZeRO+异步 ckpt+bias 均衡+Node-Limited）同时解决的。这种系统级的协同设计，才是 MoE 训练与 Dense 训练在工程难度上的本质区别。

## 九、总结与展望

### 9.1 核心要点回顾

回望整篇文章，我们沿着"为什么 MoE → 怎么做 MoE → 怎么训练好 MoE"的主线走了一遍：

- **第一、二章**：MoE 的本质是用稀疏激活换取参数规模——总参 $\Theta_{t}$ 决定容量，激活参数 $\Theta_{a}$ 决定计算成本，两者解耦让模型可以"大而省"；
- **第三章**：Routing 是 MoE 的灵魂。Top-K 选择、softmax 归一化、token vs expert choice、aux loss 均衡，每一步都有具体的工程权衡；
- **第四章**：Expert Parallelism (EP) 把不同 expert 放到不同 GPU 上，用 All-to-All 在 token 维度交换数据，是分布式 MoE 的核心范式；
- **第五章**：EP 与 DP/TP/PP 必须组合使用。Parallel Folding 是把 EP 与 DP 真正解耦的关键，让两者可以独立切分；
- **第六章**：Megatron-Core 的工程栈——Token Dispatcher、GroupedGEMM、Fusion Kernel——三层加在一起才能撑起千亿规模训练；
- **第七章**：DeepSeek-V3 的四大创新（无 aux loss、Node-Limited、Shared+Fine-Grained、FP8）已成为 2025 年的新基线；
- **第八章**：工程四座大山（显存、ckpt、均衡、通信）必须系统性应对，否则任何理论优势都会被工程难度吃掉。

### 9.2 MoE 发展趋势

观察过去两年的演进，可以总结出五条清晰的趋势：

- **从"少而大"到"多而小"**：expert 数从 8 个涨到 256 个，单 expert 参数从 7B 降到 200M。激活参数稳定在 2–5% 区间，但组合空间爆炸式增长；
- **从辅助损失到无损失**：DeepSeek-V3 的 dynamic bias 方案证明 aux loss 并非必需，且 bias 方案更稳定。后续 Qwen3、Llama4 系列均跟进；
- **从节点间 All-to-All 到节点内通信**：Device-Limited Routing 把跨节点流量限制在 M 个节点内，配合拓扑感知 process group，让 NVLink 重新成为主战场；
- **从 BF16 到 FP8**：全链路低精度训练（权重、激活、通信 FP8 + 累加 FP32 + 梯度 BF16）已经从研究走向生产，显存与通信双双减半；
- **从独立框架到统一框架**：早期 DeepSpeed/Megatron/FairScale 各做一套，现在 Megatron-Core 把 EP/DP/TP/PP/SP/CP 统一在一个 process group 抽象下，工程门槛大幅降低。

### 9.3 训练效率提升方向

未来 1–2 年值得关注的工程方向：

- **计算-通信完美重叠**：MegaKernel 把 dispatch/gemm/combine 全部融到一个 kernel 内，配合 SM 级别的精细调度，目标是让通信完全消失在 GEMM 的间隙中；
- **更高效的 GroupedGEMM**：H100 的 GroupedGEMM 仍然受限于 launch 开销与 wave quantization，下一代硬件（B200/H200）的更大 L2、更高 SM 数量将带来新机会，配套的 kernel 也需要重写；
- **智能负载预测**：用一个轻量 model 提前预测下一个 batch 的 expert load 分布，提前进行 capacity 分配甚至 expert 迁移；
- **动态 expert 扩缩容**：训练过程中动态增减 expert 数量（早期少 expert 加速收敛，后期多 expert 提升容量），打破"训练时固定架构"的限制；
- **MoE + Long Context 协同**：当 seq_len 涨到 1M+ 时，token 数量爆炸，All-to-All 的 message 数量级增加，需要新的 routing 与通信范式（如 Hierarchical MoE）来应对。

### 9.4 MoE vs Dense 终极对比

在学了这么多内容之后，读者很可能会问一个最本质的问题：**到底什么时候该用 MoE、什么时候该用 Dense？** 这个问题没有绝对答案，但可以从几个维度给出明确的决策框架：

| 维度 | Dense | MoE | 判断依据 |
|------|-------|-----|----------|
| **参数效率** | 所有参数每次都用 | 只用 3–10% 参数 | 如果你的算力预算有限、但想要大模型容量，选 MoE |
| **训练稳定性** | 成熟、loss spike 少 | 需要负载均衡策略 | 如果团队经验有限、且不允许试错，选 Dense |
| **工程复杂度** | DP/TP/PP 三维 | DP/TP/PP/EP/CP 五维+ | 如果团队人数 < 5、没有专职 infra，慎用 MoE |
| **推理成本** | 可预测、稳定 | 激活参数少、但 expert 切换有开销 | 如果推理场景 latency 敏感，Dense 更可控 |
| **知识容量** | 受限于参数量 | 线性扩展 | 如果需要的世界知识量极大（多语言、多领域），MoE 优势明显 |
| **跨节点通信** | AllReduce 可控 | All-to-All 较重 | 如果集群网络差（没有 IB），MoE 很难跑起来 |
| **模型缩放** | 主要靠加层/加宽 | 可以只加 expert 数 | MoE 的缩放路径更灵活、算力成本更低 |

简单的决策规则：

- 如果你有充足的 GPU 集群（∖64 张 H100）、还不错的网络（∞IB NDR）、专业的 infra 团队，且目标模型容量在 100B+ 级别——**用 MoE**。同等算力下能拿到 3–5 倍于 Dense 的模型容量。
- 如果算力有限（<64 GPU）、网络一般（只有 RoCE）、团队人少、目标模型在 7B–70B 范围——**用 Dense**。工程复杂度低，迭代速度快，很多情况下同等工作量能达到更好的质量。
- 如果你在该两者之间犹豫，建议先用 Dense 做实验，当明确发现"参数量不够、但算力不允许继续加大 Dense"时，再迁移到 MoE。

### 9.5 从零搞建 MoE 训练 Pipeline 的建议路径

如果读者想从零开始搭建一套可用于生产的 MoE 训练 pipeline，以下是按梯度推荐的实施路径（从小到大、从简单到复杂）：

**第一阶段：单机原型验证（“1 周内可完成”）**

- 目标：在单机 8×A100/H100 上跑通一个 8-expert、TopK=2 的小模型（约 3B 总参），验证 loss 正常下降、expert load 大致均匀。
- 推荐框架：Megatron-Core（完整度最高）或 DeepSpeed-MoE（上手最快）。
- 关键检查点：确保 router 开始学到非均匀 routing（routing entropy < log(N)）、没有 dead expert、grad norm 稳定。
- 可能踩的坑：router 初始化不对称导致 expert collapse； aux loss 系数 $\alpha$ 调得过大导致 perplexity 不降。

**第二阶段：多机并行扩展（“2–4 周”）**

- 目标：扩展到 2–4 个节点（16–32 张卡），开启 EP（比如 EP=8，每卡 1 expert），验证跨节点 All-to-All 正常工作。
- 关键检查点：通信占比是多少（用 Nsight Systems 拍）？是否出现跨节点流量飙升？吞吐是否线性扩展？
- 可能踩的坑：忽略了 process group 的拓扑感知配置，导致 EP 组跨节点而本可以节点内；或者 NCCL 环境变量没配对。

**第三阶段：规模化与生产化（“4–12 周”）**

- 目标：扩展到 64–256 张卡，开启 Parallel Folding（EP=32/64）、GroupedGEMM、异步 checkpoint、bias-based 负载均衡。模型规模拉到 64–256 expert、总参 100B+。
- 关键检查点：MFU 是否达到 35%+？checkpoint 写入是否影响训练？负载均衡 ratio 是否 < 1.5？
- 可能踩的坑：GroupedGEMM + FP8 同时开启时 scale 传递错误；异步 ckpt 与 gradient offload 争 PCIe 带宽；多维并行组合时 tensor shape 不匹配。

**第四阶段：极致优化（“持续迭代”）**

- 目标：在 500–2048 张卡上拉到最大吞吐。开启 Node-Limited Routing、FP8 全链路、Fusion Kernel、通信-计算 overlap。
- 关键检查点：MFU 是否达到 40%+？“无 spike + 零 dead expert + routing entropy 接近上限”的三元目标是否同时满足？
- 关键软件依赖：Megatron-Core 最新版 + DeepEP/DeepGEMM + NCCL 2.19+ + FlashAttention 2.5+。

这条路径的核心原则是“每一步都先跑通再优化”——不要试图一步到位地在 2048 张卡上同时开启所有优化，那样出了 bug 根本不知道哪一层出的。每个阶段先验证正确性（loss 收敛、expert load 均匀、无 NaN），再追求性能（MFU、吞吐、通信占比）。

MoE 不是 LLM 的终点，但它已经是当下"参数规模 vs 计算成本"权衡的最佳解。理解它的每一层设计哲学与工程取舍，才能在下一波架构创新中把握主动权。
